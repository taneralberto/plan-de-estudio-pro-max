# 1.3 — Testing Fundamentals

> "Testing desde el día 1" solo funciona si sabés QUÉ testear, CÓMO, y CUÁNDO es suficiente. Este tema es el fundamento mínimo para que cada línea de código que escribás tenga un test que la respalde.

- [ ] **Qué es un test y por qué existe**

Un test no es "para pasar CI" ni "para llegar al 80% de coverage". Un test es una **pregunta que le hacés a tu código** antes de que un usuario la haga por vos. Cada test responde a una pregunta: "¿qué pasa si...?" — ¿qué pasa si el input es null? ¿qué pasa si el array está vacío? ¿qué pasa si el usuario no está autenticado? Los tests existen porque tu cerebro no puede sostener todos los casos edge de un sistema al mismo tiempo — los tests los documentan en código ejecutable.

**Por qué importa:** Sin tests, cada cambio que hacés al código es una apuesta. "Creo que esto no rompe nada" no es ingeniería — es fe. Los tests convierten esa fe en evidencia. Y en la era de AI, donde generás código que parece correcto pero puede tener edge cases ocultos, los tests son tu red de seguridad más confiable.

**Ejercicio:** Abrí un módulo de FreePress que ya escribiste (o cualquier proyecto). Identificá 3 casos edge que no consideraste cuando lo escribiste. Escribí un test para cada uno. ¿Cuántos fallan?

**Done cuando:** Podés explicar por qué un test es una pregunta, no una confirmación. Y podés mirar una función y listar al menos 3 casos edge que un test debería verificar antes de escribir el código.

**Recursos:**
- Artículo: *"Write Tests. Not too many. Mostly Integration."* de Kent C. Dodds — el marco mental correcto para priorizar qué testear. Buscarlo en [kentcdodds.com](https://kentcdodds.com/blog/write-tests).
- Video: *"What is a Test?"* de Fireship (YouTube) — overview en 100 segundos si nunca escribiste un test.

- [ ] **Estructura AAA (Arrange-Act-Assert) — El patrón universal**

Todo test bien escrito tiene tres partes: **Arrange** (preparás el escenario — datos, mocks, estado inicial), **Act** (ejecutás la acción que querés verificar), **Assert** (verificás que el resultado es el esperado). Este patrón es universal — aplica en Jest, Vitest, Pytest, Go testing, cualquier framework. Si un test no sigue AAA, probablemente está haciendo más de una cosa o está mal estructurado.

```javascript
// Arrange
const user = { name: 'Tana', role: 'admin' };
const article = { title: 'Test', authorId: user.id };

// Act
const canEdit = canUserEditArticle(user, article);

// Assert
expect(canEdit).toBe(true);
```

**Por qué importa:** La estructura AAA hace que un test sea legible para otro humano (incluyendo a tu yo futuro). Si leés "Arrange" y ves datos de setup, "Act" y ves la llamada, "Assert" y ves la verificación — entendés el test en 10 segundos. Sin AAA, un test es código que hace algo pero no se sabe qué.

**Ejercicio:** Reescribí 3 tests que hayas escrito (o que encuentres en un proyecto) usando AAA explícito con comentarios `// Arrange`, `// Act`, `// Assert`. Si alguno no encaja limpiamente en AAA, es probable que el test esté haciendo más de una cosa.

**Done cuando:** Todo test que escribís sigue AAA sin esfuerzo. Podés leer un test ajeno y identificar las tres secciones en 10 segundos.

**Recursos:**
- [Jest — Getting Started](https://jestjs.io/docs/getting-started) — la documentación oficial. Leé hasta "Using Matchers" inclusive.
- Artículo: *"The AAA Pattern in Unit Tests"* de Martin Fowler — por qué funciona y por qué importa la estructura.

- [ ] **Qué testear vs qué no testear**

No todo necesita un test. **Testeá:** lógica de negocio (cálculos, transformaciones, reglas), edge cases (null, vacío, límites), integraciones críticas (auth, pagos, permisos). **No testeés:** getters/setters triviales, código generado automáticamente, detalles de implementación internos que pueden cambiar sin afectar comportamiento. La regla: testeá el QUÉ hace el código (comportamiento observable), no el CÓMO lo hace (implementación interna).

**Por qué importa:** Testear lo incorrecto es peor que no testear nada. Un test que verifica el internals de una función (ej: que llama a un helper privado específico) se rompe cuando refactorizás — aunque el comportamiento sea el mismo. Estos "tests frágiles" hacen que el equipo odie los tests y los ignore. Un test que verifica comportamiento sobrevive refactors.

**Ejercicio:** Mirá los tests de un módulo de FreePress. Clasificá cada test en: (1) testeando comportamiento — ✅, (2) testeando implementación — ⚠️ frágil, (3) testeando algo trivial — ❌ innecesario. ¿Cuántos caen en cada categoría?

**Done cuando:** Podés explicar la diferencia entre "testeando comportamiento" y "testeando implementación" con un ejemplo concreto. Sabés cuándo un test es frágil y por qué.

**Recursos:**
- Artículo: *"Testing Implementation Details"* de Kent C. Dodds — el clásico sobre por qué testeás comportamiento, no internals.
- Capítulo: *Working Effectively with Legacy Code* de Michael Feathers — la sección sobre "seams" y cómo identificar qué testear en código existente.

- [ ] **Mocks básicos: cuándo usarlos, cuándo son un smell**

Un mock reemplaza una dependencia real (base de datos, API externa, servicio) por una versión controlada que vos definís. Útil para aislar la unidad que estás testeando. Pero los mocks tienen un costo: cada mock es un contrato implícito entre el test y la implementación. Si la implementación cambia y el mock no, el test pasa pero el código real falla. **Regla:** mockeá dependencias externas (APIs, DBs, servicios de terceros). NO mockeés dependencias internas del mismo módulo — eso es testeando implementación.

**Por qué importa:** Un test con demasiados mocks no está testeando el código real — está testeando que los mocks funcionan como vos los configuraste. Es una ilusión de cobertura. En la era de AI, esto es especialmente peligroso: la AI genera mocks que hacen pasar el test pero no verifican el comportamiento real.

**Ejercicio:** En FreePress, escribí un test de endpoint que usa supertest SIN mockear la base de datos (usá SQLite in-memory). Después escribí el mismo test mockeando la DB. Compará: ¿cuál te dice más sobre si el código funciona realmente? ¿cuál es más frágil?

**Done cuando:** Podés explicar cuándo un mock agrega valor y cuándo es un smell. Escribís tests de integración que usan dependencias reales (SQLite in-memory) en lugar de mockear todo.

**Recursos:**
- Artículo: *"Mocking is a Code Smell"* de Eric Elliott — por qué los mocks excesivos son un problema y qué usar en su lugar.
- [Supertest](https://github.com/visionmedia/supertest) — la herramienta estándar para testear endpoints HTTP en Node.js sin mockear el servidor.

- [ ] **Coverage como guía, no como meta**

Coverage mide qué líneas de código se ejecutaron durante los tests. 80% coverage significa que el 20% del código nunca se ejecutó en un test — pero NO significa que el 80% funcione correctamente. Una función puede tener 100% coverage y tener un bug que ningún test verifica (porque el test ejecuta la línea pero no verifica el resultado correcto). **Coverage te dice dónde FALTA un test, no dónde hay uno bueno.**

**Por qué importa:** Perseguir un número de coverage genera tests triviales que ejecutan código pero no verifican nada significativo. "Llegamos al 90%" es una victoria de métrica, no de calidad. Lo correcto: usá coverage para identificar código NO testeado, y escribí tests significativos para esas zonas.

**Ejercicio:** Corré `jest --coverage` en FreePress. Identificá los 3 archivos con menor coverage. ¿Son zonas críticas (auth, lógica de negocio) o zonas triviales (config, utils genéricos)? Escribí tests significativos solo para las zonas críticas.

**Done cuando:** Entendés que 100% coverage no significa 0 bugs. Usás coverage como mapa ("¿dónde me falta testear?") no como meta ("¿cuánto me falta para el 80%?").

**Recursos:**
- Artículo: *"Code Coverage is a Poor Target"* de Martin Fowler — por qué coverage es útil como guía pero peligroso como meta.
- [Jest — Coverage](https://jestjs.io/docs/configuration#collectcoverage-boolean) — cómo configurar y leer reportes de coverage.

- [ ] **Testing de endpoints con supertest**

En la Etapa 1, FreePress usa Express. Supertest permite testear endpoints HTTP sin levantar un servidor real — monta la app Express y le dispara requests como si fuera un cliente. Es la forma más directa de verificar que tu API hace lo que debería: status codes correctos, response bodies esperados, headers apropiados, y manejo de errores.

```javascript
const request = require('supertest');
const app = require('../app'); // tu Express app

test('GET /api/articles returns 200 and array', async () => {
  // Arrange & Act
  const res = await request(app).get('/api/articles');

  // Assert
  expect(res.status).toBe(200);
  expect(Array.isArray(res.body)).toBe(true);
});
```

**Por qué importa:** Los tests de endpoint son tests de integración — verifican que el código funciona de punta a punta (request → router → controller → DB → response). Son más valiosos que los unit tests puros porque prueban que las piezas encajan. En la Etapa 1 con Express + SQLite, podés escribir tests de integración reales sin mockear nada.

**Ejercicio:** Escribí tests de integración para los endpoints de FreePress: (1) `GET /api/articles` devuelve 200 con array, (2) `POST /api/articles` sin auth devuelve 401, (3) `POST /api/articles` con auth devuelve 201 y el artículo creado. Usá SQLite in-memory para la DB.

**Done cuando:** Podés escribir un test de endpoint para cualquier ruta de FreePress en menos de 5 minutos usando supertest. Entendés por qué los tests de integración son más valiosos que los unit tests aislados.

**Recursos:**
- [Supertest — GitHub](https://github.com/visionmedia/supertest) — README con ejemplos completos.
- Artículo: *"Testing Express.js Apps with Supertest"* — guía paso a paso para testear APIs.
