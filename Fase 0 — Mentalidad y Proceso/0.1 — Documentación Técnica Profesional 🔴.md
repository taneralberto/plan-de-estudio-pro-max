# 0.1 — Documentación Técnica Profesional 🔴

Antes de escribir una línea de código, un ingeniero de software sabe COMUNICAR qué va a construir, por qué, y cómo. La documentación no es burocracia — es la diferencia entre un equipo que avanza alineado y uno que construye lo mismo tres veces de tres formas distintas.

---

- [ ]  **PRD (Product Requirements Document)**

Qué es, cuándo se usa, quién lo escribe, qué contiene. La diferencia entre un PRD y una historia de usuario (una es un contrato de producto, la otra es una conversación). Cómo pasar de "el cliente quiere X" a un documento que TODO el equipo entienda igual, sin ambigüedades.

**Por qué importa:** Sin un PRD, cada dev interpreta los requisitos distinto. El resultado es que se construye lo correcto técnicamente pero lo incorrecto para el negocio. Un PRD bien escrito elimina esa categoría de errores antes de escribir una sola línea de código.

**Ejercicio:** Escribí un PRD para el editor WYSIWYG de FreePress. Debe incluir: el problema que resuelve, los usuarios afectados, los requisitos funcionales (qué hace), los no-funcionales (rendimiento, accesibilidad), los casos de uso principales, y lo que explícitamente está FUERA del scope. Cuando terminés, pasáselo a alguien (o a un agente de AI) y pedíle que lo lea y te diga qué preguntas le quedan sin responder — esas son tus ambigüedades.

**Done cuando:** Podés escribir un PRD para una feature nueva de FreePress que alguien que nunca vio el proyecto pueda leer y entender exactamente qué hay que construir, para quién, y qué queda fuera — sin hacerte ninguna pregunta.

**Recursos:**

- Libro: *Inspired* de Marty Cagan — el estándar de la industria para producto. No hace falta leerlo de tapa a tapa: los capítulos sobre discovery y definición de producto son los relevantes para este tema.
- [Lenny's Newsletter](https://www.lennysnewsletter.com/) — tiene PRD templates reales usados en Google, Stripe, Notion. Buscá "PRD template" en el sitio. Leer 2-3 PRDs reales vale más que cualquier teoría.

---

- [ ]  **RFC (Request for Comments)**

Qué es, por qué existe, cuándo escribir uno. La diferencia entre un RFC y un PRD: el PRD define QUÉ construir (producto), el RFC define CÓMO construirlo (ingeniería). Un RFC es una propuesta técnica que circula entre el equipo para recibir feedback ANTES de implementar — evita que tres devs implementen la misma cosa de tres formas distintas y descubran el desastre seis semanas después.

**Por qué importa:** Sin un proceso de RFC, las decisiones técnicas importantes se toman en conversaciones informales de Slack que nadie registra, nadie cuestiona, y nadie puede revisar después. El RFC fuerza a quien propone a pensar con rigor, y da al equipo un canal formal para objetar antes de que sea tarde.

**Ejercicio:** Escribí un RFC proponiendo la arquitectura del sistema de plugins de FreePress. Debe incluir: contexto del problema, propuesta técnica, alternativas que evaluaste y por qué las descartaste, trade-offs de la solución elegida, y preguntas abiertas. Usá el formato de Oxide RFD como referencia estructural.

**Done cuando:** Podés escribir un RFC técnico para una decisión de FreePress que incluya al menos dos alternativas evaluadas con sus trade-offs, y que alguien técnico pueda leerlo y darte feedback concreto sobre algo que no habías considerado.

**Recursos:**

- Buscá "Oxide RFD-0001" en GitHub ([oxidecomputer/rfd](https://github.com/oxidecomputer/rfd)) — el mejor ejemplo real de un proceso de RFCs en la industria. Leé el RFD-0001 que describe el proceso mismo.
- Artículo: *"Design Docs at Google"* de Malte Ubl — cómo Google usa design docs (su versión de RFCs). Buscarlo en [industrialempathy.com](https://www.industrialempathy.com/posts/design-docs-at-google/).

- [ ]  **ADR (Architecture Decision Records)**

Formato ligero para documentar DECISIONES arquitectónicas: qué se decidió, qué alternativas se evaluaron, por qué se eligió esta opción sobre las otras. Es el registro histórico que te salva cuando en 6 meses alguien pregunta "¿por qué usamos PostgreSQL y no MongoDB?" — en lugar de reconstruir la discusión de memoria, abrís el ADR y está todo ahí.

**Por qué importa:** Las decisiones arquitectónicas sin documentar se convierten en conocimiento tribal — solo las sabe quien estuvo en la reunión. Cuando esa persona se va, el conocimiento desaparece. Los ADRs convierten ese conocimiento en un activo del proyecto, no de una persona.

**Nota importante sobre los ADRs de FreePress:** El stack (Angular + NestJS + PostgreSQL + Redis) está pre-decidido antes de que estudies este tema. Los ADRs que vas a escribir son *retroactivos* — estás articulando y justificando decisiones ya tomadas, no tomándolas en tiempo real. Eso está bien como ejercicio: aprendés a estructurar el razonamiento arquitectónico. Pero en un proyecto real, un ADR se escribe ANTES de decidir. Cuando FreePress crezca y tengas que tomar nuevas decisiones (¿agregamos Redis Streams o RabbitMQ? ¿microservicios o seguimos con el monolito?), ahí vas a escribir ADRs reales.

**Ejercicio:** Escribí 3 ADRs para FreePress usando el formato MADR: (1) por qué PostgreSQL y no MongoDB, (2) por qué NestJS y no Express puro o Fastify, (3) la estructura de carpetas del proyecto. Para cada uno: contexto, decisión, alternativas consideradas, consecuencias positivas y negativas.

**Done cuando:** Podés escribir un ADR para una decisión nueva de FreePress (una que todavía no tomaste) en menos de 30 minutos, con alternativas reales evaluadas y consecuencias honestas — incluyendo las negativas de la opción elegida.

**Recursos:**

- [adr.github.io](https://adr.github.io) — el estándar de la industria, con historia, templates y herramientas.
- [Formato MADR](https://adr.github.io/madr/) (Markdown Architectural Decision Records) — el más práctico y el más adoptado hoy. Tiene templates listos para copiar.
- Herramienta: `adr-tools` — CLI para crear y gestionar ADRs desde la terminal. Opcional pero útil.

- [ ]  **Technical Writing — Escribir como ingeniero**

No es prosa literaria, es comunicación PRECISA. La escritura técnica tiene una sola métrica de éxito: ¿el lector puede hacer lo que necesita hacer sin preguntarte nada? Cómo escribir READMEs que alguien realmente lea, cómo documentar APIs (OpenAPI/Swagger), cómo escribir post-mortems, cómo comunicar decisiones técnicas a no-técnicos.

**Por qué importa:** El código que escribís lo leen máquinas y humanos. Las máquinas no se quejan si el código es oscuro — los humanos sí. Un README mal escrito hace que tu proyecto open source sea ignorado. Una documentación de API confusa hace que los consumidores te abran issues innecesarios. Technical writing es el multiplicador de todo lo demás que hacés.

**Ejercicio:** Escribí el README completo de FreePress. Debe incluir: qué es el proyecto en dos oraciones, cómo levantarlo localmente (paso a paso, sin asumir nada), cómo correr los tests, la estructura de carpetas explicada, y cómo contribuir. Criterio de calidad: dáselo a alguien que nunca vio el proyecto y mirá si puede levantarlo sin preguntarte nada.

**Done cuando:** Alguien que nunca vio FreePress puede leer el README y levantar el proyecto localmente sin hacerte ninguna pregunta. Ese es el único criterio que importa.

**Recursos:**

- Curso gratuito: [Google Technical Writing](https://developers.google.com/tech-writing) — dos módulos, gratuitos, estructurados, con ejercicios. Es el mejor recurso disponible para technical writing aplicado a software. Hacelo completo.
- Libro: *Docs for Developers* de Jared Bhatt — más práctico que teórico, orientado a devs que necesitan documentar proyectos reales.
