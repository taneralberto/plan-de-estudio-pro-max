# 1.2 — Sistemas Operativos (Profundizar) 🔴

El sistema operativo es el contrato entre tu código y el hardware. Cada vez que abrís un archivo, hacés un request HTTP, o corrés un proceso de Node.js, el SO está en el medio. Entender cómo funciona explica comportamientos que de otra forma parecen magia — o peor, bugs sin explicación.

---

## Procesos, Threads y Concurrencia

- [ ] **Procesos, Threads y Concurrencia**

Qué es un proceso vs un thread. Race conditions, deadlocks, mutexes, semáforos — y cuándo aparece cada problema. Por qué Node.js es single-threaded pero NO bloqueante (event loop). Worker threads en Node y cuándo realmente los necesitás.

**Por qué importa:** El event loop de Node.js es el motor de todo lo que hacés en el backend. Sin entender la diferencia entre un proceso y un thread, no podés razonar sobre por qué un Worker Thread resuelve CPU-bound work o por qué una promise no bloquea el loop. Cuando FreePress tenga que procesar imágenes o generar PDFs en el servidor, esto deja de ser teoría.

**Ejercicio:** Creá un programa en Node.js que demuestre un race condition (dos "threads" modificando una variable compartida sin sincronización). Luego arreglalo usando un Mutex manual con una promise. Documentá la diferencia en comportamiento.

**Done cuando:** Podés explicar por qué `setTimeout(fn, 0)` no ejecuta `fn` inmediatamente aunque el delay sea 0. Podés decir en qué casos conviene un Worker Thread en Node.js y en qué casos el event loop alcanza — con ejemplos concretos.

## Memory Management — Stack vs Heap

- [ ] **Memory Management — Stack vs Heap**

Qué es el stack y qué es el heap, y cómo el motor de JavaScript decide dónde va cada valor. Por qué JavaScript tiene garbage collection y C no. Qué es un memory leak en JS (sí, existen aunque haya GC), cómo se producen, y cómo detectarlos con Chrome DevTools.

**Por qué importa:** Los memory leaks en Node.js son más comunes de lo que parece — event listeners no removidos, closures que retienen referencias, caches que crecen sin límite. En FreePress, si el servidor crece en memoria con el tiempo y eventualmente cae, la causa es casi siempre un leak. Sin saber buscarlos, no podés arreglarlos.

**Ejercicio:** Creá un memory leak a propósito en Node.js (pista: usá un event emitter que nunca limpia sus listeners, o un array global que crece sin límite). Luego debuggealo con `node --inspect` y Chrome DevTools — tomá un heap snapshot, identificá el leak, arreglalo, y tomá otro snapshot para confirmar.

**Done cuando:** Podés abrir Chrome DevTools en la sección Memory, tomar un heap snapshot de una app Node.js, e identificar qué objetos están creciendo entre snapshots. Podés explicar por qué `let x = []` en el scope global de un módulo ES6 puede ser un leak si nunca se vacía.

**Recursos:**
- [Hunting Memory Leaks in Node.js](https://blog.appsignal.com/2020/09/30/finding-and-fixing-memory-leaks-in-nodejs.html) — guía práctica con herramientas reales

## File Systems y I/O

- [ ] **File Systems y I/O**

Cómo funciona el I/O por debajo: syscalls, buffers, streams. Por qué `fs.readFileSync` bloquea el event loop y `fs.createReadStream` no — y qué significa eso en producción. Streams en Node.js: Readable, Writable, Transform, Duplex.

**Por qué importa:** Si FreePress maneja uploads de archivos, exportaciones, o cualquier lectura de archivos en el servidor, la diferencia entre blocking y streaming es la diferencia entre una API que responde en 200ms y una que bloquea todos los requests durante 3 segundos mientras lee un archivo grande.

**Ejercicio:** Procesá un archivo de 1GB con `fs.readFileSync` y medí el consumo de RAM y el tiempo de bloqueo. Luego hacé lo mismo con `fs.createReadStream` y un Transform stream para procesar el contenido línea por línea. Documentá la diferencia en memoria y tiempo de respuesta.

Si alguien te muestra código que usa `readFileSync` en un endpoint de Express, podés explicar exactamente qué problema tiene.

## Signals y IPC (Inter-Process Communication)

- [ ] **Signals y IPC (Inter-Process Communication)**

Qué pasa cuando hacés `Ctrl+C` (SIGINT) o cuando un proceso recibe `SIGTERM`. Cómo se comunican procesos entre sí: pipes, sockets Unix, shared memory. Por qué Docker y Docker Compose usan SIGTERM para graceful shutdown y por qué ignorarlo es un bug en producción.

**Por qué importa:** En FreePress en producción, cuando hacés un deploy nuevo, el proceso viejo recibe SIGTERM y tiene que terminar limpiamente — cerrar conexiones a la DB, terminar requests en curso, limpiar caches. Si no manejás SIGTERM, tu app corta conexiones a mitad de un request y corrompés data. Esto no es avanzado: es lo básico de cualquier backend en producción.

**Ejercicio:** Agregá graceful shutdown a la app NestJS de FreePress — manejá `SIGTERM` y `SIGINT` para cerrar la conexión a PostgreSQL y Redis correctamente antes de que el proceso termine. Verificá que funciona corriendo la app con Docker y haciendo `docker stop` (que envía SIGTERM).

**Done cuando:** Podés explicar la diferencia entre `SIGTERM` y `SIGKILL` — por qué uno se puede capturar y el otro no. Podés mostrar el código de graceful shutdown en FreePress y explicar qué pasaría si ese código no existiera durante un deploy con zero-downtime.

**Recursos:**
- [Graceful shutdown in Node.js](https://snyk.io/blog/10-best-practices-to-containerize-nodejs-web-applications-with-docker/#:~:text=Use%20the%20host%20network%20when,when%20the%20application%20gets%20signaled.) — contexto en el mundo de containers
