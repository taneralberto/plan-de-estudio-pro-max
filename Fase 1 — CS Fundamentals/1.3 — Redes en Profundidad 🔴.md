# 1.3 — Redes en Profundidad 🔴 [ADVANCED]

DNS, TCP, TLS, HTTP — cada request pasa por múltiples capas. Sin entenderlas, debuggear es adivinanza.

- [ ] **Modelo TCP/IP capa por capa**

No memorizar las 7 capas OSI — entender las 4 que importan: Link (Ethernet, WiFi), Internet (IP, routing), Transport (TCP vs UDP — cuándo cada uno), Application (HTTP, DNS, TLS). Qué pasa REALMENTE cuando escribís una URL en el browser: DNS lookup, TCP handshake, TLS handshake, HTTP request/response.

**Por qué importa:** Cuando un request de FreePress falla con un error de red, hay varias capas donde puede estar el problema. Si no sabés qué capa hace qué, no sabés dónde buscar. ¿Es un problema de DNS? ¿De certificado TLS? ¿De timeout TCP? La respuesta cambia completamente el debugging.

**Ejercicio:** Usá Wireshark para capturar un request HTTP a un servidor local. Identificá los paquetes del TCP handshake (SYN, SYN-ACK, ACK), los paquetes TLS, y el HTTP request/response. Documentá qué ves en cada capa.

**Done cuando:** Podés explicar, sin mirar notas, qué pasa en la red cuando alguien escribe `https://freepress.app` y aprieta Enter — desde el DNS lookup hasta que el browser renderiza la página. Podés identificar en qué capa del modelo TCP/IP ocurre cada paso.

**Recursos:**
- [How HTTPS Works](https://howhttps.works/) — comic interactivo que explica el flujo completo visualmente
- [Wireshark](https://www.wireshark.org/) — la herramienta estándar para capturar y analizar tráfico de red

- [ ] **DNS — El sistema telefónico de internet**

Cómo funciona la resolución DNS de punta a punta: DNS recursivo, root servers, TLD servers, authoritative nameservers. TTL y por qué el caching importa. Por qué cambiar un registro DNS tarda horas (propagación). Registros A, CNAME, MX, TXT — cuándo se usa cada uno.

**Por qué importa:** Cuando deployás FreePress a producción, vas a configurar DNS. Cuando algo falla, la primera sospecha siempre es "¿será el DNS?". Sin entender TTL y propagación, vas a esperar horas sin saber si el cambio llegó o no. Y si configurás mal un CNAME, tu dominio puede quedar roto silenciosamente.

**Ejercicio:** Usá `dig` para resolver un dominio paso a paso — primero consultá un root server, luego el TLD, luego el authoritative. Después configurá un dominio custom para FreePress (o usá un subdominio de prueba) y verificá con `dig` que los registros A y CNAME apuntan donde corresponde.

**Done cuando:** Podés explicar por qué `dig google.com` puede darte una respuesta diferente a `dig @8.8.8.8 google.com`. Podés explicar qué significa un TTL de 300 segundos y cuándo tiene sentido bajar el TTL antes de migrar un dominio. Podés configurar los registros DNS necesarios para deployar FreePress a un proveedor de hosting.

**Recursos:**
- [DNS in One Picture](https://roadmap.sh/guides/dns-in-one-picture) — el flujo de resolución DNS completo en un diagrama
- [DNSChecker.org](https://dnschecker.org/) — para verificar propagación DNS desde múltiples ubicaciones geográficas

- [ ] **TLS/SSL — Cómo funciona HTTPS**

El handshake TLS: cómo el cliente y el servidor negocian una clave de sesión sin que nadie en el medio pueda leerla (Diffie-Hellman key exchange). Qué son los certificados digitales y las Certificate Authorities. Encriptación simétrica vs asimétrica — por qué TLS usa ambas y cuál es el rol de cada una. Por qué Let's Encrypt cambió internet. Qué es certificate pinning y cuándo tiene sentido.

**Por qué importa:** FreePress en producción necesita HTTPS. Sin entender TLS, vas a seguir tutoriales de Certbot sin saber qué estás haciendo — y cuando algo falle (certificado expirado, error de cadena de confianza, handshake failure), no vas a saber por dónde empezar. Además, las entrevistas senior siempre preguntan sobre HTTPS.

**Ejercicio:** Configurá TLS manualmente en FreePress con Let's Encrypt y Certbot (o con un certificado autofirmado para desarrollo). Inspecccioná el certificado con `openssl s_client -connect tudominio.com:443` y con las DevTools del browser. Identificá: quién firmó el certificado, cuándo expira, qué algoritmo de encriptación usa.

**Done cuando:** Podés explicar por qué HTTPS no significa que el servidor es "seguro" — solo que la conexión está encriptada. Podés explicar la diferencia entre un certificado autofirmado y uno firmado por una CA, y por qué el browser acepta uno y rechaza el otro. Podés renovar un certificado Let's Encrypt manualmente si el proceso automático falla.

**Recursos:**
- [The Illustrated TLS 1.3 Connection](https://tls13.xargs.org/) — la mejor explicación visual que existe, byte por byte
- [How Let's Encrypt Works](https://letsencrypt.org/how-it-works/) — la explicación oficial del proceso ACME

- [ ] **Latencia, Throughput y Bandwidth — No son lo mismo**

La diferencia entre latencia (tiempo hasta el primer byte), throughput (datos por segundo), y bandwidth (capacidad máxima del canal) — y por qué optimizar uno no necesariamente mejora los otros. CDNs y cómo reducen latencia acercando los datos al usuario. Edge computing. Cómo medir latencia real de una API.

**Por qué importa:** "La API está lenta" no es suficiente información para arreglar nada. Necesitás saber si es latencia (tarda en responder el primer byte) o throughput (responde rápido pero transfiere poco a poco). Un CDN resuelve latencia geográfica pero no resuelve queries lentas en la base de datos. Sin estas distinciones, las optimizaciones son a ciegas.

**Ejercicio:** Medí la latencia y el throughput de los principales endpoints de FreePress con `k6` o `wrk`. Después agregá un header de caché a las respuestas de contenido estático y medí de nuevo. Documentá qué cambió y por qué.

**Done cuando:** Podés mirar los resultados de una prueba de carga y distinguir si el cuello de botella es latencia o throughput. Podés explicar por qué servir assets estáticos de FreePress desde un CDN reduce la latencia para usuarios en otro país, pero no cambia el tiempo de una query a la DB.

**Recursos:**
- [High Performance Browser Networking](https://hpbn.co/) — de Ilya Grigorik (Google), gratis online. El capítulo de latencia es el mejor que existe
- [k6](https://k6.io/) — herramienta moderna de load testing para medir throughput y latencia de APIs
