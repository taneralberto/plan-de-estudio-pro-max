# 1.5 — Linux y Shell Productivo 🟡 [ADVANCED]

Toda la Fase 4 (Docker, CI/CD, cloud) asume que sabés moverte en un servidor Linux.

- [ ] **El modelo Unix — Todo es un archivo**

El principio fundamental de Unix: todo es un archivo — sockets, dispositivos, pipes, directorios. El filesystem como interfaz universal. La jerarquía de directorios: `/etc` (configuración del sistema), `/var` (datos variables — logs, bases de datos), `/usr` (programas de usuario), `/proc` (interfaz al kernel en tiempo real — `/proc/cpuinfo`, `/proc/meminfo`). Procesos y señales: qué es un proceso (PID, PPID, grupos de procesos), señales más importantes (`SIGTERM` — pedir educadamente que termine, `SIGKILL` — terminar forzado sin cleanup, `SIGHUP` — recargar configuración, `SIGINT` — Ctrl+C). Por qué la diferencia entre `SIGTERM` y `SIGKILL` importa: un proceso de NestJS que recibe `SIGTERM` puede cerrar conexiones de base de datos limpiamente; `SIGKILL` no le da esa oportunidad.

**Por qué importa:** Toda la Fase 4 de este plan (Docker, CI/CD, cloud) asume que sabés moverte en un servidor Linux. Sin el modelo Unix, vas a copiar comandos de Stack Overflow sin entender qué hacen. Cuando el container de FreePress no para limpiamente y quedás con conexiones de DB huérfanas, o cuando un proceso zombie no podés matar, o cuando querés saber qué proceso está usando el puerto 3000 — todos esos problemas tienen solución en 10 segundos si entendés el modelo Unix.

**Ejercicio:** Explorá `/proc` en un servidor Linux (o en WSL): leé `/proc/self/status` para ver los detalles de tu proceso actual. Luego iniciá el backend de FreePress, encontrá su PID con `ps aux | grep node`, explorá `/proc/<PID>/fd` para ver los file descriptors abiertos (incluyendo sockets de DB), y mandá `SIGTERM` manualmente — verificá que el proceso cierra limpiamente con el handler de shutdown de NestJS antes de que el proceso termine.

**Done cuando:** Podés diagnosticar en menos de 2 minutos qué proceso está usando un puerto específico, cómo está usando recursos (CPU, memoria, file descriptors), y cómo terminarlo limpiamente vs forzado. Podés explicar por qué `docker stop` usa `SIGTERM` (con timeout) antes de `SIGKILL`, y qué tiene que hacer tu aplicación para aprovechar ese tiempo.

**Recursos:**
- [*"The Linux Command Line"*](https://linuxcommand.org/tlcl.php) de William Shotts — el mejor libro introductorio a Linux y la shell. GRATIS online. Empezá por acá.
- [*"The Art of Unix Programming"*](http://www.catb.org/esr/writings/taoup/html/) de Eric S. Raymond — el libro que explica la filosofía Unix (todo es un archivo, programas pequeños que hacen una cosa bien, pipes). Gratis online. Más conceptual que práctico.
- [tldr-pages](https://tldr.sh/) — versiones resumidas y prácticas de los man pages. `tldr grep`, `tldr find`, `tldr ps` — útil para recordar flags rápidamente.

- [ ] **Pipes, redirección y composición de comandos**

stdin, stdout, stderr — los tres streams estándar de todo proceso Unix. Redirección: `>` (sobreescribir stdout a archivo), `>>` (append), `2>` (redirigir stderr), `2>&1` (redirigir stderr al mismo lugar que stdout). Pipes (`|`): conectar el stdout de un comando con el stdin del siguiente. La filosofía Unix de composición: programas pequeños que hacen una cosa bien, combinados con pipes para resolver problemas complejos. Comandos fundamentales de composición: `grep` (filtrar líneas), `awk` (transformar columnas), `sed` (editar líneas), `sort`, `uniq`, `wc`, `cut`, `xargs`. Process substitution (`<(comando)`). `jq` para manipular JSON en la terminal — indispensable en 2024.

**Por qué importa:** En producción, FreePress genera logs en JSON. Sin saber filtrar y manipular esos logs desde la terminal, estás ciego cuando algo falla y no podés abrir Kibana o Grafana. Con pipes y `jq`, podés en 30 segundos filtrar todos los logs de error del último minuto, extraer los requestIds únicos, y contar cuántos usuarios distintos fueron afectados — todo sin salir de la terminal.

**Ejercicio:** Dado el log de FreePress en formato JSON (structured logging de pino), escribí un one-liner de terminal que: filtre solo los logs con `"level": "error"` de los últimos 5 minutos (usando `jq` y comparación de timestamps), extraiga el campo `requestId` de cada uno, los ordene y cuente las ocurrencias únicas. Luego creá un script bash que haga lo mismo y lo envíe como reporte por email usando `curl` con la API de SendGrid.

**Done cuando:** Podés analizar los logs de producción de FreePress directamente desde la terminal sin herramientas gráficas. Dado un log de 100.000 líneas en JSON, podés extraer en menos de 30 segundos: cuántos errores hubo, qué endpoints fallaron más, y qué usuarios fueron afectados — usando solo `jq`, `grep`, `sort` y `uniq`.

**Recursos:**
- [*"jq Manual"*](https://jqlang.github.io/jq/manual/) — la referencia oficial de `jq`. Instalalo hoy (`brew install jq` / `apt install jq`) y usalo en cada respuesta JSON de API que recibas.
- [*"The Art of Command Line"*](https://github.com/jlevy/the-art-of-command-line) — guía práctica curada de los comandos más útiles, con énfasis en los que marcan la diferencia. Un must-read.
- [explainshell.com](https://explainshell.com/) — pegá cualquier comando complejo y te explica cada parte. Indispensable para entender los one-liners que encontrás en Stack Overflow.

- [ ] **Bash scripting y automatización**

Variables, condicionales (`if`, `case`), loops (`for`, `while`), funciones, argumentos posicionales (`$1`, `$2`, `$@`). Exit codes: `$?` para capturar el código de salida del último comando — `0` es éxito, cualquier otro valor es error. `set -euo pipefail`: las tres opciones que todo script bash debería tener al principio (`-e` aborta si algún comando falla, `-u` trata variables no definidas como error, `-o pipefail` propaga errores en pipes). Cron jobs: sintaxis del crontab, `crontab -e`, logging de cron jobs. `systemd` services: cómo registrar un proceso como servicio del sistema para que arranque automáticamente y se reinicie si falla.

**Por qué importa:** En el CI/CD de FreePress vas a escribir y leer scripts bash constantemente. Un script sin `set -e` puede fallar silenciosamente en el paso 3 y seguir ejecutando los pasos 4-10 como si nada, resultando en un deploy corrupto que nadie detecta. Un cron job de backup que falla sin logear el error es un backup que no existe cuando lo necesitás. Saber escribir scripts bash robustos no es un skill de sysadmin — es un skill de cualquier developer que trabaja en sistemas reales.

**Ejercicio:** Escribí un script de backup de FreePress en bash con todas las buenas prácticas: (1) `set -euo pipefail` al inicio; (2) función de cleanup que se ejecuta con `trap` en EXIT (borra archivos temporales aunque el script falle); (3) backup de la base de datos PostgreSQL con `pg_dump`, compresión con `gzip`, y upload a S3 con `aws s3 cp`; (4) logging con timestamp a un archivo; (5) notificación de éxito o fallo a Slack via webhook con `curl`. Registralo como cron job que corre cada noche a las 2am UTC.

**Done cuando:** Podés leer un script bash de producción de 100 líneas y detectar al menos 3 problemas comunes de robustez (falta de `set -e`, variables no quoted, falta de manejo de errores). Podés escribir un script bash desde cero que maneje errores correctamente, limpie recursos con `trap`, y logee apropiadamente.

**Recursos:**
- [*"The Linux Command Line"*](https://linuxcommand.org/tlcl.php) de William Shotts — los capítulos de scripting (24-36) son la mejor introducción práctica a bash. Gratis online.
- [*"Shell Check"*](https://www.shellcheck.net/) — linter online para scripts bash que detecta errores comunes. Usarlo en cada script que escribas.
- [*"Bash Strict Mode"*](http://redsymbol.net/articles/unofficial-bash-strict-mode/) — el artículo que explica `set -euo pipefail` y por qué es obligatorio. Una página, fundamental.
