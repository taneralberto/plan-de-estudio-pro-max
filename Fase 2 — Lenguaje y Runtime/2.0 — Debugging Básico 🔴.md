# 2.0 — Debugging Básico 🔴 [CORE]

Antes de escribir código complejo, necesitás saber encontrar por qué NO funciona. El debugging NO es "probar cosas al azar hasta que ande" — es una habilidad sistemática.

---

- [ ] **Chrome DevTools — Lo básico indispensable**

Breakpoints: cómo ponerlos, qué pasa cuando el código se detiene. Call Stack: ver la cadena de llamadas que te llevó hasta ahí. Scope: qué variables están disponibles en cada momento. Step over vs Step into vs Step out: cuándo usar cada uno. Watch expressions: monitorear valores específicos mientras avanzás.

**Por qué importa:** Cada vez que usás `console.log` para debuggear, estás trabajando 10x más lento de lo que podrías. Los breakpoints te dejan inspeccionar estado en cualquier punto, pausar, avanzar paso a paso, y entender exactamente qué está pasando. Sin esta habilidad, debugging es adivinar.

**Ejercicio:** Agregá un bug intencional en FreePress (ej: una función que retorna undefined en ciertos casos). Usá Chrome DevTools para: (1) poner un breakpoint donde se produce el undefined, (2) inspeccionar el call stack, (3) encontrar el origen del problema en menos de 5 minutos, sin `console.log`.

**Done cuando:** Podés debuggear un problema en código que no conocés usando breakpoints, call stack y scope inspection — sin agregar ni un solo `console.log`. Podés explicar la diferencia entre Step Over y Step Into y cuándo usar cada uno.

**Recursos:**
- [Chrome DevTools Docs — Pause code with breakpoints](https://developer.chrome.com/docs/devtools/javascript/breakpoints/) — la guía oficial, clara y completa
- Video: "Chrome DevTools Crash Course" en YouTube (cualquier versión reciente, lo básico no cambia)

---

- [ ] **Console methods que no son console.log**

`console.log` es el martillo que hace todo parecer un clavo. Pero existen otros: `console.table()` para ver arrays de objetos en formato tabla, `console.dir()` para inspeccionar objetos con profundidad, `console.time()` / `console.timeEnd()` para medir performance, `console.assert()` para validar condiciones, `console.group()` para organizar logs relacionados.

**Por qué importa:** Si estás loggeando arrays de 100 objetos con `console.log`, estás perdiendo tiempo leyendo basura. `console.table(data)` te da una tabla legible en segundos. Si estás midiendo performance con `Date.now()` manual, estás reinventando `console.time()`. Estas herramientas existen para hacerte más eficiente — usalas.

**Ejercicio:** Tomá un array de objetos de FreePress (ej: lista de artículos). Mostralo con `console.log` y después con `console.table`. Medí cuánto tarda un loop con `console.time`. Escribí una aserción con `console.assert` que falle intencionalmente para ver cómo se ve.

**Done cuando:** Podés elegir el método de console correcto para cada situación sin dudar. No usás `console.log` para todo — sabés cuándo `table`, `dir`, `time` o `assert` son mejores opciones.

**Recursos:**
- [MDN — Console API](https://developer.mozilla.org/en-US/docs/Web/API/console) — referencia completa de todos los métodos

---

- [ ] **Node.js debugging — backend sin console.log**

`node --inspect`: cómo conectar Chrome DevTools a un proceso de Node. Debugging en VS Code: configuración de launch.json, breakpoints en el editor, debug console. Debugging de tests: cómo debuggear tests de Jest o Vitest paso a paso. Debugging de procesos remotos: conectar a un servidor en producción (con precaución).

**Por qué importa:** Tu backend de FreePress corre en Node. Si solo sabés debuggear frontend, estás ciego cuando el problema está en el servidor. El debugging de Node es casi igual al de browser — pero necesitás saber cómo conectar las herramientas.

**Ejercicio:** Configurá debugging en VS Code para el backend de FreePress. Agregá un bug en un endpoint (ej: retorna 500 en ciertas condiciones). Usá el debugger de VS Code para encontrar el origen del problema, con breakpoints y variable inspection.

**Done cuando:** Tenés configurado debugging en VS Code para el backend de FreePress y podés debuggear un problema en un endpoint sin agregar `console.log` al código.

**Recursos:**
- [VS Code — Node.js Debugging](https://code.visualstudio.com/docs/nodejs/nodejs-debugging) — guía oficial de VS Code
- [Jest — Debugging Tests](https://jestjs.io/docs/troubleshooting#debugging-tests) — cómo debuggear tests de Jest

- [ ] **La metodología de debugging — No es adivinar**

El error común: "probá esto, probá aquello, reiniciá, capaz es esto". Eso NO es debugging — es shooting in the dark.

**El método correcto:**
1. **Reproducir** el bug de forma consistente (si no podés reproducirlo, no podés debuggearlo)
2. **Reducir** el problema: aislá el código mínimo que reproduce el bug
3. **Formular hipótesis** de qué está mal (no "capaz algo", sino "probablemente X retorna undefined cuando Y")
4. **Testear la hipótesis** con el debugger (poner breakpoint donde X se calcula, verificar)
5. **Repetir** si la hipótesis estaba mal, pero ahora con más información

**Por qué importa:** Un bug de 2 horas se convierte en 30 minutos cuando usás metodología. Sin metodología, podés pasar días probando cosas sin avanzar. El debugger es la herramienta, pero la metodología es lo que te hace eficiente.

**Ejercicio:** Tomá un bug real o inventado en FreePress. Aplicá la metodología paso a paso: (1) escribí pasos exactos para reproducir, (2) aislá el código mínimo, (3) escribí tu hipótesis ANTES de usar el debugger, (4) verificá la hipótesis con breakpoints, (5) documentá qué aprendiste.

**Done cuando:** Podés explicar el método de debugging de 5 pasos y aplicarlo a un problema nuevo sin "probar cosas al azar". Cuando te encontrás con un bug, tu primer impulso es "reproducir, reducir, hipótesis", no "capaz si reinicio funciona".

**Recursos:**
- Artículo: *"Debugging: The 9 Indispensable Rules"* de David Agans — el libro corto que define la metodología estándar de debugging
- Video: "How to Debug Anything" por ThePrimeagen — enfoque práctico y directo
