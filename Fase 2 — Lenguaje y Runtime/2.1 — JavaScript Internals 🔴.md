## El Event Loop — De verdad

- [ ]  No "JavaScript es asíncrono". Entender la arquitectura completa: Call Stack (donde se ejecuta el código sincrónico, uno a la vez), Web APIs / libuv (donde viven los timers, I/O, y eventos del sistema operativo), Callback Queue (también llamada Task Queue o Macrotask Queue — setTimeout, setInterval, I/O callbacks), y Microtask Queue (Promises, queueMicrotask, MutationObserver). El loop en sí: vaciar el call stack → procesar TODOS los microtasks → tomar UN macrotask → repetir. Por eso `Promise.resolve()` se ejecuta ANTES que `setTimeout(fn, 0)`. El event loop de Node.js (libuv con múltiples fases: timers, I/O, idle, poll, check, close) vs el del browser.

**Por qué importa:** Cada vez que escribís código asíncrono — que es todo el tiempo en Node.js — estás interactuando con el event loop. Si no entendés el orden de ejecución, vas a tener bugs de race conditions y timing que son casi imposibles de debuggear sin este modelo mental. Además, es la pregunta técnica más común en entrevistas senior de JavaScript.

**Ejercicio:** Antes de correr cada bloque, predecí en papel el orden exacto del output. Después corrélo y verificá. Hacé esto con al menos 10 ejemplos que mezclen: `Promise.resolve()`, `setTimeout(fn, 0)`, `await`, `queueMicrotask()`, y `.then()` encadenados. Si alguna predicción falla, no sigas hasta entender por qué.

## V8 Engine — Cómo JavaScript se convierte en código máquina

- [ ]  El pipeline completo de V8: Parsing del source code → Abstract Syntax Tree (AST) → Ignition (el intérprete bytecode, que ejecuta rápido sin optimizar) → TurboFan (el compilador JIT que optimiza el código caliente). Qué son las "hidden classes" (V8 asigna una clase interna a cada objeto según su shape — la forma de sus propiedades) y por qué agregar propiedades en orden distinto al de la función constructora crea hidden classes distintas y mata la performance. Qué es inlining (V8 copia el cuerpo de una función pequeña en el lugar donde se llama para evitar el overhead de la llamada).

**Por qué importa:** JavaScript no es lento — el código JavaScript mal escrito es lento. Entender cómo V8 optimiza (y deoptimiza) tu código te permite escribir JavaScript que el motor puede optimizar bien. No es optimización prematura — es escribir código que no sabotea al compilador.

**Ejercicio:** Creá dos versiones de la misma función: una que cree objetos con propiedades en orden consistente, otra que las agregue dinámicamente en orden variable. Medí la diferencia de performance con `console.time` o con el Performance tab de DevTools con al menos 1 millón de iteraciones. Documentá la diferencia y explicá por qué existe.

## Closures, Prototypes y `this`

- [ ]  **Closures:** no es magia, es una consecuencia del lexical scope. Una función "cierra" sobre las variables del scope donde fue DEFINIDA (no donde se ejecuta). Por eso una función que devuelve otra función puede acceder a las variables de la función externa incluso después de que esta terminó de ejecutar. **Prototypes:** cómo funciona realmente la herencia en JS. Cada objeto tiene un `[[Prototype]]` interno que apunta a otro objeto. Cuando accedés a una propiedad que no existe en el objeto, JS sube la prototype chain hasta encontrarla o llegar a `null`. `class` en JS es syntactic sugar sobre prototypes — no hay clases reales. **`this`:** las 4 reglas en orden de precedencia: new binding (`new Fn()` → `this` es el nuevo objeto), explicit binding (`call`, `apply`, `bind`), implicit binding (el objeto antes del punto: `obj.fn()` → `this` es `obj`), default binding (en strict mode: `undefined`; en sloppy mode: `globalThis`).

**Por qué importa:** Closures son la base de patrones fundamentales de JavaScript: módulos, memoización, currying, React hooks. Prototypes son cómo REALMENTE funciona la herencia — si no los entendés, `class` es una caja negra. Y `this` es la fuente de bugs más confusa de JavaScript para la mayoría de devs.

**Ejercicio:** Implementá un sistema de herencia completo SIN usar `class` — solo con prototypes, `Object.create()`, y funciones constructoras. Después demostrá que el comportamiento es idéntico a la versión con `class`. Esto fuerza a entender qué hace `class` por debajo.

## Module Systems — CommonJS vs ESM

- [ ]  Por qué existen dos sistemas de módulos en JavaScript: CommonJS (`require()`, `module.exports`) nació en Node.js en 2009 porque los browsers no tenían módulos. ESM (`import`/`export`) es el estándar oficial del lenguaje, introducido en ES2015. La diferencia fundamental: CJS es síncrono y evalúa los módulos en runtime; ESM es estático y permite análisis en tiempo de compilación (lo que habilita tree shaking). Cómo interoperan (y por qué es un dolor). Qué es `"type": "module"` en `package.json`. Exports map en `package.json` para exponer distintos entry points. Qué son los "dual packages" (un paquete que funciona con CJS y ESM) y por qué son complicados.

**Por qué importa:** En Node.js en 2025 conviven los dos sistemas. Vas a encontrar librerías que solo soportan ESM (rompiendo código CJS existente), errores crípticos de "Cannot use import statement in a module", y configuraciones de `tsconfig.json` y bundlers que dependen de entender esta diferencia. Sin este conocimiento, esos errores son cajas negras.

**Ejercicio:** Creá un paquete npm pequeño (puede ser una función de utilidad para FreePress) que funcione correctamente tanto con `require()` como con `import` — un dual package real. Configurá el `package.json` con el exports map correcto. Verificá que funciona en ambos contextos.
