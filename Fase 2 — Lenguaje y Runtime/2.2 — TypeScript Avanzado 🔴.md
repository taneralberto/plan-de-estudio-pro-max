# 2.2 — TypeScript Avanzado 🔴

TypeScript no es "JavaScript con tipos". Es un lenguaje con su propio sistema de tipos — un sistema Turing-completo que tiene su propio "runtime" a nivel de compilación. La mayoría de devs usa el 20% de TypeScript y cree que lo domina. Este archivo cubre el 80% restante: el sistema de tipos en profundidad, el compilador, y cómo TypeScript interactúa con el ecosistema de librerías.

---

## Type System en profundidad

- [ ]  **Generics:** funciones y tipos parametrizados por tipo, como templates en otros lenguajes. `function identity<T>(arg: T): T` es el ejemplo mínimo — pero el uso real es mucho más sofisticado. **Conditional types:** `T extends U ? X : Y` — tipos que se resuelven de forma diferente según una condición. Base del tipo `NonNullable`, `ReturnType`, `Parameters` de la stdlib. **Mapped types:** transformar cada propiedad de un tipo existente — `{ [K in keyof T]: ... }`. Base de `Partial<T>`, `Required<T>`, `Readonly<T>`. **Template literal types:** `type EventName = \`on${Capitalize<string>}\`` — tipos construidos como strings con interpolación. **Type narrowing:** cómo TypeScript reduce el tipo de una variable dentro de un bloque (`typeof`, `instanceof`, `in`, discriminated unions, type guards custom con `is`). **Discriminated unions:** el patrón más poderoso de TypeScript — una unión de tipos que comparten una propiedad discriminante, que permite narrowing exhaustivo con `never`.

**Por qué importa:** Con el sistema de tipos avanzado podés hacer que errores que antes se detectaban en runtime (o nunca) se detecten en tiempo de compilación. Un event emitter type-safe, una función que garantiza que no olvidaste manejar todos los casos de un estado — ese nivel de seguridad solo es posible si entendés el sistema de tipos en profundidad, no solo las anotaciones básicas.

**Ejercicio:** Implementá un event emitter completamente type-safe en TypeScript usando generics y mapped types. Debe cumplir: (1) `emitter.on('click', handler)` solo acepta handlers con la firma correcta para el evento 'click', (2) TypeScript debe dar error si intentás emitir un evento que no existe, (3) el tipo del payload del handler debe inferirse automáticamente del nombre del evento — sin casteos, sin `any`.

**Done cuando:** Podés implementar los tipos de utilidad de TypeScript desde cero (`Partial<T>`, `Required<T>`, `ReturnType<T>`, `DeepReadonly<T>`) sin mirar la implementación oficial, y podés explicar cómo funciona cada uno.

**Recursos:**

- [Type Challenges](https://github.com/type-challenges/type-challenges) — colección de ejercicios para dominar el type system, organizados por dificultad. Empezá por "easy" y avanzá. Es el recurso más efectivo para este tema.
- Libro: *Effective TypeScript* de Dan Vanderkam — 62 reglas concretas con explicación de por qué. No hace falta leerlo lineal — usalo como referencia cuando algo del type system no te cierra.
- [TypeScript Playground](https://www.typescriptlang.org/play) — para experimentar con tipos sin setup. Indispensable mientras estudiás este tema.

## TypeScript Compiler — `tsconfig.json` que sabés lo que estás haciendo

- [ ]  Las opciones de compilación que más importan y qué hacen exactamente: `strict` (habilita un conjunto de checks estrictos — `strictNullChecks`, `noImplicitAny`, `strictFunctionTypes` y otros — siempre activado en proyectos serios), `target` (a qué versión de JS compila — no confundir con `lib`), `module` (qué sistema de módulos genera el output — `commonjs`, `esnext`, etc.), `moduleResolution` (cómo TypeScript resuelve imports — `node`, `bundler`, `node16` — fuente de muchos errores crípticos), `paths` (alias de imports — `@app/*` en lugar de `../../../`), `declaration` (genera `.d.ts` — necesario para publicar librerías), `composite` y `references` (para monorepos con múltiples proyectos TypeScript que se referencian entre sí). Por qué `skipLibCheck: true` es un parche que esconde errores reales.

**Por qué importa:** Un `tsconfig.json` mal configurado hace que TypeScript dé falsos negativos (errores donde no los hay) o falsos positivos (no reporta errores reales). FreePress tiene Angular en el frontend y NestJS en el backend — dos `tsconfig` distintos con necesidades distintas. Sin entender las opciones, estás copiando configuraciones de Stack Overflow sin saber qué hacen.

**Ejercicio:** Auditá el `tsconfig.json` de FreePress (tanto el del frontend Angular como el del backend NestJS). Para cada opción presente, documentá en un comentario qué hace y por qué está ahí. Para cada opción que NO esté y que debería estar (según las best practices de cada framework), agregala con justificación. Al final, tenés que poder defender cada línea del `tsconfig`.

**Done cuando:** Podés crear un `tsconfig.json` desde cero para un proyecto Node.js nuevo sin copiar de ningún lado, explicar qué hace cada opción que elegiste, y resolver el error `"Cannot find module" or its corresponding type declarations` cuando aparece — sin buscar en Stack Overflow.

**Recursos:**

- [TSConfig Reference](https://www.typescriptlang.org/tsconfig) — la documentación oficial de cada opción del compilador. La referencia definitiva.
- Artículo: [*"The TSConfig Cheat Sheet"*](https://www.totaltypescript.com/tsconfig-cheat-sheet) de Matt Pocock — las opciones más importantes explicadas en contexto. Excelente punto de partida.

## Declaration Files y `@types`

- [ ]  Qué son los archivos `.d.ts`: descripciones de tipos para código que no tiene tipos propios — sea JavaScript puro, una librería en C, o código generado. Cómo TypeScript los busca: primero en la librería misma (campo `types` o `typings` en `package.json`), luego en `@types/<package>` de DefinitelyTyped. Qué es DefinitelyTyped: el repositorio de la comunidad con tipos para miles de librerías que no los incluyen nativamente. Cuándo y cómo escribir tus propias declarations: para módulos sin tipos, para extender tipos existentes (module augmentation), para declarar variables globales que vienen del entorno (ej: variables de entorno tipadas).

**Por qué importa:** Vas a trabajar con librerías que no tienen tipos, con código JavaScript legacy que necesitás integrar en un proyecto TypeScript, o con APIs globales que necesitás tipar. Sin entender `.d.ts`, esas situaciones terminan en `// @ts-ignore` o `any` — que es exactamente lo que TypeScript intenta evitar.

**Ejercicio:** Tomá una librería de JavaScript sin tipos que uses en FreePress (o buscá una pequeña en npm que no tenga `@types`). Escribí el archivo `.d.ts` completo para ella — no descargues los tipos, escribilos vos a partir de la documentación y el código fuente. Después compará tu versión con la de DefinitelyTyped si existe.

**Done cuando:** Podés escribir un `.d.ts` para una librería JavaScript sin tipos, agregar propiedades a un tipo existente de una librería usando module augmentation, y explicar la diferencia entre un `declare module`, un `declare global`, y un `declare namespace`.

**Recursos:**

- [TypeScript docs — Declaration Files](https://www.typescriptlang.org/docs/handbook/declaration-files/introduction.html) — la guía oficial. La sección "Do's and Don'ts" es especialmente valiosa.
- [DefinitelyTyped](https://github.com/DefinitelyTyped/DefinitelyTyped) — el repositorio. Vale leer algunos `.d.ts` de librerías que ya conocés para ver cómo se estructuran.
- Artículo: [*"TypeScript Module Augmentation"*](https://www.typescriptlang.org/docs/handbook/declaration-merging.html) — documentación oficial sobre cómo extender tipos existentes.
