# 2.3 — Build Tools y Module Bundlers 🟡 [ADVANCED]

El código que escribís no es el código que se ejecuta. Transpilación, bundling, minificación, tree shaking — entender este pipeline no es opcional.

- [ ]  **Qué hace un bundler y POR QUÉ existe**

El problema original: los browsers no soportaban módulos (`import`/`export`) y cargar 200 archivos JavaScript por separado era lento (una request HTTP por archivo en la era de HTTP/1.1). La solución: tomar todos los módulos y empaquetarlos en uno (o pocos) archivos. La evolución: Webpack (2012, el primero en hacerlo bien — configuración compleja pero todopoderoso), Rollup (2015, optimizado para librerías — mejor tree shaking), esbuild (2020, escrito en Go — 10-100x más rápido que Webpack), Vite (2021, usa esbuild en dev y Rollup en prod — el mejor DX actual). Conceptos que todo bundler implementa: **tree shaking** (eliminar código que nunca se usa — solo funciona con ESM estático), **code splitting** (dividir el bundle en chunks que se cargan bajo demanda), **lazy loading** (cargar un chunk solo cuando se necesita, no al inicio).

**Por qué importa:** Cuando el bundle de FreePress pesa demasiado, cuando el build tarda 3 minutos, o cuando una librería que removiste sigue apareciendo en el output, necesitás entender el pipeline para diagnosticar. Sin este conocimiento, esas situaciones son cajas negras que resolvés reiniciando el servidor y rezando.

**Ejercicio:** Analizá el bundle de FreePress con un bundle analyzer (webpack-bundle-analyzer para Angular, rollup-plugin-visualizer para Vite). Identificá las 3 dependencias más pesadas. Para cada una, respondé: ¿es necesaria en ese tamaño? ¿Se puede hacer lazy? ¿Hay una alternativa más liviana? Documentá el análisis.

**Done cuando:** Podés mirar el output de un bundle analyzer y explicar por qué una dependencia aparece en el bundle principal en lugar de en un chunk lazy, qué es tree shaking y por qué no funcionó para esa dependencia, y cómo arreglarlo.

**Recursos:**

- [Webpack Bundle Analyzer](https://github.com/webpack-contrib/webpack-bundle-analyzer) — visualizador para Angular/Webpack.
- [rollup-plugin-visualizer](https://github.com/btd/rollup-plugin-visualizer) — visualizador para Vite/Rollup.

- [ ]  **Vite en profundidad**

Por qué Vite es dramáticamente más rápido que Webpack en desarrollo: en dev usa ESM nativo del browser — no bundlea nada, el browser pide cada módulo directamente al servidor, y Vite solo transpila lo que se pide (on-demand). En producción usa Rollup para generar un bundle optimizado. Hot Module Replacement (HMR): cómo Vite reemplaza solo el módulo que cambió sin recargar la página, preservando el estado de la aplicación. Plugin system: cómo extender Vite con plugins de Rollup (compatibles) y plugins propios. Configuración para casos reales: proxy para APIs, variables de entorno, configuración de paths, optimización de dependencias.

**Por qué importa:** FreePress usa Angular con un toolchain basado en esbuild (Angular 17+). Entender el modelo de Vite (que es el mismo modelo conceptual) te permite diagnosticar problemas de build, configurar el entorno correctamente, y entender por qué el dev server es tan rápido comparado con Webpack clásico.

**Ejercicio:** Configurá Vite para el frontend de FreePress (si no lo usás ya). Implementá: (1) un proxy para las llamadas a la API de NestJS en desarrollo, (2) variables de entorno tipadas con TypeScript (`import.meta.env.VITE_API_URL`), (3) un alias de paths (`@app/*` apuntando a `src/`). Documentá cada configuración con el motivo.

**Done cuando:** Podés explicar por qué Vite no bundlea en desarrollo (y qué implicaciones tiene eso), configurar un `vite.config.ts` desde cero para un proyecto real con proxy y aliases, y diagnosticar por qué un módulo no se está actualizando con HMR.

**Recursos:**

- [Vite docs — Why Vite](https://vitejs.dev/guide/why.html) — la documentación oficial explicando el modelo. Leer esta sección primero, antes de cualquier otra cosa de Vite.
- [Vite docs — Config Reference](https://vitejs.dev/config/) — referencia completa de configuración. Usarla cuando necesites una opción específica.

- [ ]  **Monorepos — Turborepo, Nx, pnpm workspaces**

El problema que resuelven: cuando tenés múltiples paquetes o aplicaciones relacionadas (en el caso de FreePress: el frontend Angular, el backend NestJS, los tipos compartidos, la CLI, la librería de componentes), mantenerlos en repos separados genera fricción — cambios que afectan a varios paquetes requieren PRs coordinados, versionado sincronizado a mano, y `npm link` para desarrollo local. Un monorepo los pone todos en el mismo repo con herramientas que los gestionan. **pnpm workspaces:** la base — define los paquetes del monorepo, gestiona dependencias compartidas evitando duplicados. **Turborepo:** task orchestration — corre builds, tests y lints en paralelo respetando dependencias entre paquetes, con caché inteligente. **Nx:** similar a Turborepo pero con más herramientas integradas — generators, affected analysis (solo corre tests de lo que cambió), visualización del grafo de dependencias.

**Por qué importa:** FreePress está diseñado para crecer en mini-proyectos satélite (`freepress-cli`, `fp-ui`, `fp-ai-assistant`) con tipos compartidos entre frontend y backend. Sin un monorepo, compartir tipos requiere publicar un paquete npm para cada cambio. Con un monorepo, los tipos compartidos son un paquete interno que cualquier parte del proyecto puede importar directamente.

**Ejercicio:** Convertí FreePress en un monorepo con pnpm workspaces y Turborepo. La estructura mínima: `packages/shared` (tipos TypeScript compartidos entre frontend y backend), `apps/frontend` (Angular), `apps/backend` (NestJS). Configurá Turborepo para que `turbo build` buildee los tres en el orden correcto (shared primero, luego las apps) y con caché.

**Done cuando:** Podés crear un monorepo con pnpm workspaces desde cero, configurar Turborepo para que las tareas respeten el orden de dependencias entre paquetes, y hacer que un tipo definido en `packages/shared` sea importable en ambas apps sin publicarlo a npm.

**Recursos:**

- [Turborepo docs — Getting Started](https://turbo.build/repo/docs) — la documentación oficial. La sección de "core concepts" es la más importante.
- [pnpm workspaces docs](https://pnpm.io/workspaces) — cómo configurar el monorepo a nivel de gestión de dependencias.
- Artículo: [*"Monorepos in JavaScript & TypeScript"*](https://www.robinwieruch.de/javascript-monorepos/) de Robin Wieruch — comparativa práctica de las opciones disponibles.
