# 2.4 — Git Avanzado y Modelo de Objetos 🟡 [ADVANCED]

> **Movido desde Fase 0:** Git es una herramienta del entorno de desarrollo, no una metodología de equipo.

- [ ] **El modelo de objetos de Git**

Qué son los cuatro tipos de objetos que Git almacena internamente: blobs (el contenido de un archivo — sin nombre, solo contenido), trees (un directorio — lista de blobs y otros trees con sus nombres), commits (un snapshot del proyecto — apunta a un tree raíz, tiene metadata de autor/mensaje/timestamp, y apunta al commit padre), refs (punteros con nombre a commits — `main`, `HEAD`, tags). Todo en Git es inmutable y direccionado por el hash SHA-1 de su contenido. Entender esto hace que todo lo demás — rebase, cherry-pick, reflog — tenga sentido como consecuencia natural del modelo.

**Por qué importa:** La mayoría de devs usa Git como una caja negra: `add`, `commit`, `push`, y reza que no haya conflictos. Cuando algo sale mal (historia corrupta, commits "perdidos", rebase que se fue de las manos) no saben cómo recuperarse porque no entienden qué está pasando por debajo. Entender el modelo de objetos convierte Git de caja negra a herramienta completamente predecible — y te da la confianza de saber que casi nada es permanentemente irrecuperable.

**Ejercicio:** Explorá el modelo de objetos de un repositorio real de FreePress: usá `git cat-file -t <hash>` para ver el tipo de un objeto y `git cat-file -p <hash>` para ver su contenido. Partiendo del commit HEAD, navegá manualmente hasta ver el contenido de un archivo fuente sin usar `git show` — solo a través del modelo de objetos. Dibujá el grafo de objetos de los últimos 3 commits.

**Done cuando:** Dado cualquier hash de commit de FreePress, podés navegar manualmente el grafo de objetos (commit → tree → blobs) usando solo `git cat-file`, sin comandos de alto nivel. Podés explicar en una oración por qué `git checkout` de una rama anterior no "pierde" los commits de la rama que dejaste.

- [ ] **Rebase, cherry-pick, bisect y reflog**

`git rebase`: reescribe la historia moviendo commits para que partan de un punto diferente — mantiene la historia lineal, pero cambia los hashes (por eso no se hace rebase de ramas públicas). `git rebase -i` (interactive): permite reordenar, combinar (squash), editar y eliminar commits de forma quirúrgica. `git cherry-pick`: aplica el cambio de un commit específico en otra rama sin llevar toda la historia. `git bisect`: binary search automatizado en el historial — dado un commit "bueno" y uno "malo", Git encuentra el commit exacto que introdujo el bug en O(log n) pasos. `git reflog`: el historial de todo lo que hiciste localmente — el último recurso cuando "perdiste" trabajo, ya que los objetos siguen existiendo hasta el garbage collector.

**Por qué importa:** El desarrollador que solo sabe `merge` está forzado a tener una historia de commits llena de "fix typo", "WIP", "ahora sí". El que sabe `rebase -i` puede convertir 8 commits de trabajo en 2 commits semánticos y coherentes antes de abrir el PR — lo que hace el code review mucho más fácil. Y cuando algo sale mal en producción, `git bisect` puede encontrar el commit culpable entre 1000 commits en menos de 10 pasos.

**Ejercicio:** Dos ejercicios concretos en FreePress: (1) Introducí un bug deliberado en un commit, hacé 8-10 commits más de trabajo normal encima, y usá `git bisect` con un script de test automatizado (`git bisect run npm test`) para encontrar el commit exacto que lo introdujo sin intervención manual. (2) Tomá una rama de feature con 6-8 commits desordenados y usá `git rebase -i` para reorganizarlos en 2-3 commits semánticos antes de mergear — sin perder ningún cambio.

**Done cuando:** Podés usar `git bisect run` con un script de test para encontrar automáticamente un commit culpable en el historial de FreePress. Podés recuperar trabajo "perdido" de un rebase salido mal usando `git reflog`. Podés explicar en una oración la diferencia entre `git merge` y `git rebase` en términos de qué hacen con los hashes de los commits.

**Recursos:**

- [*"Git Branching — Rebasing"*](https://git-scm.com/book/en/v2/Git-Branching-Rebasing) — la sección de Pro Git sobre rebase con diagramas visuales del antes y después.

- [ ] **Hooks y automatización local**

Git hooks: scripts que Git ejecuta automáticamente en momentos específicos del flujo de trabajo. Los más útiles: `pre-commit` (corre antes de crear el commit — ideal para lint y tests rápidos), `commit-msg` (valida el formato del mensaje — para enforcar conventional commits), `pre-push` (corre antes de pushear — para tests más lentos que no querés en cada commit). Cómo versionar los hooks (Git no los versiona por defecto — `husky` resuelve esto para proyectos Node.js). La diferencia entre hooks locales (pre-commit, pre-push) y server-side hooks (pre-receive, post-receive — en el servidor Git).

**Por qué importa:** Un pre-commit hook que corre el linter en FreePress hace imposible commitear código con errores de ESLint — sin depender de que el CI lo detecte minutos después. La retroalimentación es instantánea y local. El ahorro acumulado de ciclos "commit → push → CI falla → fix → commit → push" a lo largo de un proyecto es enorme. Y un `commit-msg` hook que enforza conventional commits hace que el CHANGELOG se genere automáticamente.

**Ejercicio:** Configurá `husky` en FreePress con tres hooks: (1) `pre-commit` — corre ESLint y Prettier sobre los archivos staged (usando `lint-staged` para no revisar todo el proyecto en cada commit); (2) `commit-msg` — valida que el mensaje siga el formato de conventional commits (`feat:`, `fix:`, `chore:`, etc.) usando `commitlint`; (3) `pre-push` — corre la suite completa de tests unitarios y rechaza el push si alguno falla.

**Done cuando:** En FreePress es imposible hacer commit con errores de lint o con un mensaje que no siga conventional commits — el hook lo rechaza automáticamente con un mensaje de error claro. Los hooks están versionados en el repo y funcionan para cualquier developer que clone el proyecto.

**Recursos:**
- [Husky Docs](https://typicode.github.io/husky/) — la herramienta estándar para gestionar Git hooks en proyectos Node.js. Setup en 2 minutos.
- [lint-staged](https://github.com/lint-staged/lint-staged) — corre linters solo sobre archivos staged, no sobre todo el proyecto. Complemento esencial de Husky para pre-commit rápidos.
- [commitlint](https://commitlint.js.org/) — valida mensajes de commit contra conventional commits. El compañero natural de Husky para `commit-msg`.
