> **Extensión de 2.1:** Este archivo complementa `2.1 — JavaScript Internals`. Aquí cubrimos qué hacer cuando el event loop no alcanza.

- [ ] **Concurrencia vs Paralelismo — La distinción que importa**

Concurrencia: manejar múltiples tareas que progresan al mismo tiempo, no necesariamente ejecutándose simultáneamente — el event loop de Node.js es un ejemplo perfecto de concurrencia sin paralelismo real. Paralelismo: ejecutar múltiples tareas literalmente al mismo tiempo en distintos cores de CPU. Node.js es single-threaded por diseño: el event loop hace concurrencia, pero todo el JavaScript corre en un solo hilo. Para I/O-bound work (requests de red, lectura de archivos) la concurrencia del event loop es suficiente y muy eficiente. Para CPU-bound work (criptografía, procesamiento de imágenes, parsing de archivos grandes) el event loop se bloquea — necesitás paralelismo real.

**Por qué importa:** El backend de FreePress usa NestJS sobre Node.js. Si un endpoint hace una operación CPU-intensiva (ej: generar un PDF de un artículo, redimensionar una imagen, calcular un hash criptográfico pesado) en el request handler, bloquea el event loop entero — ninguna otra request puede ser procesada mientras tanto. Con 100 requests concurrentes, una sola operación CPU-bound de 200ms convierte tu API de sub-10ms a potencialmente 20 segundos de latencia para todos. Entender cuándo el event loop es suficiente y cuándo necesitás Worker Threads es la diferencia entre un backend que escala y uno que se degrada bajo carga.

**Ejercicio:** Demostrá el problema en FreePress: creá un endpoint que haga trabajo CPU-intensivo (ej: calcular el número primo N-ésimo de forma naive). Generá carga concurrente con `autocannon` y observá cómo el tiempo de respuesta de TODOS los endpoints sube, no solo el endpoint pesado. Luego mové la operación a un Worker Thread y volvé a medir — verificá que los otros endpoints ya no se ven afectados.

- [ ] **Worker Threads y child_process en Node.js**

Worker Threads (`worker_threads` module): hilos reales que comparten memoria con el hilo principal a través de `SharedArrayBuffer` y se comunican via `postMessage`. Útiles para CPU-bound work que necesita compartir datos eficientemente. `child_process`: módulo para lanzar procesos separados — `spawn` (streaming de stdout/stderr), `exec` (captura el output completo), `fork` (proceso hijo Node.js con canal IPC). Cluster module: lanzar múltiples instancias del proceso de Node.js (una por core) que comparten el mismo puerto — horizontal scaling en una sola máquina. La diferencia entre los tres: Worker Threads comparten memoria (rápido pero con riesgo de race conditions), child_process son completamente aislados (más seguro, más overhead), cluster es child_process especializado para servidores HTTP.

**Por qué importa:** FreePress procesa imágenes cuando se suben artículos — resize, compresión, generación de thumbnails. Sin Worker Threads, esa operación bloquea el event loop de NestJS durante cada upload. Con Worker Threads, el procesamiento ocurre en paralelo sin afectar la disponibilidad del servidor. Y cuando FreePress necesite escalar verticalmente en un servidor de 8 cores, el Cluster module permite usar todos los cores con cambios mínimos de código.

**Ejercicio:** Implementá el procesamiento de imágenes de FreePress con Worker Threads: cuando se sube una imagen de portada para un artículo, delegá el resize y compresión (usando Sharp) a un pool de Worker Threads (no un Worker Thread nuevo por request — un pool de tamaño fijo igual al número de CPUs disponibles con `os.cpus().length`). - [ ] **Race conditions, deadlocks y primitivas de sincronización**

Race condition: cuando el resultado de una operación depende del orden no determinístico en que múltiples threads o procesos acceden a un recurso compartido. Deadlock: dos o más threads esperando cada uno un recurso que el otro tiene — se bloquean mutuamente para siempre. Mutex (Mutual Exclusion): primitiva de sincronización que garantiza que solo un thread a la vez puede acceder a una sección crítica. Semaphore: como un mutex pero permite N accesos simultáneos (útil para connection pools, rate limiters). En Node.js con Worker Threads: `SharedArrayBuffer` + `Atomics` para operaciones atómicas sin race conditions. En el mundo de bases de datos: `SELECT FOR UPDATE` como mutex a nivel de fila.

**Por qué importa:** En FreePress con Worker Threads, si dos Workers intentan escribir al mismo archivo de log simultáneamente sin sincronización, el resultado es un archivo corrupto. Si el pool de conexiones de PostgreSQL no está correctamente limitado, múltiples requests concurrentes pueden agotar las conexiones disponibles. Estos problemas son notoriamente difíciles de reproducir porque dependen de timing — aparecen en producción bajo carga, no en desarrollo. Entender las primitivas de sincronización te da las herramientas para diseñar código que sea correcto bajo concurrencia, no solo bajo un solo usuario.

**Ejercicio:** Demostrá una race condition en Node.js con Worker Threads: dos Workers incrementando un contador compartido en `SharedArrayBuffer` sin sincronización — el resultado será incorrecto. Luego arreglalo usando `Atomics.add()` para hacer el incremento atómico. En el contexto de FreePress, identificá un escenario real de race condition en el backend (ej: dos requests simultáneos intentando crear el mismo slug de artículo) y arreglalo con la solución correcta (constraint UNIQUE en PostgreSQL + manejo del error de unicidad en NestJS).

**Done cuando:** Podés leer código con acceso a recursos compartidos (SharedArrayBuffer, archivos, base de datos) e identificar si hay riesgo de race condition. Podés implementar la solución correcta para cada contexto: `Atomics` para SharedArrayBuffer, `Mutex` para recursos en memoria, constraints de DB para consistencia a nivel de persistencia.
