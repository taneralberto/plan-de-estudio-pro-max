# 3.1 — Domain-Driven Design (DDD) 🔴

---

- [ ] **Conceptos estratégicos — Bounded Contexts y Ubiquitous Language**

El concepto MÁS importante de DDD no es "Entities" o "Repositories". Es **Bounded Context**: cada parte de tu sistema tiene su propio modelo y lenguaje. En FreePress, "Article" significa algo diferente para el módulo de editor que para el módulo de analytics/SEO.

**Por qué importa:** Sin Bounded Contexts, tu modelo de datos crece hasta convertirse en un God Object que intenta satisfacer a todos los módulos a la vez. En FreePress, si "Article" tiene 40 campos mezclando datos de edición, SEO, analytics y notificaciones, es porque nunca se definió qué parte del sistema "posee" cada concepto. Ese modelo se vuelve imposible de cambiar sin romper algo más.

**Ejercicio:** Mapeá los Bounded Contexts de FreePress. Identificá al menos 4 contextos (ej: Editorial, Analytics, Autenticación, Notificaciones) y para cada uno escribí qué significa "Article" o "User" dentro de ese contexto — o si ese concepto no existe ahí. Documentalo en un diagrama de contextos con las relaciones entre ellos.

**Done cuando:** Podés tomar un modelo de dominio confuso (God Object con 30+ campos) y, en 20 minutos, identificar cuántos Bounded Contexts están colapsados en ese objeto y proponer la separación. Podés explicar por qué "User" en el contexto de autenticación y "Author" en el contexto editorial son conceptos diferentes aunque refieran a la misma persona real.

**Recursos:**
- [*Domain-Driven Design Distilled*](https://www.amazon.com/Domain-Driven-Design-Distilled-Vaughn-Vernon/dp/0134434420) de Vaughn Vernon — la versión corta y práctica, la mejor entrada al tema. Empezá acá.
- [*Learning Domain-Driven Design*](https://www.amazon.com/Learning-Domain-Driven-Design-Aligning-Architecture/dp/1098100131) de Vlad Khononov — el más moderno y accesible, con ejemplos de código concretos en lugar de teoría pura.

- [ ] **Conceptos tácticos — Entities, Value Objects, Aggregates, Repositories**

Cuándo una cosa es una Entity (tiene identidad propia — un `Article` con ID es el mismo aunque cambie su título) vs un Value Object (se define por sus atributos — `Money(100, "USD")` es igual a cualquier otro `Money(100, "USD")`). Qué es un Aggregate y por qué es el boundary de consistencia transaccional: todo lo que debe cambiar junto en una sola transacción vive dentro del mismo Aggregate.

**Por qué importa:** Sin este vocabulario, terminás con tablas de base de datos que modelan "cómo guardo datos" en lugar de "cómo funciona el negocio". La diferencia entre Entity y Value Object no es filosófica — tiene consecuencias directas en cómo diseñás tus tablas, tus validaciones y tus transacciones. Si `Address` es un Value Object, no tiene ID propio en la base de datos. Si es una Entity, sí. Esa decisión afecta el esquema entero.

**Ejercicio:** Modelá el dominio de FreePress con DDD táctico: identificá bounded contexts, entities, value objects y aggregates. Específicamente: ¿`Tag` es una Entity o un Value Object? ¿`Comment` pertenece al Aggregate de `Article` o tiene su propio Aggregate? Justificá cada decisión con el criterio de consistencia transaccional.

**Done cuando:** Podés tomar un modelo de dominio nuevo (ej: un sistema de reservas de hotel) y en 30 minutos clasificar sus conceptos en Entities, Value Objects y Aggregates con justificación. Podés explicar por qué poner demasiadas cosas en un Aggregate lleva a problemas de contención de locks en la base de datos.

**Recursos:**
- [*Domain-Driven Design Distilled*](https://www.amazon.com/Domain-Driven-Design-Distilled-Vaughn-Vernon/dp/0134434420) de Vaughn Vernon — cubre los bloques tácticos en forma accesible (caps. 5-7).
- [*Implementing Domain-Driven Design*](https://www.amazon.com/Implementing-Domain-Driven-Design-Vaughn-Vernon/dp/0321834577) de Vaughn Vernon — el libro técnico de referencia para implementación real con código.

- [ ] **Context Mapping — Cómo se relacionan los bounded contexts**

Anticorruption Layer, Open Host Service, Published Language, Shared Kernel. Cómo dos equipos (o dos servicios) se comunican sin acoplarse. El Context Map es el mapa político de tu sistema: quién le manda a quién, quién consume a quién, quién tiene que adaptarse.

**Por qué importa:** Cuando el módulo de Analytics de FreePress necesita datos del módulo Editorial, la forma en que se conectan no es un detalle de implementación — es una decisión de arquitectura. Si Analytics consume directamente las entidades de Editorial, estás creando acoplamiento estructural: cualquier cambio en Editorial puede romper Analytics. Un Anticorruption Layer es la capa que traduce entre modelos para que cada contexto pueda evolucionar independientemente.

**Ejercicio:** Dibujá el Context Map de FreePress con al menos 4 contextos. Para cada relación entre contextos, identificá el patrón: ¿es un Shared Kernel? ¿Customer-Supplier? ¿Anticorruption Layer? Documentá por qué elegiste ese patrón para esa relación específica, no otro.

**Done cuando:** Podés leer el diagrama de arquitectura de un sistema existente e identificar qué patrón de Context Mapping está usando implícitamente cada integración — incluso si no estaba diseñada con DDD. Podés argumentar cuándo un ACL agrega valor real y cuándo es sobreingeniería.

**Recursos:**
- [*Domain-Driven Design Distilled*](https://www.amazon.com/Domain-Driven-Design-Distilled-Vaughn-Vernon/dp/0134434420) de Vaughn Vernon — el capítulo de Context Mapping es el más práctico del libro.
- [DDD Reference de Eric Evans](https://www.domainlanguage.com/ddd/reference/) — el glosario oficial, gratuito. Útil como referencia rápida de patrones.
- [*Team Topologies*](https://teamtopologies.com/book) de Skelton & Pais — no es un libro de DDD, pero el concepto de "team cognitive load" y stream-aligned teams mapea directamente con Bounded Contexts y Context Mapping. Lectura complementaria muy valiosa.
