# 3.2 — Patrones de Arquitectura Avanzados 🔴 [CORE]

- [ ] **SOLID Principles — La base de código mantenible**

Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion. No como acrónimo para entrevistas — como principios que determinan si tu código se puede modificar sin romper todo lo demás.

**Por qué importa:** Sin SOLID, "Clean Architecture" y "DDD" son palabras vacías. La Dependency Rule de Clean Architecture ES el Dependency Inversion Principle aplicado a arquitectura. Cada principio resuelve un problema específico de código que se vuelve inmantenible.

**Ejercicio:** Tomá una clase de FreePress (o un proyecto tuyo) que tenga más de una responsabilidad. Identificá qué hace cada método. Separá en clases distintas siguiendo Single Responsibility Principle. Documentá el antes/después.

**Done cuando:** Podés mirar una clase y decir si viola algún principio SOLID, cuál, y cómo refactorizarla. Podés explicar por qué "una clase que hace muchas cosas" es un problema de Single Responsibility, no solo "mala práctica".

**Recursos:**
- [*Clean Code*](https://www.amazon.com/Clean-Code-Handbook-Software-Craftsmanship/dp/0132350882) de Robert Martin — capítulos 8-10 cubren SOLID en profundidad
- [SOLID Principles Explained](https://www.baeldung.com/solid-principles) — explicación clara con ejemplos en Java (aplica igual a TypeScript)
- [SOLID en TypeScript](https://khalilstemmler.com/articles/solid-principles/solid-typescript/) — artículos de Khalil Stemmler con ejemplos específicos en TS

- [ ] **Event-Driven Architecture (EDA)**

Eventos de dominio, event sourcing, event bus. La diferencia entre commands, events y queries. Por qué "cuando un usuario publica un artículo" debería emitir un evento, no llamar directamente al servicio de indexación.

**Por qué importa:** Cuando el módulo editorial de FreePress publica un artículo, ¿quién más necesita saberlo? El indexador de búsqueda, el sistema de notificaciones, el pipeline de analytics, el generador de sitemap. Si cada uno de esos llama directamente al siguiente, el módulo editorial termina dependiendo de la implementación interna de 4 sistemas distintos. Un evento de dominio (`ArticlePublished`) invierte esa dependencia: el editorial no sabe quién escucha, y cada consumidor puede aparecer o desaparecer sin tocar el publisher.

**Ejercicio:** Diseñá el flujo de publicación de un artículo en FreePress usando EDA. Listá todos los eventos de dominio que ocurren (`ArticlePublished`, `ArticleUpdated`, etc.), qué información lleva cada evento, y quién los consume. Implementá en NestJS un EventEmitter interno simple para `ArticlePublished` con al menos un consumidor real (ej: actualizar el índice de búsqueda).

**Done cuando:** Podés tomar un flujo de negocio con múltiples efectos secundarios (publicar un artículo, procesar un pago, registrar un usuario) y diseñar los eventos de dominio correspondientes — qué emite, qué consume, qué información lleva cada evento. Podés explicar la diferencia entre un event y un command y por qué importa para el flujo de control.

**Recursos:**
- [*Building Event-Driven Microservices*](https://www.amazon.com/Building-Event-Driven-Microservices-Leveraging-Organizational/dp/1492057894) de Adam Bellemare — el libro más práctico sobre EDA, cubre desde los conceptos hasta Kafka y particionado.
- [Martin Fowler — "What do you mean by Event-Driven?"](https://martinfowler.com/articles/201701-event-driven.html) — artículo fundamental que distingue Event Notification, Event-Carried State Transfer, Event Sourcing y CQRS. Leerlo antes del libro.

- [ ] **CQRS (Command Query Responsibility Segregation)**

Separar el modelo de lectura del de escritura. Cuándo tiene sentido (sistemas con alta lectura y poca escritura, reportes complejos que no encajan en el modelo de escritura) y cuándo es overkill (un CRUD simple donde el mismo modelo sirve para leer y escribir). Event Sourcing + CQRS como combo: el event store como fuente de verdad, las proyecciones como modelos de lectura derivados.

**Por qué importa:** FreePress tiene un panel de analytics que agrega datos de lectura, tiempo en página, dispositivos, referrers. Construir esas vistas desde el mismo modelo normalizado que usás para editar artículos va a ser doloroso: queries complejas, joins pesados, modelo que no encaja. CQRS te permite tener un modelo de escritura limpio (DDD) y modelos de lectura optimizados para cada vista — sin comprometer ninguno de los dos.

**Ejercicio:** Tomá el dashboard de analytics de FreePress. Diseñá un modelo de lectura separado (una tabla desnormalizada o una colección Redis) optimizado específicamente para responder "top 10 artículos por vistas en los últimos 7 días". Definí el Command que actualiza ese modelo (ej: `IncrementArticleView`) y la Query que lo consume. No hace falta implementarlo completo — alcanza con el diseño del modelo y los tipos en TypeScript.

**Done cuando:** Podés analizar una feature nueva de FreePress y decidir con criterio si CQRS agrega valor o complejidad innecesaria. Podés diseñar el modelo de escritura y al menos un modelo de lectura para un caso concreto, justificando la desnormalización. Podés explicar qué problema introduce CQRS que el CRUD clásico no tiene (consistencia eventual entre modelos).

**Recursos:**
- [Martin Fowler — "CQRS"](https://martinfowler.com/bliki/CQRS.html) — artículo conciso que explica cuándo tiene sentido y cuándo no. Empezá acá antes del libro.
- [*Implementing Domain-Driven Design*](https://www.amazon.com/Implementing-Domain-Driven-Design-Vaughn-Vernon/dp/0321834577) de Vaughn Vernon — el capítulo sobre CQRS + Event Sourcing es la referencia técnica más completa.
- [*Designing Data-Intensive Applications*](https://www.amazon.com/Designing-Data-Intensive-Applications-Reliable-Maintainable/dp/1449373321) de Martin Kleppmann — caps. 11-12 sobre stream processing y derived data modelan exactamente el problema que CQRS resuelve desde una perspectiva de sistemas.

- [ ] **API Design — REST, GraphQL, gRPC**

REST: el modelo de madurez de Richardson (niveles 0-3), HATEOAS, por qué la mayoría de APIs "REST" son nivel 2 como mucho. GraphQL: cuándo resuelve problemas reales (over-fetching, clientes que necesitan datos distintos) y cuándo agrega complejidad innecesaria (un backend con un solo cliente). gRPC: comunicación entre servicios internos, Protocol Buffers, streaming bidireccional.

**Por qué importa:** FreePress tiene una "API pública documentada" como feature. Sin saber diseñar APIs, esa API va a acumular deuda desde el día uno: endpoints inconsistentes, versionado roto, contratos que cambian y rompen a los clientes. Una API mal diseñada es deuda técnica permanente — porque una vez que tenés clientes externos, no podés cambiar la API sin coordinar con ellos. Diseñarla bien desde el principio es 10 veces más barato que migrarla después.

**Ejercicio:** Diseñá la API pública de FreePress en REST nivel 3. Definí los recursos, los endpoints, los códigos de respuesta, la estrategia de versionado (URI versioning vs header versioning — justificá la elección), y la paginación (cursor vs offset — justificá cuándo usarías cada uno). Luego diseñá el mismo recurso principal en GraphQL y listá los trade-offs reales que encontraste entre ambos enfoques.

**Done cuando:** Podés revisar una API REST existente e identificar en qué nivel del modelo de Richardson está y qué le faltaría para subir un nivel. Podés explicar cuándo GraphQL introduce problemas que REST no tiene (N+1 queries, complejidad de caching, surface de ataque mayor) y cuándo los resuelve. Podés diseñar un esquema de versionado de API que permita evolucionar el contrato sin romper clientes existentes.

**Recursos:**
- [*Designing Web APIs*](https://www.amazon.com/Designing-Web-APIs-Building-Developers/dp/1492026921) de Brenda Jin — el libro más práctico sobre diseño de APIs REST, con ejemplos reales de APIs populares.
- [Martin Fowler — "Richardson Maturity Model"](https://martinfowler.com/articles/richardsonMaturityModel.html) — artículo de referencia, leerlo antes del libro.
- [*The GraphQL Guide*](https://graphql.guide/) — guía completa de GraphQL con ejemplos modernos, incluye sección sobre cuándo NO usar GraphQL.

- [ ] **Microservicios — Patrones y Anti-patrones**

Saga pattern (cómo manejar transacciones distribuidas sin 2PC), Circuit Breaker (cómo evitar que la falla de un servicio cascade al resto), Service Mesh (Istio, Linkerd — cómo manejar la comunicación entre servicios a nivel de infraestructura), API Gateway (punto de entrada único, auth, rate limiting, routing). Strangler Fig como estrategia de migración. Distributed monolith como el anti-patrón más común: todos los servicios desplegados por separado pero acoplados igual que un monolito.

**Por qué importa:** El 80% de los proyectos que "adoptan microservicios" terminan con un distributed monolith: tienen el costo operacional de microservicios (múltiples deploys, red entre servicios, observabilidad compleja) sin ninguno de los beneficios (deploys independientes, equipos autónomos, escalabilidad granular). Entender los patrones y anti-patrones te permite tomar la decisión correcta antes de comprometerte — y saber cuándo un monolito bien estructurado es la respuesta correcta (casi siempre al principio).

**Ejercicio:** Analizá FreePress como si fuera a migrar de monolito a microservicios. Identificá cuáles módulos serían candidatos naturales a servicios independientes y cuáles no (usando el criterio de cohesión y volatilidad de cambio, no "porque es una feature"). Para el módulo que elegiría separar primero, diseñá el Strangler Fig: ¿qué se migra primero, cómo coexisten el monolito y el microservicio durante la transición, cómo se maneja la data?

**Done cuando:** Podés revisar una arquitectura de microservicios propuesta e identificar si es un distributed monolith (señales: servicios que no pueden desplegarse independientemente, base de datos compartida, llamadas síncronas en cadena). Podés explicar cuándo el Saga pattern (coreografía vs orquestación) es preferible sobre 2PC y cuáles son los trade-offs de consistencia eventual que introduce.

**Recursos:**
- [*Building Microservices*](https://www.amazon.com/Building-Microservices-Designing-Fine-Grained-Systems/dp/1492034029) de Sam Newman — la biblia. Segunda edición (2021) incluye lecciones aprendidas de la industria.
- [*Monolith to Microservices*](https://www.amazon.com/Monolith-Microservices-Evolutionary-Patterns-Transform/dp/1492047848) de Sam Newman — la secuela práctica, específicamente sobre el proceso de migración con patrones concretos.
- [Martin Fowler — "Microservices"](https://martinfowler.com/articles/microservices.html) — el artículo original que definió el término. Leerlo para entender las condiciones en las que microservicios tienen sentido real.
