# 3.3 — System Design en Profundidad 🔴

---

- [ ] **Cómo pensar en System Design**

Requisitos funcionales y no-funcionales. Capacity estimation (back-of-the-envelope): cuántos usuarios, cuántas requests por segundo, cuánto storage, cuánto ancho de banda. Trade-offs fundamentales: consistency vs availability (CAP theorem), latency vs throughput, read-heavy vs write-heavy. Cómo estructurar una discusión de system design en 45 minutos — el framework de clarificación → estimación → diseño de alto nivel → deep dive.

**Por qué importa:** System Design no es un skill que se usa solo en entrevistas — es el skill que te permite tomar decisiones arquitectónicas con criterio en lugar de intuición. Cuando el CTO pregunta "¿podemos soportar 10x de tráfico?", la respuesta correcta no es "sí" o "no" — es un análisis de dónde están los cuellos de botella hoy y qué habría que cambiar. Sin el framework de capacity estimation, esa conversación es un juego de opiniones. Con él, es una conversación técnica.

**Ejercicio:** Hacé una capacity estimation de FreePress para el estado actual (audiencia pequeña) y para dos escenarios futuros: 100k usuarios activos mensuales y 1M usuarios activos mensuales. Para cada escenario estimá: requests por segundo en pico, storage necesario para artículos + media + logs, ancho de banda de salida. Usá números reales, no "muchos" o "bastantes".

**Done cuando:** Ante un sistema desconocido (ej: "diseñá el backend de Instagram Stories"), podés hacer una capacity estimation back-of-the-envelope en 5 minutos que identifique si el sistema es read-heavy o write-heavy, cuánto storage necesita por año, y cuántas requests por segundo en el pico. Podés explicar el CAP theorem con un ejemplo concreto donde elegir C > A o A > C tenga consecuencias reales para el usuario.

**Recursos:**
- [*System Design Interview Vol. 1 y 2*](https://www.amazon.com/System-Design-Interview-insiders-Second/dp/B08CMF2CQF) de Alex Xu — los mejores libros para aprender el framework de system design. Empezá con Vol. 1, cap. 1-2 para el framework base.
- [*Designing Data-Intensive Applications*](https://www.amazon.com/Designing-Data-Intensive-Applications-Reliable-Maintainable/dp/1449373321) de Martin Kleppmann — EL libro de referencia. No es un libro de entrevistas, es el libro que explica por qué los sistemas están diseñados como están. Nivel biblia.
- [Canal YouTube: ByteByteGo](https://www.youtube.com/@ByteByteGo) de Alex Xu — casos visuales de system design en 5-10 minutos. Excelente para calibrar el nivel de profundidad esperado.

- [ ] **Patrones de escalabilidad**

Load balancing (round-robin, least connections, consistent hashing — y cuándo cada uno tiene sentido). Caching en capas: browser cache, CDN (Cloudflare, CloudFront), application cache (Redis), database query cache — cuándo cachear en cada capa y cuándo no. Database scaling: read replicas para escalar lecturas, sharding para escalar escrituras, connection pooling para escalar conexiones. Message queues como buffer: RabbitMQ, Kafka, Redis Streams — cuándo un queue es la respuesta y cuándo no.

**Por qué importa:** La mayoría de los problemas de performance de una app en producción no son problemas de algoritmos — son problemas de arquitectura de escalabilidad. Sin entender estos patrones, cuando tu app se caiga bajo carga vas a estar adivinando qué escalar. Con ellos, podés leer las métricas (CPU del servidor, wait time de DB, cache hit rate) y saber exactamente qué componente es el cuello de botella y cuál es el patrón que lo resuelve.

**Ejercicio:** Diseñá FreePress para soportar 1 millón de usuarios activos mensuales. Partí desde la arquitectura actual (un monolito con Postgres y Redis) e identificá qué componente falla primero bajo esa carga. Luego diseñá la solución para ese cuello de botella específico (no para todos a la vez). Documentá el diseño: qué cambia, qué se agrega, por qué en ese orden.

**Done cuando:** Podés leer un flamegraph o un dashboard de métricas de una app bajo carga e identificar si el cuello de botella está en CPU, en I/O de base de datos, en latencia de red, o en contención de locks. Podés proponer el patrón de escalabilidad correcto para ese cuello de botella y explicar qué problema nuevo introduce ese patrón (ej: un cache introduce invalidación; read replicas introduce replication lag).

**Recursos:**
- [*Designing Data-Intensive Applications*](https://www.amazon.com/Designing-Data-Intensive-Applications-Reliable-Maintainable/dp/1449373321) de Martin Kleppmann — caps. 1, 5, 6 y 10 cubren replicación, particionado y consistency. El libro de referencia para este tema.
- [*System Design Interview Vol. 1*](https://www.amazon.com/System-Design-Interview-insiders-Second/dp/B08CMF2CQF) de Alex Xu — los capítulos de URL Shortener, Rate Limiter y Consistent Hashing son excelentes casos prácticos de patrones de escalabilidad.
- [highscalability.com](http://highscalability.com) — casos reales de arquitectura de empresas (Slack, Airbnb, Twitter). Leer al menos 3 casos de sistemas que conozcas para ver cómo aplican estos patrones en producción real.

- [ ] **Diseños de sistemas reales — Casos de estudio**

Estudiar cómo están diseñados sistemas a escala masiva: YouTube (video processing pipeline — por qué el video se procesa de forma asíncrona y en múltiples resoluciones), Spotify (recommendations — cómo funciona un pipeline de ML a escala), Slack (real-time messaging — WebSockets, presencia, ordering de mensajes), GitHub (CI/CD, git hosting — cómo almacenan millones de repos y corren builds en paralelo).

**Por qué importa:** Los casos de estudio son donde el theory meets reality. Cada uno de estos sistemas resolvió problemas que vos vas a encontrar en versiones más pequeñas: YouTube resolvió el problema de media processing que FreePress va a tener con imágenes y videos; Slack resolvió el real-time messaging que FreePress necesita para comentarios en vivo; GitHub resolvió el hosting de código que cualquier herramienta de colaboración necesita. Estudiarlos te da intuición sobre trade-offs reales, no teóricos.

**Ejercicio:** Elegí uno de los sistemas de la lista (recomendación: Slack o YouTube — son los más relevantes para FreePress). Leé al menos 2 artículos de su engineering blog sobre un problema específico que resolvieron. Luego escribí un documento de una página: qué problema tenían, qué solución eligieron, qué trade-offs aceptaron, y qué parte de esa solución podrías aplicar en FreePress (aunque sea a escala reducida).

**Done cuando:** Podés tomar cualquier sistema conocido (Netflix, Uber, WhatsApp) y describir su arquitectura de alto nivel con los trade-offs principales — no de memoria, sino razonando desde los requisitos. Dado "necesito entregar video a 200M usuarios concurrentes", podés llegar a "necesito CDN + procesamiento asíncrono + múltiples encodings" por razonamiento, no por haberlo leído.

**Recursos:**
- [highscalability.com](http://highscalability.com) — casos reales de arquitectura documentados en profundidad. Empezá con los de Slack, YouTube y Discord.
- [Canal YouTube: ByteByteGo](https://www.youtube.com/@ByteByteGo) de Alex Xu — explicaciones visuales de sistemas reales en 8-12 minutos. Ideal para calibrar el nivel de abstracción correcto.
- [*System Design Interview Vol. 1 y 2*](https://www.amazon.com/System-Design-Interview-insiders-Second/dp/B08CMF2CQF) de Alex Xu — cada capítulo es un caso de estudio completo (URL Shortener, YouTube, Google Drive, Chat System). Usarlo como referencia de profundidad esperada.
- [Engineering blogs de referencia](https://github.com/kilimchoi/engineering-blogs) — lista curada de blogs de ingeniería de empresas reales. Los de Slack Engineering, Discord Engineering y GitHub Engineering son especialmente relevantes para FreePress.
