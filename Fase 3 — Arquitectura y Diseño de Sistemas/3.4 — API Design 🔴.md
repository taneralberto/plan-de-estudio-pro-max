# 3.4 — API Design 🔴 [ADVANCED]

> **Nota:** El archivo `3.2 — Patrones de Arquitectura Avanzados` cubre REST, GraphQL y gRPC como paradigmas. Este archivo profundiza en diseño del contrato: versionado, paginación, idempotencia — lo que hace una API mantenible a lo largo del tiempo.

- [ ] **Diseño del contrato REST — Más allá de los verbos HTTP**

El modelo de madurez de Richardson como mapa de progreso: Nivel 0 (un endpoint para todo — RPC sobre HTTP), Nivel 1 (recursos como sustantivos — `/articles`, `/users`), Nivel 2 (verbos HTTP con semántica correcta — GET para leer, POST para crear, PUT/PATCH para actualizar, DELETE para eliminar, con status codes apropiados), Nivel 3 (HATEOAS — las respuestas incluyen links a las acciones disponibles). Naming de recursos: sustantivos en plural, jerarquía que refleja relaciones (`/articles/{id}/comments`), pero sin anidamiento infinito. Status codes correctos y sus diferencias: `200 vs 201 vs 204`, `400 vs 422`, `401 vs 403`, `404 vs 410`. Idempotencia: `GET`, `PUT` y `DELETE` deben ser idempotentes (múltiples llamadas idénticas producen el mismo resultado), `POST` no lo es por defecto.

**Por qué importa:** FreePress tiene una "API pública documentada" como feature. Una API mal diseñada es deuda técnica permanente — porque una vez que tenés clientes externos (plugins, integraciones de terceros, apps móviles), no podés cambiar los contratos sin coordinación. Un endpoint que devuelve `200` para un error enmascarado, o que usa `POST` para operaciones que deberían ser `GET`, va a generar bugs en todos los clientes que la consuman. Diseñar la API bien desde el principio es 10 veces más barato que migrarla después.

**Ejercicio:** Diseñá la API pública de FreePress siguiendo el Nivel 2 de Richardson como mínimo. Documentá en un archivo OpenAPI/Swagger: los recursos principales (articles, users, tags, comments), todos los endpoints con sus verbos HTTP, los status codes que devuelve cada uno (incluyendo errores), y los request/response bodies con sus tipos. Verificá que todas tus operaciones de lectura son idempotentes y que los POST de creación devuelven 201 con el header `Location` apuntando al recurso creado.

**Done cuando:** Podés revisar una API REST existente e identificar en qué nivel del modelo de Richardson está, qué status codes están mal usados, y qué operaciones no son idempotentes cuando deberían serlo. Podés diseñar una API nueva para una feature de FreePress siguiendo las convenciones de naming, verbos y status codes correctamente, sin consultar la documentación.

- [*"Designing Web APIs"*](https://www.amazon.com/Designing-Web-APIs-Building-Developers/dp/1492026921) de Brenda Jin — el libro más práctico sobre diseño de APIs REST con ejemplos de APIs reales y populares.

- [ ] **Versionado, paginación e idempotency keys**

Versionado de APIs: URI versioning (`/v1/articles` — simple y visible), header versioning (`API-Version: 2024-01`) y content negotiation (`Accept: application/vnd.freepress.v2+json` — más correcto pero más complejo). Cuándo hacer un breaking change vs un cambio backward-compatible (agregar campos es safe, renombrar o eliminar no lo es). Paginación: cursor-based (más eficiente y consistente para datasets que cambian — usar un cursor opaco como `?after=eyJpZCI6MTIzfQ==`) vs offset-based (`?page=2&limit=20` — simple pero inconsistente si hay inserciones). Idempotency keys: un header `Idempotency-Key: <uuid>` que el cliente genera y el servidor usa para deduplicar requests — crítico para operaciones no idempotentes (pagos, envío de emails) donde el cliente puede reintentar sin saber si el primer intento llegó.

**Por qué importa:** La paginación offset de FreePress (`?page=2`) tiene un bug silencioso: si se publica un nuevo artículo mientras el usuario está navegando, la página 2 va a tener un artículo duplicado (el último de la página 1 "cayó" a la página 2). Con cursor-based pagination esto no pasa. Y las idempotency keys son la diferencia entre "el usuario recibió el email de confirmación dos veces porque reintentó el request" y "el sistema detectó el retry y no lo procesó de nuevo".

**Ejercicio:** Implementá en la API de FreePress: (1) Cursor-based pagination para el listado de artículos — el cursor es el `id` del último artículo recibido, codificado en base64 para que sea opaco. El endpoint acepta `?after=<cursor>&limit=20` y devuelve `{data: [...], nextCursor: "...", hasMore: boolean}`; (2) Idempotency keys para el endpoint de creación de artículos — el servidor guarda el resultado de cada `Idempotency-Key` en Redis por 24 horas y devuelve el mismo resultado para requests duplicados.

**Done cuando:** Podés explicar cuándo cursor pagination produce resultados incorrectos (hint: nunca — es exactamente el punto) vs cuándo offset pagination produce duplicados o saltos. Podés implementar idempotency keys en cualquier endpoint no idempotente de FreePress, con almacenamiento del resultado en Redis y TTL apropiado.

**Recursos:**
- [Stripe API Docs — Idempotency](https://stripe.com/docs/api/idempotent_requests) — el ejemplo más claro de implementación de idempotency keys en producción. Stripe es la referencia de diseño de APIs en la industria.
- [*"Cursor-based Pagination"*](https://use-the-index-luke.com/no-offset) de Markus Winand — por qué offset pagination escala mal y cómo cursor pagination lo resuelve, con ejemplos SQL.
- [*"API Versioning Has No Right Way"*](https://apisyouwonthate.com/blog/api-versioning-has-no-right-way) de APIs You Won't Hate — análisis honesto de los trade-offs de cada estrategia de versionado.

- [ ] **GraphQL — Cuándo sí y cuándo no**

GraphQL resuelve problemas reales: over-fetching (el cliente recibe más datos de los que necesita), under-fetching (el cliente necesita N requests para obtener datos relacionados — el N+1 problem del REST), y la necesidad de que distintos clientes (mobile con ancho de banda limitado, desktop con pantalla grande) consuman la misma API de forma diferente. Los problemas que introduce: N+1 queries en el servidor (sin DataLoader), caching más complejo que REST (cada query puede ser distinta), mayor superficie de ataque (cualquier cliente puede pedir cualquier combinación de datos), y tooling de testing más complejo. Cuándo GraphQL tiene sentido real: múltiples clientes con necesidades de datos distintas, frontend teams que evolucionan rápido y necesitan flexibilidad, sistemas con grafos de datos complejos y relacionados.

**Por qué importa:** Muchos proyectos adoptan GraphQL porque "es moderno" sin evaluar si resuelve sus problemas reales. FreePress con un solo frontend Angular probablemente no justifica GraphQL — REST bien diseñado es más simple de implementar, cachear y mantener. Pero si FreePress tuviera una app mobile, un widget embebible, y un dashboard de analytics, cada uno consumiendo datos distintos del mismo backend — ahí GraphQL empieza a tener sentido. La decisión debe basarse en los problemas reales, no en la popularidad de la tecnología.

**Ejercicio:** Implementá el endpoint de artículo de FreePress en GraphQL usando NestJS + `@nestjs/graphql` con code-first approach. Implementá DataLoader para evitar el N+1 problem al resolver el autor de cada artículo. Luego compará: ¿cuántas queries a la base de datos hace el endpoint REST equivalente vs el GraphQL con DataLoader para cargar 10 artículos con sus autores? Documentá cuánto código adicional requirió GraphQL vs REST para la misma funcionalidad.

**Done cuando:** Podés analizar un proyecto nuevo y determinar con criterio técnico si GraphQL resuelve un problema real o agrega complejidad sin beneficio. Podés implementar un schema GraphQL básico en NestJS con DataLoader para evitar N+1, y explicar por qué DataLoader es obligatorio en cualquier implementación de GraphQL seria.

**Recursos:**
- [*"The GraphQL Guide"*](https://graphql.guide/) — guía completa con ejemplos modernos, incluyendo sección sobre cuándo NO usar GraphQL.
- [DataLoader](https://github.com/graphql/dataloader) — la librería estándar para batching y caching de queries en GraphQL. Sin esto, cualquier implementación de GraphQL tendrá N+1 problems.
- [*"GraphQL is not what you think it is"*](https://bessey.dev/blog/2024/05/24/why-im-over-graphql/) — perspectiva crítica y honesta sobre los trade-offs de GraphQL en producción. Balanceá con los casos de éxito para tener visión completa.
