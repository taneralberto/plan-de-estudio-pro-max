# 3.5 — Messaging y Event Streaming 🟡

---

- [ ] **El problema que resuelven los message queues**

Comunicación síncrona (REST, gRPC): el caller espera la respuesta del callee — si el callee está lento o caído, el caller también sufre. Comunicación asíncrona con queues: el producer deposita un mensaje en la queue y continúa sin esperar — el consumer lo procesa cuando puede. Desacoplamiento temporal: producer y consumer no necesitan estar disponibles al mismo tiempo. Desacoplamiento de velocidad: si el producer genera mensajes más rápido de lo que el consumer puede procesar, la queue actúa como buffer. Dead Letter Queue (DLQ): dónde van los mensajes que no pudieron ser procesados después de N reintentos — crítico para no perder mensajes sin saberlo. At-least-once vs exactly-once delivery: los queues distribuidos garantizan que el mensaje llega, pero pueden entregarlo más de una vez — tus consumers deben ser idempotentes.

**Por qué importa:** FreePress tiene un sistema de notificaciones: cuando se publica un artículo, los suscriptores reciben un email. Sin un message queue, el endpoint de publicación llama directamente al servicio de email — si el servicio de email está lento, el endpoint de publicación también lo está. Con un queue, el endpoint deposita un mensaje "ArticlePublished" y retorna 200 inmediatamente. El servicio de email consume el mensaje de forma asíncrona. Si el servicio de email falla, el mensaje se reintenta automáticamente. La publicación del artículo no se ve afectada.

**Ejercicio:** Implementá el sistema de notificaciones de FreePress con un message queue: cuando se publica un artículo, el endpoint NestJS deposita un mensaje en RabbitMQ (usando `@golevelup/nestjs-rabbitmq`). Un consumer separado lee los mensajes y envía emails con Resend o SendGrid. Configurá la DLQ para mensajes que fallan 3 veces y un endpoint de admin que liste los mensajes en la DLQ. Verificá que si el consumer de emails está caído, los artículos se pueden publicar sin error y los emails se envían cuando el consumer vuelve.

**Done cuando:** El sistema de notificaciones de FreePress es resiliente a fallos del servicio de email — los artículos se publican sin errores y los emails se envían eventualmente, aunque el servicio de email haya estado caído durante horas. Podés explicar por qué tus consumers deben ser idempotentes cuando usás at-least-once delivery y qué pasa si no lo son.

**Recursos:**
- [RabbitMQ Docs — Tutorials](https://www.rabbitmq.com/tutorials) — los 6 tutoriales oficiales de RabbitMQ (Work Queues, Pub/Sub, Routing, Topics, RPC) son la mejor introducción práctica. Completarlos todos toma medio día.
- [*"Enterprise Integration Patterns"*](https://www.enterpriseintegrationpatterns.com/) de Hohpe & Woolf — el catálogo de patrones de mensajería. No leerlo de tapa a tapa — usarlo como referencia cuando necesitás un patrón específico.

- [ ] **RabbitMQ vs Kafka — Cuándo usar cada uno**

RabbitMQ: un message broker tradicional. Los mensajes se entregan a consumers y se eliminan una vez procesados (acknowledged). Ideal para: task queues, workflows, RPC asíncrono, cuando el procesamiento de cada mensaje es el objetivo. Kafka: un distributed event log. Los mensajes se guardan indefinidamente (configurable) y múltiples consumer groups pueden leerlos independientemente desde cualquier offset. Ideal para: event sourcing, audit logs, pipelines de datos, cuando necesitás reproducir eventos o múltiples sistemas leen los mismos eventos. Redis Streams: Kafka simplificado sobre Redis — sin las garantías de durabilidad de Kafka pero suficiente para muchos casos de uso y sin infraestructura adicional si ya usás Redis.

**Por qué importa:** Elegir Kafka para un task queue de notificaciones de email es como usar una base de datos distribuida para guardar 3 configuraciones. Kafka tiene overhead operacional significativo: ZooKeeper o KRaft, particionado, replication factor, retention policies. RabbitMQ es mucho más simple de operar para casos de uso de message queue tradicionales. La decisión incorrecta implica meses de operaciones innecesariamente complejas. Para FreePress en sus primeras etapas, RabbitMQ o Redis Streams son casi siempre la respuesta correcta.

**Ejercicio:** Modelá dos features de FreePress con la tecnología correcta y justificá: (1) El pipeline de procesamiento de imágenes (resize, thumbnails) cuando se sube media — ¿RabbitMQ task queue o Kafka event stream? Justificá con el criterio de si necesitás reproducir los eventos o si el procesamiento es el objetivo. (2) Un audit log de todas las acciones sobre artículos (publicado, editado, eliminado) para cumplimiento y debugging — ¿RabbitMQ o Kafka? Justificá con el criterio de retención, múltiples consumers, y reproducibilidad.

**Done cuando:** Dado un nuevo requerimiento de comunicación asíncrona en FreePress, podés determinar en 5 minutos si corresponde RabbitMQ, Kafka, Redis Streams, o ninguno (comunicación síncrona directa) — con justificación técnica. Podés explicar por qué "Kafka escala mejor" no es una justificación suficiente para elegirlo sobre RabbitMQ en proyectos de tamaño medio.

**Recursos:**
- [*"Building Event-Driven Microservices"*](https://www.amazon.com/Building-Event-Driven-Microservices-Leveraging-Organizational/dp/1492057894) de Adam Bellemare — el libro más completo sobre EDA con Kafka como protagonista. El capítulo de comparación con brokers tradicionales es el más valioso para esta decisión.
- [Kafka Docs — Introduction](https://kafka.apache.org/documentation/#introduction) — la documentación oficial de Kafka, especialmente la sección "Design" que explica el modelo de log y consumer groups.
- [Redis Streams Docs](https://redis.io/docs/data-types/streams/) — documentación de Redis Streams con comparación explícita con Kafka. La opción más pragmática si ya usás Redis.
