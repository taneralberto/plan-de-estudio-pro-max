# 4.1 — Containerización Avanzada 🔴

- [ ] **Docker multi-stage builds, layer caching, security**

Imágenes mínimas (distroless, Alpine). Multi-stage builds para separar el entorno de build del de producción y reducir la superficie de ataque. `.dockerignore` para no copiar node_modules ni secrets al contexto. Escaneo de vulnerabilidades con Trivy o Snyk. Non-root users en containers — por qué correr como root en un container es un riesgo real.

**Por qué importa:** Una imagen de Node.js sin optimizar puede pesar 1.5GB y tener cientos de vulnerabilidades conocidas porque incluye el compilador, los devDependencies y herramientas de build que jamás se usan en producción. Una imagen bien construida con multi-stage pesa 150MB, corre como non-root, y tiene una fracción de la superficie de ataque. Cuando alguien explota una vulnerabilidad en tu container, la diferencia entre root y non-root determina si pueden escapar al host.

**Ejercicio:** Tomá los Dockerfiles actuales de FreePress (Angular frontend + NestJS backend). Medí el tamaño actual de cada imagen con `docker images`. Aplicá multi-stage builds, `.dockerignore`, imagen base Alpine o distroless, y non-root user. Medí el tamaño final. Corré Trivy sobre ambas versiones (`trivy image <nombre>`) y comparé la cantidad de vulnerabilidades críticas antes y después.

**Done cuando:** La imagen del backend de FreePress pesa menos de 300MB, corre como non-root, y Trivy reporta 0 vulnerabilidades críticas (HIGH o CRITICAL). Podés explicar por qué cada instrucción del Dockerfile está en ese orden específico y qué impacto tiene en el layer caching.

**Recursos:**
- [Docker Docs — Multi-stage builds](https://docs.docker.com/build/building/multi-stage/) — documentación oficial con ejemplos. Empezá acá.
- [Trivy](https://trivy.dev/) — escáner de vulnerabilidades open source. Instalalo y usalo desde el primer Dockerfile que escribas.
- [distroless images (Google)](https://github.com/GoogleContainerTools/distroless) — imágenes sin shell ni package manager. Ideal para producción.
- [*Docker Deep Dive*](https://www.amazon.com/Docker-Deep-Dive-Nigel-Poulton/dp/1916585256) de Nigel Poulton — el libro más accesible para entender Docker de verdad, no solo los comandos.

- [ ] **Docker Compose en profundidad**

Networking entre services (cómo los containers se ven entre sí por nombre de servicio), `depends_on` vs healthchecks (por qué `depends_on` solo espera que el container arranque, no que el servicio esté listo), volumes (bind mounts para desarrollo, named volumes para persistencia en producción), environment variables y secrets. Docker Compose profiles para activar distintos conjuntos de servicios según el entorno.

**Por qué importa:** El `docker-compose.yml` de FreePress es la especificación completa del entorno de desarrollo: Angular, NestJS, PostgreSQL, Redis. Si está mal configurado — por ejemplo, si NestJS arranca antes de que Postgres esté listo para aceptar conexiones — vas a tener race conditions que se manifiestan como errores intermitentes imposibles de reproducir. Entender healthchecks y networking es la diferencia entre un entorno de desarrollo confiable y uno que falla una de cada cinco veces que lo levantás.

**Ejercicio:** Revisá el `docker-compose.yml` de FreePress. Agregá healthchecks reales para PostgreSQL y Redis (no solo `depends_on`). Configurá profiles: uno `dev` que incluya herramientas de desarrollo (pgAdmin, Redis Insight) y uno `prod` que solo incluya los servicios esenciales. Asegurate de que ninguna variable de entorno sensible esté hardcodeada en el compose.

**Done cuando:** El entorno de FreePress levanta con `docker compose up` sin errores de conexión, sin importar el orden en que los containers terminan de iniciarse. Los healthchecks garantizan que NestJS no intenta conectarse a Postgres hasta que Postgres acepta conexiones reales. Podés explicar la diferencia entre un bind mount y un named volume y cuándo usar cada uno.

**Recursos:**
- [Docker Compose docs — Healthchecks](https://docs.docker.com/compose/compose-file/05-services/#healthcheck) — referencia oficial con ejemplos de postgres y redis.
- [Docker Compose docs — Profiles](https://docs.docker.com/compose/how-tos/profiles/) — cómo activar subconjuntos de servicios por entorno.
- [*Docker Deep Dive*](https://www.amazon.com/Docker-Deep-Dive-Nigel-Poulton/dp/1916585256) de Nigel Poulton — cubre Compose en profundidad con ejemplos prácticos.
