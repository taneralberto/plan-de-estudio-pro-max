# 4.3 — Infrastructure as Code (IaC) 🟡

---

- [ ] **Terraform — Infraestructura declarativa**

Qué es IaC y por qué no querés configurar servidores a mano nunca más. HCL (HashiCorp Configuration Language), providers (AWS, GCP, DigitalOcean), state (cómo Terraform sabe qué existe ya en la nube), plan/apply (ver cambios antes de aplicarlos), destroy. Modules para reutilizar configuración. Workspaces para manejar múltiples entornos (staging, producción) con el mismo código.

**Por qué importa:** Sin IaC, tu infraestructura es un snowflake — un servidor configurado a mano que nadie más sabe cómo reproducir. Si se cae, tardás horas en reconstruirlo de memoria. Con Terraform, si el servidor se cae, hacés `terraform apply` y en 5 minutos tenés infraestructura idéntica corriendo en otra región. Además, la infraestructura vive en Git: podés hacer code review de cambios de infra, hacer rollback si algo sale mal, y saber exactamente qué cambió y cuándo.

**Ejercicio:** Definí la infraestructura de FreePress en Terraform usando el provider de tu cloud preferido (DigitalOcean o AWS Free Tier). Incluí: una instancia de cómputo para el backend, una base de datos managed (RDS o DO Managed Postgres), DNS, y reglas de firewall. Usá variables para que el mismo código sirva para staging y producción cambiando solo los valores. El `terraform plan` tiene que poder correrse en CI.

**Done cuando:** Podés destruir completamente la infraestructura de FreePress con `terraform destroy` y reconstruirla idéntica con `terraform apply` en menos de 10 minutos, sin tocar ninguna consola web. Podés explicar qué pasa con el state si dos personas corren `terraform apply` al mismo tiempo y cómo se resuelve (remote state con locking).

**Recursos:**
- [HashiCorp Learn — Terraform](https://developer.hashicorp.com/terraform/tutorials) — tutoriales oficiales, excelentes y gratuitos. Empezá con el tutorial de AWS o DigitalOcean según tu cloud.
- [*Terraform: Up & Running*](https://www.amazon.com/Terraform-Running-Writing-Infrastructure-Code/dp/1098116747) de Yevgeniy Brikman — el libro de referencia práctico. Cubre módulos, remote state, y testing de infraestructura.
- [Terraform Registry](https://registry.terraform.io/) — repositorio oficial de providers y módulos. Antes de escribir algo desde cero, buscá si ya existe un módulo.

- [ ] **Ansible vs Terraform — Cuándo usar cada uno**

Terraform es declarativo e idempotente: describís el estado deseado de la infraestructura (qué servidores, qué redes, qué bases de datos) y Terraform lo hace realidad. Ansible es imperativo y procedural: describís las tareas a ejecutar en orden (instalá nginx, copiá este archivo de config, reiniciá el servicio). No son competencia — son complementarios: Terraform crea el servidor, Ansible lo configura.

**Por qué importa:** El error más común es usar Terraform para configurar software (instalando cosas con `remote-exec`) o usar Ansible para crear infraestructura. Mezclar las responsabilidades lleva a código frágil que es difícil de mantener. Entender la frontera clara entre "crear infraestructura" (Terraform) y "configurar software" (Ansible/cloud-init) te permite construir pipelines de IaC que son predecibles, testeables y reproducibles.

**Ejercicio:** Tomá la infraestructura que creaste con Terraform en el ejercicio anterior. Escribí un playbook de Ansible que configure el servidor: instale Docker, configure el firewall con ufw, copie el `docker-compose.yml` de FreePress, y levante los containers. El resultado final tiene que ser FreePress corriendo en producción con solo dos comandos: `terraform apply` y `ansible-playbook deploy.yml`.

**Done cuando:** Podés analizar un script de configuración de servidor (bash, Ansible, Terraform) e identificar qué responsabilidad está asumiendo cada uno y si está en el lugar correcto. Podés explicar por qué Ansible sin Terraform (configurando servidores creados a mano) sigue siendo un snowflake, y por qué Terraform sin Ansible (usando `remote-exec` para instalar software) es frágil.

**Recursos:**
- [Ansible Docs — Getting Started](https://docs.ansible.com/ansible/latest/getting_started/index.html) — documentación oficial con guía de inicio rápido. Instalalo y corré tu primer playbook antes de leer el libro.
- [*Ansible for DevOps*](https://www.ansiblefordevops.com/) de Jeff Geerling — el libro más práctico y accesible sobre Ansible. El autor también tiene un curso gratuito en YouTube.
- [Jeff Geerling en YouTube](https://www.youtube.com/@JeffGeerling) — videos sobre Ansible, Raspberry Pi, y homelab. Muy práctico para ver Ansible en contextos reales.
