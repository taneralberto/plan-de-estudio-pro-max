# 4.4 — Cloud Fundamentals 🔴

---

- [ ] **Los servicios que TODO dev debe conocer (AWS/GCP/Azure)**

Compute (EC2/Cloud Run/Droplets), Storage (S3/Cloud Storage/Spaces), Database managed (RDS/Cloud SQL), CDN (CloudFront/Cloud CDN), DNS (Route53/Cloud DNS), IAM (Identity and Access Management — roles, policies, least privilege), VPC (redes privadas virtuales — subnets, security groups, NAT). No los 200+ servicios — los 10 que usás el 90% del tiempo.

**Por qué importa:** FreePress va a necesitar infraestructura cloud real en algún momento: un servidor para el backend, un bucket para imágenes y assets, un CDN para distribuir el frontend globalmente, una base de datos managed que alguien más respalde. Sin entender estos servicios, vas a tomar decisiones por defecto que son caras, inseguras o difíciles de escalar. IAM mal configurado es el vector de ataque número uno en cloud — la mayoría de los breaches de datos en AWS son por credentials con permisos excesivos.

**Ejercicio:** Deployá FreePress en AWS (Free Tier) o DigitalOcean. Usá: una instancia de cómputo para el backend, S3 (o DO Spaces) para storage de imágenes con una política de bucket que solo permita acceso público de lectura a los assets, CloudFront (o DO CDN) para distribuir el frontend Angular, y RDS Postgres (o DO Managed DB) para la base de datos. Configurá IAM roles con least privilege — el backend solo debe tener acceso al bucket específico de FreePress, no a todos los buckets de la cuenta.

**Done cuando:** Podés configurar un entorno completo de producción desde cero usando solo la consola o CLI del cloud provider, con IAM roles mínimos, sin credenciales hardcodeadas en el código, y con el frontend servido desde CDN. Podés explicar qué es una VPC, por qué la base de datos debe estar en una subnet privada (sin acceso directo desde internet), y cómo el backend accede a ella de forma segura.

**Recursos:**
- [AWS Cloud Practitioner Essentials](https://aws.amazon.com/training/digital/aws-cloud-practitioner-essentials/) — curso gratuito oficial de AWS. La mejor introducción al ecosistema completo antes de ensuciarse las manos.
- [AWS Well-Architected Framework](https://aws.amazon.com/architecture/well-architected/) — los 6 pilares de diseño cloud (Operational Excellence, Security, Reliability, Performance, Cost, Sustainability). Leerlo después de tener el primer deploy funcionando.
- [DigitalOcean Tutorials](https://www.digitalocean.com/community/tutorials) — alternativa más accesible y barata que AWS para proyectos personales. La documentación es excepcionalmente clara.

- [ ] **Serverless — Functions as a Service**

Lambda (AWS), Cloud Functions (GCP), Azure Functions. El modelo de ejecución: la función duerme, un evento la dispara, ejecuta, y vuelve a dormir. Cold starts (cuánto tarda en arrancar una función fría y cómo mitigarlo). Pricing model (pagás por invocación y tiempo de ejecución, no por servidor idle). Limitaciones: tiempo máximo de ejecución, no hay estado local persistente, networking más complejo dentro de VPC.

**Por qué importa:** Serverless no reemplaza a los servidores — resuelve un problema específico muy bien: ejecutar código en respuesta a eventos esporádicos sin pagar por infraestructura idle. Para FreePress, casos de uso naturales son: procesar imágenes cuando se suben (resize, optimize), enviar emails transaccionales, generar PDFs o exports. Usar un servidor dedicado para esas tareas es pagar 24/7 por algo que se ejecuta 100 veces por día. Serverless lo hace gratis (dentro del free tier de Lambda).

**Ejercicio:** Implementá el procesamiento de imágenes de FreePress como una función Lambda: cuando un artículo se publica y sube una imagen de portada a S3, la Lambda se dispara automáticamente, genera 3 versiones optimizadas (thumbnail, medium, original comprimido con Sharp), y las sube de vuelta a S3. Medí el cold start time y el costo estimado mensual para 1000 artículos publicados.

**Done cuando:** Podés analizar una tarea de negocio (enviar un email, procesar un archivo, limpiar datos viejos) y decidir con criterio si es un buen candidato para serverless o si un worker tradicional es más apropiado. Podés explicar el problema del cold start, cuándo es un problema real (APIs síncronas con SLA de latencia) y cuándo no importa (procesamiento asíncrono en background).

**Recursos:**
- [AWS Lambda — Getting Started](https://docs.aws.amazon.com/lambda/latest/dg/getting-started.html) — documentación oficial con ejemplos en Node.js/TypeScript.
- [Serverless Framework](https://www.serverless.com/framework/docs) — herramienta open source para deployar funciones en cualquier cloud con un `serverless.yml`. Abstrae el boilerplate de configuración.
- [*Learning Serverless*](https://www.amazon.com/Learning-Serverless-Building-Applications-Functions/dp/1492057134) de Jason Katzer — libro práctico sobre arquitecturas serverless reales, incluyendo cuándo NO usarla.

- [ ] **Managed Services vs Self-Hosted**

El trade-off central del cloud: cuándo delegarle la operación a alguien más (managed service) y cuándo asumirla vos (self-hosted). Ejemplos: RDS vs montar PostgreSQL en un EC2 vos mismo; Auth0/Clerk vs implementar auth desde cero; Elastic Cloud vs montar Elasticsearch en tu VPS; Cloudflare Workers vs tu propio CDN. El criterio no es el costo — es la ecuación entre costo operacional (tiempo de tu equipo) + costo financiero + control + lock-in.

**Por qué importa:** Cada managed service que usás es tiempo de operación que no gastás vos. RDS te da backups automáticos, failover, actualizaciones de seguridad y monitoreo — todo sin que toques nada. Pero también te cuesta 5x más que un Postgres self-hosted y te acopla al cloud provider. La decisión incorrecta en cualquier dirección tiene consecuencias: self-hosted todo es un segundo trabajo de sysadmin; managed todo puede volverse prohibitivamente caro o impedir migración futura.

**Ejercicio:** Para FreePress, tomá 3 decisiones concretas de managed vs self-hosted con justificación escrita: (1) Base de datos: ¿managed Postgres o self-hosted? (2) Autenticación: ¿Clerk/Auth0 o implementación propia con JWT? (3) Search: ¿Algolia/Typesense Cloud o Elasticsearch self-hosted? Para cada decisión, documentá los criterios que usaste (costo, complejidad operacional, lock-in, requisitos de compliance).

**Done cuando:** Podés recibir un nuevo proyecto con su contexto (etapa, equipo, presupuesto, requisitos de compliance) y producir una tabla de decisiones de managed vs self-hosted para los servicios principales, con justificación técnica y económica para cada elección. Podés identificar cuándo el lock-in de un managed service es un riesgo real vs cuando es un trade-off aceptable.

**Recursos:**
- [AWS Well-Architected Framework — Cost Optimization](https://docs.aws.amazon.com/wellarchitected/latest/cost-optimization-pillar/welcome.html) — el pilar de costos tiene una sección específica sobre cómo evaluar managed vs self-hosted.
- [SST (Serverless Stack)](https://sst.dev/) — framework moderno para cloud que toma muchas decisiones de managed services por vos. Útil para ver qué servicios managed son el consenso de la industria.
- [Hacker News — "Ask HN: When to use managed services"](https://hn.algolia.com/?q=managed+services+vs+self+hosted) — la discusión real de la industria sobre estos trade-offs. Buscar threads recientes para ver qué está usando gente real en producción.
