# 4.5 — CI/CD (Integración y Entrega Continua) 🔴 [CORE]

- [ ] **Pipelines de CI/CD — De verdad**

No es solo "correr tests automáticos". Es el flujo completo y ordenado: lint → test unitarios → test de integración → build → análisis de seguridad (SAST) → build de imagen Docker → push al registry → deploy a staging → smoke tests → deploy a producción. GitHub Actions, GitLab CI, Jenkins. Entender stages, jobs, artifacts (qué produce cada job y qué consume el siguiente), caching de dependencias, matrix builds (correr los mismos tests contra múltiples versiones de Node), y gestión de secrets en CI (nunca en el código).

**Por qué importa:** Un pipeline de CI bien construido es el único firewall real contra regressions en producción. Sin él, cada deploy es un acto de fe. Con él, nadie puede mergear código roto a main — el pipeline lo rechaza antes. El costo de detectar un bug en CI (segundos) vs en producción (horas de debugging + impacto en usuarios) justifica completamente la inversión de configurarlo bien desde el principio.

**Ejercicio:** Configurá un pipeline completo para FreePress en GitHub Actions con los siguientes stages: (1) lint con ESLint y Prettier — falla si hay errores; (2) tests unitarios del backend — falla si hay tests rotos; (3) build de imagen Docker del backend con caché de layers; (4) push de la imagen al GitHub Container Registry (GHCR) con tag del commit SHA; (5) deploy automático a un entorno de staging cuando se mergea a `main`. Ningún secret debe aparecer en los logs.

**Done cuando:** Cada PR de FreePress corre el pipeline completo automáticamente y falla con mensaje claro si algo está roto. Un merge a `main` despliega a staging sin intervención manual. Podés explicar qué es un artifact en GitHub Actions, cómo pasarlo entre jobs, y por qué importa para evitar re-buildear lo mismo dos veces.

**Recursos:**
- [GitHub Actions docs](https://docs.github.com/en/actions) — documentación oficial excelente. La sección de "Workflows" y "Using secrets" son las más importantes para empezar.
- [*Continuous Delivery*](https://www.amazon.com/Continuous-Delivery-Deployment-Automation-Addison-Wesley/dp/0321601912) de Jez Humble y David Farley — el clásico que definió el campo. Leer al menos los primeros 5 capítulos para entender los principios antes de la implementación.
- [actionlint](https://github.com/rhysd/actionlint) — linter estático para archivos de GitHub Actions. Te avisa de errores antes de pushear y esperar que el pipeline falle.

- [ ] **Deploy Strategies**

Blue-green (dos entornos idénticos — uno activo, uno en standby — se switchea el tráfico de golpe), canary (se manda un porcentaje pequeño del tráfico a la nueva versión y se observa antes de rollout completo), rolling updates (se reemplazan las instancias de a una manteniendo disponibilidad). Feature flags (LaunchDarkly, Unleash, o flags simples en base de datos) para separar deploy de release — podés deployar código que nadie puede usar todavía. Rollback strategies: cuándo hacer rollback automático y cuándo es mejor avanzar con un fix.

**Por qué importa:** "Deploy" y "release" son conceptos distintos, y confundirlos es un riesgo operacional. Si deployás un feature nuevo y los errores se disparan al 2%, ¿cuánto tiempo tardás en detectarlo, decidir hacer rollback, y ejecutarlo? Con una estrategia de deploy bien definida, un canary deployment lo detecta en 5 minutos con el 1% de los usuarios afectados. Sin ella, el 100% de los usuarios están expuestos y el rollback es manual y lento.

**Ejercicio:** Implementá feature flags en FreePress usando una tabla simple en PostgreSQL (o una variable de entorno) que controle si una nueva feature (ej: el editor enriquecido nuevo) está habilitada para todos los usuarios, para un porcentaje, o solo para usuarios con un flag específico en su perfil. Simulá un canary release: deployá la feature habilitada solo para el 10% de los usuarios (basado en user ID % 10) y agregá un log de qué usuarios la están viendo.

**Done cuando:** Podés explicar cuándo usarías blue-green vs canary vs rolling update para un deploy específico de FreePress (por ejemplo: una migración de base de datos que cambia el schema vs un cambio solo en la UI). Podés describir cómo un feature flag te permite hacer un rollback instantáneo sin un nuevo deploy.

**Recursos:**
- [Martin Fowler — "Feature Toggles"](https://martinfowler.com/articles/feature-toggles.html) — el artículo de referencia sobre tipos de feature flags y cuándo usar cada uno. Leer antes de implementar cualquier sistema de flags.
- [Unleash](https://www.getunleash.io/) — plataforma open source de feature flags. Se puede self-hostear. Útil para ver cómo se diseña este sistema a escala real.
- [*Continuous Delivery*](https://www.amazon.com/Continuous-Delivery-Deployment-Automation-Addison-Wesley/dp/0321601912) de Jez Humble — los capítulos de deployment strategies son los más prácticos del libro.

- [ ] **GitOps**

Infraestructura y configuración de deploys declaradas en Git como fuente de verdad absoluta. ArgoCD y Flux como implementaciones: observan el estado del repositorio y reconcilian el cluster para que siempre coincida con lo declarado en Git. Pull-based vs push-based deployment: en push-based (CI clásico) el pipeline empuja cambios al cluster; en pull-based (GitOps) el cluster observa Git y se actualiza solo. Audit trail completo: cada cambio de infraestructura tiene un commit, un autor, y un mensaje.

**Por qué importa:** En un modelo tradicional de CI/CD, el pipeline tiene credenciales del cluster con permisos amplios para hacer deploys. Si el sistema de CI se compromete, el atacante tiene acceso directo al cluster de producción. GitOps invierte el modelo: el cluster tiene credenciales para leer Git (menos crítico), y ningún sistema externo necesita acceso de escritura al cluster. Además, "qué está en producción" siempre tiene una respuesta exacta — es lo que está en la rama `main` del repo de infraestructura.

**Ejercicio:** Configurá ArgoCD en el cluster local de Minikube que creaste en 4.2. Creá un repositorio de infraestructura separado con los manifiestos YAML de FreePress. Configurá ArgoCD para que observe ese repo y sincronice automáticamente los cambios. Simulá un deploy: modificá la versión de la imagen en el YAML, commiteá a Git, y observá cómo ArgoCD detecta el cambio y actualiza el Deployment en el cluster sin que vos hagas nada más.

**Done cuando:** Podés explicar la diferencia entre push-based y pull-based deployment con sus trade-offs de seguridad. Podés describir cómo GitOps resuelve el problema de "configuration drift" (cuando el estado real del cluster difiere de lo que dice el código) y cómo detecta y corrige esa desviación automáticamente.

**Recursos:**
- [ArgoCD Docs — Getting Started](https://argo-cd.readthedocs.io/en/stable/getting_started/) — la guía oficial para instalar ArgoCD en Minikube y hacer el primer sync. Muy accesible.
- [WeaveWorks — "What is GitOps?"](https://www.weave.works/blog/what-is-gitops-really) — el artículo original que acuñó el término GitOps. Contexto histórico y principios fundacionales.
- [*GitOps and Kubernetes*](https://www.amazon.com/GitOps-Kubernetes-Continuous-Deployment-Environments/dp/1617297275) de Billy Yuen et al. — el libro más completo sobre GitOps con ArgoCD y Flux en Kubernetes.
