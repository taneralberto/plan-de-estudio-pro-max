# 4.6 — Observabilidad (Los 3 Pilares) 🔴 [CORE]

- [ ] **Logs — Structured Logging**

No más `console.log("algo pasó")`. Structured logging con JSON: cada log es un objeto con campos fijos (`timestamp`, `level`, `requestId`, `userId`, `message`, `error`). Log levels y cuándo usar cada uno (debug solo en desarrollo, info para eventos de negocio importantes, warn para situaciones anómalas pero recuperables, error para fallos que requieren atención). Correlation IDs: un ID único generado por request que se propaga a todos los logs de esa request, para poder reconstruir exactamente qué pasó. ELK Stack (Elasticsearch + Logstash + Kibana) o Grafana Loki como alternativa más liviana.

**Por qué importa:** Cuando FreePress se cae a las 3am, tenés dos herramientas para entender qué pasó: los logs y las métricas. Si tus logs son `console.log("error en el controller")` sin contexto, sin stacktrace completo, sin el ID de usuario, sin el request ID — estás ciego. Un structured log con correlation ID te permite filtrar en Kibana todos los logs de un request específico y ver exactamente la secuencia de eventos que llevó al error, en segundos.

**Ejercicio:** Implementá structured logging en FreePress usando `pino` (la librería más performante para Node.js). Configurá un middleware de NestJS que genere un `requestId` único por request (usando `uuid` o `nanoid`) y lo adjunte a todos los logs de esa request usando AsyncLocalStorage. El log de cada request completado debe incluir: `requestId`, `method`, `path`, `statusCode`, `duration`, `userId` (si está autenticado). Configurá Grafana Loki con Docker Compose para ver los logs centralizados.

**Done cuando:** Podés tomar un error reportado por un usuario (con su userId y la hora aproximada) y reconstruir en menos de 2 minutos el request completo que causó el error — todos los logs, en orden, incluyendo las queries a la base de datos que se ejecutaron — usando solo el sistema de logs. El log de un error incluye el stack trace completo, el contexto del request, y el userId.

**Recursos:**
- [Pino — Node.js logger](https://getpino.io/#/) — la librería de logging más rápida para Node.js/NestJS. Documentación excelente con ejemplos de integración con NestJS.
- [Grafana Loki](https://grafana.com/oss/loki/) — alternativa liviana al ELK Stack. Más fácil de operar self-hosted. Ideal para empezar.
- [*Observability Engineering*](https://www.amazon.com/Observability-Engineering-Achieving-Production-Excellence/dp/1492076445) de Charity Majors — LA referencia del campo. Los primeros 3 capítulos definen qué es observabilidad real y por qué los logs tradicionales no son suficientes.

- [ ] **Métricas — Qué medir y cómo**

RED method para servicios (Rate — cuántas requests por segundo, Errors — qué porcentaje falla, Duration — cuánto tarda el percentil 95 y 99). USE method para recursos de infraestructura (Utilization — qué tan usado está el CPU/memoria/disco, Saturation — hay cosas esperando en cola, Errors — cuántos errores de hardware/OS). Prometheus como sistema de recolección de métricas (pull-based: Prometheus va a buscar las métricas al servicio). Grafana como visualización. Alertas que no te despierten a las 3am por falsos positivos: alertar sobre síntomas (latencia > 500ms, error rate > 1%) no sobre causas (CPU > 80%).

**Por qué importa:** Las métricas te dicen QUÉ está roto antes de que los usuarios te lo digan. Sin métricas, sabés que algo está roto cuando el teléfono suena. Con Prometheus + Grafana correctamente configurados, sabés que la latencia del endpoint `/api/articles` subió al percentil 99 hace 2 minutos — antes de que ningún usuario se queje. La diferencia entre alertar sobre síntomas vs causas es la diferencia entre alerts accionables y alert fatigue.

**Ejercicio:** Configurá Prometheus y Grafana con Docker Compose para FreePress. Exponé métricas del backend NestJS usando `@willsoto/nestjs-prometheus`. Creá un dashboard en Grafana con los tres pilares del RED method para el servicio principal. Configurá al menos dos alertas: (1) error rate > 1% por 5 minutos consecutivos, y (2) latencia P99 > 1 segundo. Las alertas deben llegar a un canal de Slack o como notificación de email.

**Done cuando:** Tenés un dashboard de Grafana para FreePress que muestra Rate, Error rate y Duration (P50, P95, P99) en tiempo real. Podés generar carga artificial con `k6` o `autocannon` y ver en tiempo real cómo cambian las métricas. Podés explicar por qué alertar sobre "CPU > 80%" es peor que alertar sobre "Error rate > 1%" y cuál es la raíz de esa diferencia.

**Recursos:**
- [Prometheus Docs — Getting Started](https://prometheus.io/docs/introduction/overview/) — documentación oficial clara con ejemplos de configuración.
- [Grafana Docs — Dashboards](https://grafana.com/docs/grafana/latest/dashboards/) — guía para crear dashboards. Empezá con el dashboard pre-built de Node.js metrics.
- [USE Method — Brendan Gregg](https://www.brendangregg.com/usemethod.html) — el artículo original del inventor del método. Una página, fundamental.

- [ ] **Distributed Tracing**

Cómo seguir un request a través de múltiples servicios. En un monolito, el stack trace te dice todo. En un sistema con múltiples servicios (o incluso en un monolito con múltiples capas), el stack trace no cruza fronteras de red. OpenTelemetry es el estándar abierto que unifica la instrumentación: un SDK, múltiples backends (Jaeger, Zipkin, Tempo). Spans (unidad de trabajo con duración), traces (colección de spans de un mismo request), context propagation (cómo el trace ID viaja entre servicios a través de headers HTTP).

**Por qué importa:** Cuando un request de FreePress tarda 3 segundos y pasa por el backend NestJS, luego a PostgreSQL, luego a Redis, luego a un servicio de thumbnails — ¿dónde está el tiempo? Sin tracing, es un misterio. Con OpenTelemetry y Jaeger, podés abrir el trace de ese request y ver exactamente: 50ms en el handler, 2700ms esperando una query a Postgres (ahí está el problema), 150ms en Redis. El bottleneck es obvio en segundos.

**Ejercicio:** Instrumentá FreePress con OpenTelemetry usando `@opentelemetry/sdk-node`. Configurá el SDK para exportar traces a Jaeger (en Docker Compose). Instrumentá automáticamente: las requests HTTP de NestJS, las queries de TypeORM a PostgreSQL, y los comandos de Redis. Ejecutá el flujo de publicar un artículo y abrí el trace en Jaeger — identificá cuánto tiempo toma cada operación y cuál es la más lenta.

**Done cuando:** Podés abrir Jaeger, buscar un trace por `requestId`, y ver el waterfall completo de todas las operaciones que ocurrieron durante ese request con sus duraciones exactas. Podés identificar visualmente el span más lento en un trace y correlacionarlo con el código que lo genera. Podés explicar cómo el trace ID se propaga entre servicios y qué pasa si un servicio no está instrumentado (se pierde la cadena).

**Recursos:**
- [*Observability Engineering*](https://www.amazon.com/Observability-Engineering-Achieving-Production-Excellence/dp/1492076445) de Charity Majors — LA referencia. Los capítulos sobre tracing y OpenTelemetry son los más detallados del mercado.
- [OpenTelemetry Docs — Node.js](https://opentelemetry.io/docs/languages/js/getting-started/nodejs/) — guía oficial de instrumentación para Node.js con ejemplos de NestJS.
- [Jaeger](https://www.jaegertracing.io/) — el backend de tracing open source más usado. Documentación clara con Docker Compose example incluido.
