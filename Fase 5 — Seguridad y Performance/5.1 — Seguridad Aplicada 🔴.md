# 5.1 — Seguridad Aplicada 🔴 [CORE]

No es una feature que se agrega al final — es una propiedad que se diseña desde el principio.

- [ ]  **OWASP Top 10 — Las vulnerabilidades que TENÉS que conocer**

Las 10 categorías de vulnerabilidades más críticas en aplicaciones web, actualizadas por la industria. Injection (SQL, NoSQL, Command), XSS (Stored, Reflected, DOM), CSRF, Broken Authentication, Security Misconfiguration. No teoría — PRACTICAR encontrando y explotando cada una.

**Por qué importa:** El 90% de los breaches explotó vulnerabilidades conocidas, no zero-days. Si no sabés qué es un XSS stored, estás dejando la puerta abierta sin saberlo.

**Ejercicio:** Completá al menos 5 labs de [PortSwigger Web Security Academy](https://portswigger.net/web-security) — uno por categoría OWASP. Después auditá el módulo de autenticación de FreePress buscando las mismas vulnerabilidades.

**Done cuando:** Podés explicar la diferencia entre XSS reflected y stored, y mostrar en código cómo se explota y cómo se mitiga cada uno sin mirar apuntes.

**Recursos:**

- [OWASP Top 10](https://owasp.org/www-project-top-ten/) — la referencia oficial
  - 🇪🇸 [OWASP Top 10 en español](https://owasp.org/www-project-top-ten/es/) — traducción oficial disponible
- [PortSwigger Web Security Academy](https://portswigger.net/web-security) — labs interactivos gratuitos, los mejores de la industria
  - 🇪🇸 Disponible con subtítulos/interfaz en español en la configuración

- [ ]  **Authentication y Authorization — OAuth 2.0, OpenID Connect, RBAC**

La diferencia entre autenticación (quién sos) y autorización (qué podés hacer). OAuth 2.0 flows: Authorization Code (con PKCE para SPAs), Client Credentials. JWT bien usado: qué va en el payload, qué NO va, por qué `alg: none` es una bomba. Refresh tokens y rotación. RBAC vs ABAC.

**Por qué importa:** Auth mal implementada es la vulnerabilidad #1 en aplicaciones reales. JWT no es magia — si no entendés el protocolo, vas a cometer errores que ningún linter te va a detectar.

**Ejercicio:** Implementá un flujo OAuth 2.0 Authorization Code + PKCE desde cero (sin librerías) para ENTENDER el protocolo. Después implementá RBAC en FreePress (roles: admin, editor, autor, lector) con guards en NestJS.

**Done cuando:** Podés dibujar en una pizarra el flujo completo de OAuth 2.0 Authorization Code con PKCE, explicar qué hace cada paso y por qué, y señalar qué falla si se omite el PKCE en una SPA.

**Recursos:**

- [OAuth 2.0 RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749) — la especificación original, vale leerla
- [jwt.io](https://jwt.io) — decodificá y entendé la estructura de un JWT
- Artículo: *"Stop using JWT for sessions"* de joepie91 — perspectiva crítica necesaria
- Libro: *OAuth 2 in Action* de Justin Richer — si querés ir en profundidad

- [ ]  **Supply Chain Security**

Dependencias maliciosas en npm, lockfiles, `npm audit`, Dependabot/Renovate. Qué es un ataque de typosquatting (publicar `lodahs` esperando que alguien escriba mal `lodash`). Qué es un ataque de dependency confusion. Por qué `npm install` puede ejecutar código arbitrario en tu máquina.

**Por qué importa:** El ataque a SolarWinds comprometió miles de organizaciones sin tocar su código — atacó sus dependencias. Tu `node_modules` tiene miles de paquetes que no controlás.

**Ejercicio:** Corré `npm audit` en FreePress, revisá cada vulnerabilidad encontrada y clasificala: ¿es explotable en tu contexto? ¿Cómo la mitigás? Configurá Dependabot o Renovate en el repo.

**Done cuando:** Podés explicar la diferencia entre un ataque de typosquatting y uno de dependency confusion, y tenés Dependabot o Renovate corriendo en FreePress con al menos una PR de actualización resuelta.

**Recursos:**

- [npm audit docs](https://docs.npmjs.com/cli/v10/commands/npm-audit) — documentación oficial
- Artículo: *"Dependency Confusion: How I Hacked Into Apple, Microsoft and Dozens of Other Companies"* de Alex Birsan — el paper original del ataque, lectura obligatoria
- [Dependabot docs](https://docs.github.com/en/code-security/dependabot) — configuración en GitHub

- [ ]  **Secrets Management**

La diferencia entre configuración y secretos. Por qué `.env` en el repo es un desastre esperando pasar. HashiCorp Vault, AWS Secrets Manager, Doppler como alternativas reales. Rotación de secrets: por qué importa y cómo automatizarla. Qué hacer cuando se filtran credentials (y van a filtrarse).

**Por qué importa:** GitHub escanea repos públicos buscando API keys y las revoca automáticamente — porque pasa TODO el tiempo. Un secret en el historial de git existe para siempre aunque lo "borres".

**Ejercicio:** Auditá FreePress: ¿hay algún secret en el historial de git? Usá `git log -p | grep -i "secret\|password\|key\|token"` para verificar. Configurá Doppler o variables de entorno en tu CI/CD pipeline. Documentá en un ADR cómo se manejan los secrets en el proyecto.

**Done cuando:** FreePress no tiene ningún secret en el código ni en el historial de git, y tenés un proceso documentado para rotar credentials si se filtran.

**Recursos:**

- [Doppler](https://www.doppler.com/) — secrets manager con free tier, el más simple para empezar
- [HashiCorp Vault](https://www.vaultproject.io/) — el estándar enterprise
- Artículo: *"How to handle secrets in git"* — buscá el de GitGuardian, tienen guías excelentes
- [git-secrets](https://github.com/awslabs/git-secrets) — herramienta de AWS para prevenir commits con secrets
