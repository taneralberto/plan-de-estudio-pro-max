# 5.2 — Performance Engineering 🔴

Performance no es "hacer las cosas rápido". Es entender POR QUÉ son lentas, medir con precisión, y optimizar lo que realmente importa. La regla de oro: nunca optimices sin datos.

---

- [ ]  **Profiling — Medir ANTES de optimizar**

La regla de oro: "No optimices sin datos". Node.js profiling con `--prof` y `clinic.js`, Chrome DevTools Performance tab, flamegraphs. Cómo leer un flamegraph: el eje X es tiempo, el eje Y es call stack — buscás las mesetas anchas. Identificar bottlenecks REALES, no los que imaginás.

**Por qué importa:** El 90% de las optimizaciones prematuras están en el lugar equivocado. Sin profiling, estás adivinando. Con profiling, sabés exactamente dónde está el segundo que querés recuperar.

**Ejercicio:** Profileá FreePress con `clinic.js flame`, identificá el bottleneck más costoso, optimizá, medí la mejora antes/después con números concretos. Documentá el proceso: hipótesis → medición → cambio → resultado.

**Done cuando:** Podés tomar una aplicación Node.js lenta, generar un flamegraph, identificar el hotspot, y proponer una solución justificada con datos — sin adivinar.

**Recursos:**

- [clinic.js](https://clinicjs.org/) — herramienta de profiling para Node.js, la más accesible
- [Chrome DevTools Performance](https://developer.chrome.com/docs/devtools/performance/) — para el lado frontend
- Artículo: *"How to read a flamegraph"* de Brendan Gregg — el inventor del flamegraph explica su creación

- [ ]  **Web Performance — Core Web Vitals**

LCP (Largest Contentful Paint), INP (Interaction to Next Paint), CLS (Cumulative Layout Shift). Qué mide cada uno, por qué Google los eligió como señales de ranking, y cómo optimizarlos. Lighthouse para auditorías, WebPageTest para mediciones reales desde distintas ubicaciones geográficas.

**Por qué importa:** Core Web Vitals afectan el SEO de FreePress directamente. Un LCP > 4s en mobile puede hundirte en los resultados de búsqueda. Performance no es solo UX — es negocio.

**Ejercicio:** Corré Lighthouse sobre FreePress. Identificá los 3 problemas más críticos que reporta y arreglalos. Medí el impacto en los scores antes/después.

**Done cuando:** Podés explicar qué mide cada Core Web Vital, por qué fue elegido sobre otras métricas, y qué técnicas concretas mejoran cada uno (lazy loading para LCP, debouncing para INP, reserva de espacio para CLS).

**Recursos:**

- [web.dev/performance](https://web.dev/performance/) — guías de Google, el recurso más completo
- [WebPageTest](https://www.webpagetest.org/) — medición real desde distintas ubicaciones y dispositivos
- [Lighthouse](https://developer.chrome.com/docs/lighthouse/) — auditorías automatizadas

- [ ]  **Caching Strategies**

Cache-Control headers (`max-age`, `stale-while-revalidate`, `no-store`), ETags y validación condicional, CDN caching (Cloudflare, CloudFront), application-level caching con Redis, memoización en memoria. Cache invalidation: por qué es uno de los dos problemas difíciles de CS. Cuándo cachear y cuándo NO (datos personales, datos que cambian mucho, operaciones con side effects).

**Por qué importa:** Un cache bien diseñado puede reducir la carga de tu base de datos un 80% y hacer tu app 10x más rápida. Un cache mal diseñado sirve datos stale y crea bugs imposibles de reproducir.

**Ejercicio:** Implementá en FreePress dos niveles de cache: HTTP caching para assets estáticos (Cache-Control headers correctos) y Redis cache para las queries más frecuentes (ej: lista de artículos publicados). Medí el impacto en tiempo de respuesta.

**Done cuando:** Podés diseñar una estrategia de cache para un endpoint dado, justificar el TTL elegido, y explicar cómo invalidás el cache cuando el dato cambia — sin dejar datos stale.

**Recursos:**

- [MDN — HTTP Caching](https://developer.mozilla.org/en-US/docs/Web/HTTP/Caching) — la referencia para cache headers
- Artículo: *"Caching best practices & max-age gotchas"* de Jake Archibald — lectura obligatoria
- [Redis docs — Patterns](https://redis.io/docs/manual/patterns/) — patrones de cache con Redis

- [ ]  **Database Performance**

Query optimization, `EXPLAIN ANALYZE` (cómo leer el plan de ejecución), connection pooling con PgBouncer, el problema N+1 queries y cómo detectarlo. Índices compuestos, covering indexes, partial indexes — cuándo crear cada uno. Materialized views para queries pesadas. *(Nota: los fundamentos de SQL y transactions se cubren en la Fase 7. Acá el foco es performance y profiling, no modelado.)*

**Por qué importa:** El bottleneck más común en aplicaciones web no es el código — es la base de datos. Un query sin índice sobre una tabla de 10M filas puede pasar de 10ms a 30 segundos. `EXPLAIN ANALYZE` es tu microscopio.

**Ejercicio:** Encontrá y arreglá un N+1 query en FreePress (o creá uno intencionalmente para practicar). Usá `EXPLAIN ANALYZE` para mostrar el impacto antes/después. Creá al menos un índice compuesto y medí la mejora.

**Done cuando:** Podés leer un `EXPLAIN ANALYZE` output, identificar un Sequential Scan que debería ser un Index Scan, crear el índice correcto, y medir la mejora con números reales.

**Recursos:**

- [explain.dalibo.com](https://explain.dalibo.com/) — visualizador de EXPLAIN ANALYZE para PostgreSQL
- Artículo: *"Use the Index, Luke"* en [use-the-index-luke.com](https://use-the-index-luke.com/) — el mejor recurso sobre índices en bases de datos relacionales
- [pgBadger](https://pgbadger.darold.net/) — analizador de logs de PostgreSQL para encontrar queries lentas

- [ ]  **Load Testing**

La diferencia entre load testing (carga normal), stress testing (carga extrema), soak testing (carga sostenida en el tiempo) y spike testing (pico repentino). k6, Artillery, o Apache JMeter para simular usuarios concurrentes. Capacity planning: cuántos usuarios soporta tu sistema antes de degradarse. Cómo interpretar percentiles (p50, p95, p99) — el p99 es lo que vive tu usuario más desafortunado.

**Por qué importa:** Tu app puede responder perfecto con 1 usuario y caerse con 100. Load testing te dice dónde está ese límite ANTES de que lo encuentre un usuario real en producción.

**Ejercicio:** Escribí un script de k6 que simule 100 usuarios concurrentes usando FreePress (login, leer artículos, publicar). Encontrá el punto de quiebre. Documentá los resultados: qué endpoint falla primero y por qué.

**Done cuando:** Podés escribir un script de k6 desde cero, correrlo contra FreePress, interpretar los resultados (throughput, latencia por percentil, error rate), e identificar el cuello de botella.

**Recursos:**

- [k6 docs](https://k6.io/docs/) — la herramienta más moderna y amigable para load testing
- [k6 University](https://university.k6.io/) — curso gratuito oficial
- Artículo: *"k6 vs Artillery vs JMeter"* — cualquier comparativa reciente para entender cuándo usar cada uno
