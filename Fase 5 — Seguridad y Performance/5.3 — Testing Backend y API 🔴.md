# 5.3 — Testing Backend y API 🔴 [ADVANCED]

Testing es una disciplina de diseño: los tests te obligan a pensar en contratos, responsabilidades y límites ANTES de implementar.

- [ ]  **La Testing Trophy y la Pirámide de Tests — El marco conceptual primero**

Antes de escribir un solo test, entendé el PORQUÉ de cada tipo. La Pirámide clásica (muchos unit, pocos E2E) y la Testing Trophy de Kent C. Dodds (más peso en integration). Por qué el ratio cambia según el tipo de aplicación. El concepto clave: cuánta confianza te da cada tipo de test vs cuánto cuesta escribirlo y mantenerlo.

**Por qué importa:** Sin este marco, escribís tests al azar — muchos mocks que no prueban nada real, o E2E lentos que se rompen con cualquier cambio de UI. El marco te dice DÓNDE invertir tu tiempo de testing.

**Ejercicio:** Dibujá la pirámide/trophy que debería tener FreePress dado su stack (Angular + NestJS + PostgreSQL). Justificá cada nivel: ¿cuántos unit tests? ¿cuántos integration? ¿cuántos E2E? ¿Por qué ese ratio?

**Done cuando:** Podés argumentar por qué una aplicación con lógica de negocio compleja debería tener más integration tests que unit tests, y cuándo ese argumento se invierte.

**Recursos:**

- Artículo: *"The Testing Trophy"* de Kent C. Dodds — la referencia para entender el ratio
- Artículo: *"Write tests. Not too many. Mostly integration."* de Guillermo Rauch — complementario
- Artículo: *"The Practical Test Pyramid"* de Ham Vocke (martinfowler.com) — la visión clásica con matices

- [ ]  **Unit Testing — Qué testear y qué NO**

No testees implementación, testeá comportamiento. La diferencia entre un test frágil (acoplado al "cómo") y un test robusto (acoplado al "qué"). Test doubles: mocks, stubs, spies, fakes — cuándo usar cada uno y por qué mockear TODO es un antipatrón. Coverage: medilo pero no lo idolatres. Un 100% de coverage con mocks vacíos no vale nada.

**Por qué importa:** Los unit tests mal escritos son peor que no tener tests — te dan falsa confianza y se rompen con cada refactor, frenando el desarrollo en lugar de acelerarlo.

**Ejercicio:** Escribí tests unitarios para el servicio de artículos de FreePress usando Jest/Vitest. Practicá la diferencia entre mock y stub en el mismo test suite. Después deliberadamente sobreescribí un test con mocks excesivos y explicá por qué es un problema.

**Done cuando:** Podés mirar un test y decir si está testeando comportamiento o implementación, justificar cuándo usar un stub vs un mock vs un fake, y refactorizar un test frágil sin cambiar lo que verifica.

**Recursos:**

- Libro: *Unit Testing: Principles, Practices, and Patterns* de Vladimir Khorikov — EL libro de unit testing, sin competencia
- Artículo: *"Mocks Aren't Stubs"* de Martin Fowler — clarifica la terminología de una vez por todas

- [ ]  **Integration Testing y API Testing**

Testear endpoints reales contra una base de datos real (no mocks). Testcontainers para levantar PostgreSQL en Docker durante los tests — el enfoque más confiable. Supertest para HTTP assertions en NestJS. Contract testing con Pact: verificar que el contrato entre cliente y servidor se respeta. Por qué los integration tests te dan más confianza que 100 unit tests con mocks.

**Por qué importa:** Un unit test puede pasar perfectamente mientras tu API devuelve un 500 en producción porque el ORM genera una query inválida. Los integration tests cierran esa brecha entre "funciona en aislamiento" y "funciona de verdad".

**Ejercicio:** Escribí integration tests para la API REST de FreePress que levanten PostgreSQL real con Testcontainers. Cubrí al menos: crear artículo, publicar artículo, listar artículos con paginación, y un caso de error (ej: artículo no encontrado devuelve 404, no 500).

**Done cuando:** Tenés un suite de integration tests en FreePress que corre en CI, levanta una base de datos real, y falla si el contrato de la API cambia sin querer.

**Recursos:**

- [Testcontainers para Node.js](https://node.testcontainers.org/) — la forma correcta de testear con bases de datos reales
- [Supertest](https://github.com/ladjs/supertest) — HTTP assertions para Node.js
- [Pact](https://pact.io/) — contract testing entre servicios
- Artículo: *"Testing NestJS with Testcontainers"* — buscá el más reciente, hay varios buenos

- [ ]  **E2E Testing**

Playwright o Cypress para testear el flujo completo (frontend + backend + base de datos). Cuándo vale la pena y cuándo es overkill: los E2E son lentos, frágiles si se abusa de selectores de UI, y costosos de mantener. Test data management: fixtures, factories, cómo limpiar el estado entre tests. Visual regression testing con Playwright screenshots.

**Por qué importa:** Un E2E test es el único que verifica que el sistema COMPLETO funciona como lo vería un usuario real. No reemplaza los otros niveles — los complementa para los flujos críticos (login, publicar artículo, proceso de pago).

**Ejercicio:** Escribí tests E2E con Playwright para los 3 flujos más críticos de FreePress: (1) registro de usuario y primer login, (2) crear y publicar un artículo, (3) un editor intenta acceder a una ruta de admin y recibe 403. Corrélos en CI.

**Done cuando:** Podés decidir si un flujo merece un E2E test o no (justificado por costo/beneficio), escribir el test con Playwright, y configurarlo para que corra en CI sin ser flaky.

**Recursos:**

- [Playwright docs](https://playwright.dev/) — la herramienta que está ganando la batalla contra Cypress
- [Playwright Test Generator](https://playwright.dev/docs/codegen) — genera tests grabando interacciones, buen punto de partida
- Artículo: *"Testing Trophy"* de Kent C. Dodds — la filosofía aplica al backend también
