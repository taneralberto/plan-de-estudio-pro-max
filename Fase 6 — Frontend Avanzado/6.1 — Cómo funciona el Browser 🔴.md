# 6.1 — Cómo funciona el Browser 🔴 [CORE]

- [ ] **Critical Rendering Path**

HTML parsing → DOM tree → CSSOM → Render tree → Layout → Paint → Composite. Por qué un CSS en el `<head>` bloquea el render (render-blocking resource). Por qué un `<script>` sin `defer` bloquea el parsing del HTML. Qué es el reflow (recalcular layout) y por qué es costoso. Qué es el repaint y cuándo se dispara. Cómo el browser decide qué va al GPU compositor thread (transforms, opacity) y qué permanece en el main thread.

**Por qué importa:** Cuando el editor de artículos de FreePress se siente lento al escribir, o cuando el listado de artículos tarda en aparecer, el problema casi siempre está en el Critical Rendering Path mal entendido. Agregar `async` o `defer` mal en un script puede romper el orden de ejecución del código. Poner el CSS en el `<body>` en lugar del `<head>` causa FOUC (Flash of Unstyled Content). Sin entender este modelo, optimizar performance de frontend es disparar al aire.

**Ejercicio:** Abrí Chrome DevTools → Performance → grabá la carga de FreePress. Identificá: cuánto tiempo tarda el First Contentful Paint (FCP), cuáles son los recursos que bloquean el render (aparecen en rojo), y si hay reflows costosos durante la interacción. Aplicá al menos una optimización (defer en scripts, preload en fonts, eliminar CSS bloqueante) y medí el FCP antes y después con Lighthouse.

**Done cuando:** Podés abrir el Performance panel de DevTools, grabar una interacción en FreePress, e identificar por qué una animación o scroll está "jankando" (frames por debajo de 60fps). Podés explicar por qué `transform: translateX()` es más performante que cambiar `left` en CSS y qué tiene que ver con el compositor thread.

- [web.dev — Critical Rendering Path](https://web.dev/learn/performance/understanding-the-critical-path) — la guía moderna de Google, más accesible que el artículo de Garsiel. Empezá acá.

- [ ] **Rendering: CSR vs SSR vs SSG vs ISR**

Client-Side Rendering (todo el HTML lo genera JavaScript en el browser — primera carga lenta, navegación rápida). Server-Side Rendering (el servidor genera el HTML por cada request — primera carga rápida, SEO natural, más carga en el servidor). Static Site Generation (HTML pre-generado en build time — el más rápido, pero los datos deben ser estáticos o tolerar staleness). Incremental Static Regeneration (SSG con revalidación: el HTML se regenera en background cuando envejece, sin rebuild completo). Cuándo usar cada uno y por qué Next.js, Nuxt y Analog los ofrecen todos.

**Por qué importa:** FreePress es un CMS — los artículos deben ser indexables por Google (SEO) y deben cargar rápido para el lector. CSR puro (una SPA Angular sin SSR) entrega HTML vacío al crawler de Google. Eso es un problema real: el contenido no se indexa bien, o se indexa mal. Entender las diferencias entre estos modos de rendering no es un detalle de framework — es una decisión arquitectónica que impacta directamente en SEO, performance y costo de infraestructura.

**Ejercicio:** Configurá Angular Universal (SSR) en el frontend de FreePress para las páginas públicas de lectura de artículos (no el panel de admin). Verificá el resultado abriendo el source HTML de la página (`Ctrl+U`) — el contenido del artículo debe aparecer en el HTML, no solo un `<app-root></app-root>` vacío. Medí el FCP con Lighthouse antes y después de habilitar SSR.

**Done cuando:** Podés analizar un nuevo proyecto (un e-commerce, un blog, un dashboard interno) y justificar con criterio técnico qué modo de rendering corresponde a cada sección. Podés explicar por qué una página de dashboard de analytics no necesita SSR (es privada, sin SEO, con datos en tiempo real) pero una página de producto de e-commerce sí (SEO crítico, primer load visible para el usuario).

- [ ] **Web APIs que no conocés (y deberías)**

Intersection Observer (detectar cuándo un elemento entra al viewport, sin scroll events), ResizeObserver (detectar cambios de tamaño de un elemento), MutationObserver (detectar cambios en el DOM), Performance API (medir tiempos reales en el browser con `performance.mark()` y `performance.measure()`), Web Workers (correr JavaScript en un thread separado sin bloquear el main thread), AbortController (cancelar fetch requests y otras operaciones async), Broadcast Channel (comunicación entre tabs del mismo origen).

**Por qué importa:** Muchos bugs de performance en frontend son causados por usar la API incorrecta para el trabajo. Un infinite scroll implementado con `window.scroll` event + `getBoundingClientRect()` ejecuta decenas de cálculos de layout por scroll. El mismo infinite scroll con IntersectionObserver ejecuta cero cálculos en scroll — solo se dispara cuando el elemento entra al viewport. Conocer estas APIs no es trivia — es la diferencia entre código que funciona y código que funciona bien bajo carga.

**Ejercicio:** Implementá tres features en FreePress usando las APIs correspondientes: (1) Infinite scroll en el listado de artículos con IntersectionObserver — cuando el último artículo visible entra al viewport, cargar más. (2) Un componente de imagen lazy-load con IntersectionObserver — la imagen no se descarga hasta que está cerca del viewport. (3) Cancelación de búsqueda en tiempo real con AbortController — si el usuario escribe rápido, cancelar el fetch anterior antes de hacer el nuevo.

**Done cuando:** Podés recibir un requerimiento de UI (lazy loading, infinite scroll, resize-aware layout, search con debounce y cancelación) e identificar qué Web API nativa resuelve el problema antes de buscar una librería. Podés explicar por qué un Web Worker no tiene acceso al DOM y cuándo eso es un problema vs cuándo no lo es.
