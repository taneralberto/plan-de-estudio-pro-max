# 6.2 — State Management 🔴

---

- [ ] **El problema del estado — De verdad**

Por qué existe state management, qué problemas concretos resuelve, y por qué la mayoría de apps NO necesitan Redux. Los cuatro tipos de estado que toda app maneja: local state (solo le importa a un componente — input de un form), shared state (varios componentes lo necesitan — el usuario autenticado), server state (datos que viven en el servidor y se sincronizan — los artículos de FreePress), URL state (lo que está en la URL — el filtro activo, la página actual). Cada tipo tiene herramientas distintas y mezclarlos es el origen de la mayoría de los bugs de estado.

**Por qué importa:** El error clásico es meter todo en un store global (NgRx, Redux) porque "así se hace state management". El resultado: boilerplate enorme, bugs difíciles de trazar, y estado del servidor duplicado en el cliente que queda desincronizado. En FreePress, los artículos publicados son server state — viven en Postgres, no en el browser. Ponerlos en NgRx significa sincronizarlos manualmente, gestionar loading/error states, y decidir cuándo invalidar el cache. TanStack Query hace todo eso por vos, con menos código y más correctamente.

**Ejercicio:** Auditá el state management actual de FreePress. Para cada pieza de estado que encuentres, clasificala en los cuatro tipos (local, shared, server, URL). Identificá cuál estado está siendo manejado con la herramienta incorrecta (por ejemplo: datos del servidor en un BehaviorSubject que se sincroniza manualmente). Propone la herramienta correcta para cada tipo.

**Done cuando:** Podés analizar un componente Angular con un store de NgRx y decidir si realmente necesita NgRx o si server state (TanStack Query) + signals locales sería más simple y correcto. Podés explicar por qué el estado de la URL es estado real — y por qué sincronizar filters/pagination con query params es mejor que tenerlos en un store.

- [ ] **Patrones de State Management**

Flux/Redux pattern (unidirectional data flow, actions, reducers, store — NgRx en Angular). Signals (Angular 17+, Solid, Preact — fine-grained reactivity sin el overhead de observables). Atoms (Jotai, Recoil — estado atómico y composable). Stores (Zustand, Pinia — stores minimalistas sin boilerplate). Proxy-based (MobX, Valtio — observabilidad automática de objetos mutables). TanStack Query para server state — cache automático, background refetching, optimistic updates, error/loading states out of the box.

**Por qué importa:** Conocer los patrones te permite elegir la herramienta correcta en lugar de usar siempre la misma. Un proyecto Angular grande con múltiples features complejas puede justificar NgRx. Un dashboard con datos del servidor justifica TanStack Query. Un formulario complejo justifica state local con signals. Usar NgRx para todo porque "es el estándar de Angular" es el equivalente a usar Redux para un contador — técnicamente funciona, pero el overhead es desproporcionado al problema.

**Ejercicio:** Implementá la feature de "artículos guardados" (bookmarks) en FreePress de dos formas: (1) con NgRx — actions, reducers, effects, selectors; (2) con Signals + TanStack Query — signal local para UI optimista, TanStack Query para sincronizar con el backend. Comparé el número de archivos, líneas de código, y la complejidad de manejar el estado de loading/error en cada approach. Documentá cuál elegirías y por qué.

**Done cuando:** Podés recibir una descripción de una feature nueva de FreePress y elegir el patrón de state management correcto con justificación técnica, sin defaultear a NgRx automáticamente. Podés explicar cuándo un BehaviorSubject de RxJS es suficiente, cuándo Signals es la respuesta, y cuándo NgRx agrega valor real.

- [ ] **Signals — El futuro del state management**

Qué son signals: un valor reactivo que notifica automáticamente a sus dependientes cuando cambia, sin subscriptions manuales ni detección de cambios costosa. Fine-grained reactivity: solo se re-renderiza exactamente el componente (o la parte del template) que depende del signal que cambió, no el árbol completo. Por qué Angular, Solid, Preact y Vue los adoptaron y están migrando away de Zone.js y Virtual DOM respectivamente. La diferencia con observables (signals son síncronos y siempre tienen un valor; observables son asíncronos y pueden estar vacíos) y con setState de React (signals actualiza solo los dependientes, setState puede re-renderizar el árbol entero).

**Por qué importa:** Angular históricamente usaba Zone.js para detectar cambios — un monkey-patch de todas las APIs asíncronas del browser que ejecuta change detection en el árbol completo después de cada evento. En una app grande, eso puede significar miles de checks por cada click. Signals elimina ese overhead: solo se actualiza lo que depende del signal que cambió. Si en FreePress un signal de `currentUser` cambia, solo los componentes que usan ese signal se actualizan — no toda la app. Es una mejora de performance estructural, no una optimización puntual.

**Ejercicio:** Migrá el estado del usuario autenticado en FreePress de un BehaviorSubject o un servicio con `currentUser$` observable a un Signal. Usá `computed()` para derivar el estado de "¿tiene permisos de editor?" a partir del signal del usuario. Habilitá `ChangeDetectionStrategy.OnPush` en los componentes que consumen ese signal y verificá con DevTools que solo se re-renderizan cuando el signal cambia, no en cada evento.

**Done cuando:** Podés explicar la diferencia entre `signal()`, `computed()` y `effect()` en Angular con un ejemplo concreto de cuándo usar cada uno. Podés migrar un componente de Zone.js + async pipe a Signals + `ChangeDetectionStrategy.OnPush` y medir la diferencia en change detection cycles en el Profiler de Angular DevTools.
