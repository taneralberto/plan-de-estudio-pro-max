# 6.4 — Testing Frontend 🟡

> ⚠️ **Este archivo es la continuación directa de `5.3 — Testing Backend y API`.**
> El marco conceptual — Testing Trophy, filosofía de Testing Library, qué testear y qué no — está en ese archivo y no se repite acá. Si no lo estudiaste, hacelo primero. Este archivo asume que ya sabés por qué testeamos comportamiento y no implementación.

---

- [ ] **Testing Trophy (no Testing Pyramid)**

Unit tests (testean una función o clase aislada — rápidos pero frágiles ante refactors de implementación), Integration tests (testean varios componentes o capas trabajando juntos — el punto dulce del frontend), E2E tests (testean el flujo completo en un browser real — lentos pero dan la mayor confianza). Por qué Kent C. Dodds propone la Testing Trophy en lugar de la Pyramid: en frontend, los integration tests son los más valiosos porque testean comportamiento del usuario, no detalles de implementación. La filosofía de Testing Library: "The more your tests resemble the way your software is used, the more confidence they can give you."

**Por qué importa:** Un test que verifica que el método `getTitle()` de un componente Angular retorna el string correcto no te dice si el usuario puede ver el título. Un test que renderiza el componente y verifica que hay un elemento con el texto del título te dice exactamente lo que le importa al usuario. La diferencia es que el primero pasa aunque el template esté roto, y el segundo falla si el usuario no puede ver el título — que es lo único que importa. Testear implementaciones en lugar de comportamiento genera una suite de tests que da falsa confianza.

**Ejercicio:** Escribí tests de integración para el componente de listado de artículos de FreePress usando Angular Testing Library (`@testing-library/angular`). Los tests deben verificar: (1) se renderizan los artículos recibidos como input; (2) cuando no hay artículos, se muestra un mensaje de "no hay artículos"; (3) al hacer click en un artículo, se navega a la URL correcta. Los tests NO deben acceder a propiedades internas del componente ni usar `fixture.componentInstance` directamente — solo interactuar como lo haría un usuario.

**Done cuando:** Podés leer una suite de tests de Angular y determinar si están testeando implementación o comportamiento — y reescribir los que testean implementación en términos de comportamiento del usuario. Podés explicar por qué un test que usa `component.articles = []` en lugar de mockear el servicio que los provee es frágil, y cuál es la forma correcta.

**Recursos:**
- [*"Testing Trophy"*](https://kentcdodds.com/blog/the-testing-trophy-and-testing-classifications) de Kent C. Dodds — el artículo que define el modelo. Leerlo antes de escribir el primer test.
- [Angular Testing Library](https://testing-library.com/docs/angular-testing-library/intro/) — la librería que implementa la filosofía de Testing Library para Angular. La documentación tiene ejemplos comparando "cómo no hacerlo" vs "cómo hacerlo".
- [*"Common Testing Mistakes"*](https://kentcdodds.com/blog/common-testing-mistakes) de Kent C. Dodds — los errores más frecuentes en testing frontend con sus correcciones. Leerlo después de escribir los primeros tests.

- [ ] **Visual Regression Testing**

Storybook como catálogo de componentes: cada componente documentado con sus variantes (estados, tamaños, temas) en forma aislada. Chromatic (la plataforma de Storybook) y Percy para detectar cambios visuales automáticamente en CI — comparan screenshots pixel a pixel entre PRs. Playwright `toHaveScreenshot()` para visual regression sin Storybook. Cuándo vale la pena el overhead: sistemas de diseño con muchos componentes compartidos y alta frecuencia de cambios visuales. Cuándo es overkill: apps pequeñas o con pocos componentes UI compartidos.

**Por qué importa:** El bug más frustrante en frontend es el que nadie detecta hasta que un usuario lo reporta: "el botón de publicar tiene el texto cortado desde la semana pasada". Sin visual regression testing, un cambio CSS inocente en un componente compartido puede romper la UI en 10 lugares distintos sin que ningún test lo detecte — porque los tests funcionales no ven píxeles. En FreePress, si el sistema de diseño (botones, cards, typography) cambia, el visual regression testing detecta exactamente qué componentes cambiaron visualmente antes de mergear.

**Ejercicio:** Configurá Storybook en el frontend de FreePress. Creá stories para al menos 3 componentes del sistema de diseño: el componente de `ArticleCard` (con variantes: con imagen / sin imagen, borrador / publicado), el componente de `Button` (variantes: primary / secondary / danger, sizes, disabled state), y el componente de `Badge` de estado del artículo. Luego configurá Playwright visual regression: ejecutá `playwright test --update-snapshots` para generar los baselines y verificá que el CI falla si modificás el color del botón primary.

**Done cuando:** Tu CI de FreePress falla automáticamente si un cambio CSS modifica visualmente un componente de Storybook que tiene visual regression configurado. Podés explicar cuándo aceptar un cambio visual detectado (es intencional) vs cuándo es un regression (es accidental) y cómo el workflow de revisión de Chromatic/Percy facilita esa decisión.

**Recursos:**
- [Storybook Docs](https://storybook.js.org/docs/angular/get-started/introduction) — guía oficial para Angular. El setup inicial tarda 5 minutos con `npx storybook@latest init`.
- [Playwright — Visual Comparisons](https://playwright.dev/docs/test-snapshots) — la documentación oficial de `toHaveScreenshot()`. La opción más simple para visual regression sin Chromatic.
- [Chromatic](https://www.chromatic.com/docs/) — la plataforma de Storybook para visual regression en CI. Tiene free tier generoso para proyectos open source.
