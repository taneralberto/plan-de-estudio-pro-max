# 7.1 — SQL y Modelado Relacional 🔴

---

- [ ] **Normalización y Desnormalización**

1NF (sin grupos repetidos, cada celda tiene un valor atómico), 2NF (sin dependencias parciales — cada columna depende de toda la PK, no solo parte), 3NF (sin dependencias transitivas — ninguna columna no-clave depende de otra columna no-clave), BCNF (la forma estricta de 3NF). Cuándo normalizar hasta 3NF (consistencia de datos, sin redundancia, escrituras confiables). Cuándo desnormalizar a propósito (reportes pesados, analytics, lecturas frecuentes que necesitan JOINs costosos). La tensión entre las dos no se resuelve con una regla fija — se resuelve entendiendo el patrón de acceso de la app.

**Por qué importa:** El esquema de FreePress evoluciona con el tiempo. Si el modelo está bien normalizado, agregar una nueva relación (ej: un artículo puede tener múltiples autores) es un ALTER TABLE limpio. Si está desnormalizado por conveniencia, esa misma evolución implica migrar datos y coordinar múltiples tablas. Pero un esquema sobre-normalizado puede generar queries con 6 JOINs para mostrar un listado simple. Conocer las formas normales no es saber teoría — es saber cuándo romperlas con criterio y cuándo no.

**Ejercicio:** Tomá el esquema actual de FreePress y analizá cada tabla: ¿está en 3NF? ¿Hay columnas que dependen de algo que no es la PK? Identificá al menos un caso de desnormalización que tendría sentido para performance (ej: una columna `article_view_count` desnormalizada en la tabla `articles` para evitar un COUNT costoso en cada listado) y uno que sería un error (ej: guardar `author_name` en `articles` además del `author_id`). Justificá cada decisión.

**Done cuando:** Dado un modelo de datos con anomalías (una tabla en 1NF con dependencias parciales), podés normalizarlo a 3NF paso a paso, explicando qué tabla nueva se crea en cada paso y por qué. Podés también tomar un modelo 3NF y proponer una desnormalización específica para un patrón de lectura particular, articulando qué consistencia sacrificás y cómo la compensás (ej: trigger, job periódico, actualización en la escritura).

**Recursos:**
- [*"Database Design for Mere Mortals"*](https://www.amazon.com/Database-Design-Mere-Mortals-Relational/dp/0136788041) de Michael Hernandez — el libro más accesible sobre modelado relacional. No asume conocimiento previo y tiene ejercicios prácticos.
- [PostgreSQL Docs — DDL](https://www.postgresql.org/docs/current/ddl.html) — la documentación oficial de PostgreSQL sobre tipos de datos, constraints, y modelado. La referencia más confiable para decisiones de schema.
- [*"SQL Antipatterns"*](https://www.amazon.com/SQL-Antipatterns-Programming-Pragmatic-Programmers/dp/1934356557) de Bill Karwin — 24 antipatrones comunes de modelado y queries SQL con sus soluciones. Ideal para leer después de entender normalización.

- [ ] **Transactions y ACID**

Atomicity (la transacción completa o no pasa nada — no hay estados intermedios), Consistency (la transacción lleva la base de un estado válido a otro estado válido — los constraints se verifican al final), Isolation (las transacciones concurrentes no se interfieren entre sí — con grados configurables), Durability (una vez committed, los datos sobreviven a crashes). Isolation levels: Read Uncommitted (lee datos sin commitear — dirty reads), Read Committed (solo lee datos commiteados — el default de PostgreSQL), Repeatable Read (garantiza que los mismos datos leídos dos veces en la misma transacción son iguales), Serializable (como si las transacciones corrieran en serie, una por una — el más seguro, el más caro). Fenómenos: dirty read, non-repeatable read, phantom read — cuál nivel previene cuál.

**Por qué importa:** FreePress tiene operaciones que deben ser atómicas: publicar un artículo actualiza el estado en `articles`, crea un registro en `article_versions`, e incrementa el contador de publicaciones del autor — las tres cosas deben pasar o ninguna. Sin una transacción, si el proceso muere después del primer UPDATE, el estado queda inconsistente. Y el isolation level incorrecto en un sistema con escrituras concurrentes genera race conditions que son increíblemente difíciles de reproducir y debuggear en producción.

**Ejercicio:** Creá en PostgreSQL un escenario real de race condition: dos usuarios simultáneos intentan aplicar un descuento al mismo artículo de pago (ej: precio actual → precio con descuento, pero solo si no fue descontado ya). Ejecutá las transacciones con `Read Committed` y demostrá que se produce el bug. Luego arreglalo con `SELECT FOR UPDATE` y con `Serializable`, y compará el comportamiento y el costo de cada solución.

**Done cuando:** Podés leer el código de una transacción de TypeORM y determinar qué isolation level necesita y si hay riesgo de race condition. Podés explicar por qué PostgreSQL usa Read Committed como default (balance entre concurrencia y seguridad para la mayoría de los casos) y en qué situaciones específicas de FreePress necesitarías subir a Repeatable Read o Serializable.

**Recursos:**
- [PostgreSQL Docs — Transactions](https://www.postgresql.org/docs/current/tutorial-transactions.html) — la documentación oficial de PostgreSQL sobre ACID e isolation levels. La fuente más confiable.
- [*"Designing Data-Intensive Applications"*](https://www.amazon.com/Designing-Data-Intensive-Applications-Reliable-Maintainable/dp/1449373321) de Martin Kleppmann — el capítulo 7 (Transactions) es el mejor tratamiento de ACID e isolation levels que existe. Nivel de profundidad sin igual.
- [*"SQL Antipatterns"*](https://www.amazon.com/SQL-Antipatterns-Programming-Pragmatic-Programmers/dp/1934356557) de Bill Karwin — el capítulo sobre concurrencia tiene ejemplos concretos de race conditions con SQL.

- [ ] **SQL avanzado**

Window functions: `ROW_NUMBER()` (número de fila dentro de una partición), `RANK()` y `DENSE_RANK()` (ranking con y sin gaps), `LAG()` y `LEAD()` (acceder a la fila anterior o siguiente dentro de la partición), `SUM() OVER()` (agregados acumulativos). CTEs (Common Table Expressions): queries nombradas reutilizables dentro de una query más grande — hacen legible lo que de otra forma sería un subquery anidado ilegible. Recursive CTEs para estructuras jerárquicas (ej: categorías con subcategorías). EXPLAIN y EXPLAIN ANALYZE: leer el query plan, entender Seq Scan vs Index Scan vs Bitmap Heap Scan, identificar el nodo más costoso. Materialized views: pre-calcular y almacenar el resultado de una query costosa para leerlo rápido.

**Por qué importa:** El 80% de los problemas de performance de base de datos en producción son queries SQL mal escritas o queries que no usan índices. Un developer que solo conoce `SELECT`, `JOIN` y `WHERE` va a escribir queries que funcionan en desarrollo con 100 registros y explotan en producción con 1 millón. Window functions eliminan subqueries correlacionados costosos. EXPLAIN ANALYZE te muestra exactamente dónde la base de datos gasta tiempo. Materialized views te permiten responder preguntas analíticas en milisegundos que de otra forma tomarían segundos.

**Ejercicio:** Escribí tres queries para FreePress usando SQL avanzado: (1) Con window function — rankear los artículos más leídos por semana, usando `ROW_NUMBER() OVER (PARTITION BY week ORDER BY views DESC)`. (2) Con CTE recursivo — obtener el árbol completo de categorías de FreePress (categoría → subcategoría → subsubcategoría) en una sola query. (3) Con EXPLAIN ANALYZE — encontrá la query más lenta de FreePress (la que hace Seq Scan en tablas grandes), agregá el índice faltante, y mostrá la mejora en el query plan antes y después.

**Done cuando:** Podés leer un EXPLAIN ANALYZE de PostgreSQL e identificar el nodo más costoso, si se está usando un índice o no, y proponer la optimización correcta (índice nuevo, reescribir la query, materializar el resultado). Podés reemplazar cualquier subquery correlacionado con una window function equivalente y explicar por qué la segunda es más eficiente.

**Recursos:**

- [PostgreSQL Docs — Window Functions](https://www.postgresql.org/docs/current/tutorial-window.html) — la referencia oficial con ejemplos claros de cada función.
