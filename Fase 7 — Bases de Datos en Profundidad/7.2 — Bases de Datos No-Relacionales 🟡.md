# 7.2 — Bases de Datos No-Relacionales 🟡 [ADVANCED]

- [ ] **Cuándo NO usar SQL**

Las familias de bases de datos no-relacionales y el problema específico que cada una resuelve: Document stores (MongoDB, CouchDB — datos semiestructurados o con schema variable, como perfiles de usuario con campos opcionales), Key-Value (Redis, DynamoDB — acceso por clave con latencia sub-milisegundo, caching, sessions), Wide-column (Cassandra, ScyllaDB — escrituras masivas distribuidas, series temporales, alta disponibilidad geográfica), Graph (Neo4j, Amazon Neptune — relaciones complejas y traversals, redes sociales, detección de fraude), Time-series (InfluxDB, TimescaleDB — métricas y eventos con componente temporal), Search (Elasticsearch, Typesense — búsqueda full-text, faceting, relevance ranking).

**Regla fundamental:** Usá PostgreSQL por default. Solo usá otra cosa cuando PostgreSQL NO pueda resolver tu problema eficientemente — y "no pueda" significa benchmarks reales, no intuición o "escalará mejor".

**Por qué importa:** La decisión de qué base de datos usar es una de las más difíciles de revertir en un sistema en producción. Migrar de PostgreSQL a MongoDB una vez que tenés datos en producción es meses de trabajo. La mayoría de los proyectos que usan MongoDB "porque es más flexible" terminan reimplementando manualmente las garantías que PostgreSQL da gratis (transacciones, foreign keys, joins). Conocer cuándo cada familia tiene sentido real te evita acumular deuda técnica por decisiones de base de datos tomadas sin criterio.

**Ejercicio:** Para FreePress, identificá qué datos podrían tener sentido en una base de datos no-relacional y cuáles definitivamente no. Específicamente: (1) ¿Los metadatos flexibles de artículos (campos custom que cada autor puede definir) justifican un document store o PostgreSQL JSONB resuelve el problema igual de bien? (2) ¿El sistema de búsqueda full-text de artículos justifica Elasticsearch o PostgreSQL FTS (`tsvector`) es suficiente para el volumen esperado? Para cada caso, documentá el criterio de decisión con datos concretos (volumen estimado, tipo de queries, latencia requerida).

**Done cuando:** Podés recibir un requerimiento de un sistema nuevo (ej: "necesitamos guardar las métricas de visualización por artículo en tiempo real") y determinar si PostgreSQL alcanza o si necesitás una base de datos especializada — con argumentos técnicos basados en el volumen, el patrón de acceso y los requisitos de latencia, no en preferencias personales.

**Recursos:**
- [*"Designing Data-Intensive Applications"*](https://www.amazon.com/Designing-Data-Intensive-Applications-Reliable-Maintainable/dp/1449373321) de Martin Kleppmann — los capítulos 2 y 3 comparan modelos de datos (relacional, documento, grafo) y motores de storage con precisión técnica sin igual. La referencia para esta decisión.
- [PostgreSQL Docs — JSON Types](https://www.postgresql.org/docs/current/datatype-json.html) — JSONB en PostgreSQL: cómo indexar campos JSON, cuándo es suficiente y cuándo no. Leerlo antes de decidir que "necesitás MongoDB".
- [*"Seven Databases in Seven Weeks"*](https://www.amazon.com/Seven-Databases-Weeks-Modern-Movement/dp/1680502530) de Luc Perkins — tour práctico por Redis, PostgreSQL, MongoDB, Neo4j, HBase, DynamoDB y Elasticsearch. Ideal para calibrar cuándo cada uno tiene sentido real.

- [ ] **Redis — Más que un cache**

Redis es una base de datos en memoria con persistencia opcional — no es solo un cache. Estructuras de datos que ofrece: Strings (cache simple, contadores), Lists (colas FIFO/LIFO), Sets (colecciones sin duplicados, intersecciones), Sorted Sets (leaderboards, cola con prioridad), Hashes (objetos), Streams (event log append-only, similar a Kafka en escala pequeña). Casos de uso más allá del cache: Pub/Sub para mensajería ligera entre servicios, rate limiting con sliding window usando Sorted Sets, session storage, distributed locks con `SET NX PX`, job queues con BullMQ (construido sobre Redis).

**Por qué importa:** FreePress ya usa Redis — y probablemente solo como cache de queries. Eso es dejar el 90% del valor de Redis sobre la mesa. El rate limiting de la API pública de FreePress, el sistema de notificaciones en tiempo real, la cola de jobs para procesamiento de imágenes, y las sesiones de usuarios autenticados son todos casos de uso que Redis resuelve de forma nativa, sin agregar otra dependencia de infraestructura.

**Ejercicio:** Implementá dos features en FreePress usando Redis más allá del cache: (1) Rate limiting con sliding window — la API pública de FreePress no puede recibir más de 100 requests por IP por minuto. Usá un Sorted Set de Redis donde el score es el timestamp del request. (2) Job queue con BullMQ — cuando se publica un artículo, encolar un job para generar el thumbnail de la imagen de portada. El job debe reintentarse automáticamente si falla, con backoff exponencial.

**Done cuando:** Podés mirar un requerimiento nuevo de FreePress (notificaciones push, búsqueda de usuarios online, cooldown entre publicaciones) y determinar qué estructura de datos de Redis resuelve el problema y cómo. Podés explicar cuándo Redis Streams es una alternativa válida a RabbitMQ/Kafka para mensajería y cuándo no escala lo suficiente.

**Recursos:**
- [Redis Docs — Data Structures](https://redis.io/docs/data-types/) — la documentación oficial con ejemplos de cada tipo de dato. El punto de partida para el ejercicio.
- [BullMQ Docs](https://docs.bullmq.io/) — la librería de job queues más completa para Node.js construida sobre Redis. Tiene ejemplos de retry, backoff, y concurrencia.
- [*"Redis in Action"*](https://www.amazon.com/Redis-Action-Josiah-L-Carlson/dp/1617290858) de Josiah Carlson — el libro de referencia. Cada capítulo es un caso de uso real construido con Redis (analytics, social features, ad targeting). Muy práctico.
