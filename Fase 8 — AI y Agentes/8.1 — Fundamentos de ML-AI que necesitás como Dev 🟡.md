# 8.1 — Fundamentos de ML/AI que necesitás como Dev 🟡 [ADVANCED]

- [ ] **Cómo funcionan los LLMs — Sin ser ML Engineer**

Transformers y el mecanismo de atención (cómo el modelo aprende qué palabras son relevantes para predecir la siguiente). Tokenización: los LLMs no ven palabras sino tokens (fragmentos de palabras) — un token promedia 4 caracteres en inglés, más en español. Context window: el límite de tokens que el modelo puede "ver" a la vez — todo lo que está fuera de la ventana no existe para el modelo. Temperatura y top-p: cómo controlar la aleatoriedad de las respuestas. Por qué los LLMs "alucinan": el modelo predice el token más probable dado el contexto — no tiene acceso a hechos, solo a patrones estadísticos aprendidos. Embeddings: representaciones numéricas de texto donde la similitud semántica se traduce en proximidad en el espacio vectorial.

**Por qué importa:** Si integrás AI en FreePress sin entender estos conceptos, vas a tomar decisiones incorrectas constantemente: usar temperatura alta cuando necesitás respuestas deterministas (moderación de contenido), no entender por qué el modelo "olvidó" algo que le dijiste hace tres mensajes (context window lleno), o confiar en respuestas factuales de un modelo sin verificación (alucinación). Estos fundamentos no son trivia académica — son el modelo mental que te permite usar LLMs de forma efectiva y predecir cuándo van a fallar.

**Ejercicio:** Experimentá con los parámetros del modelo directamente: usando la API de Claude, hacé la misma pregunta factual sobre FreePress (ej: "¿qué features tiene FreePress?") con temperatura 0, 0.5 y 1.0. Observá la variación en las respuestas. Luego probá llenar el context window con artículos de FreePress hasta el límite y verificá que el modelo "olvida" los primeros. Documentá en qué temperatura usarías para: moderación de contenido, sugerencias creativas de títulos, y generación de resúmenes.

**Done cuando:** Podés explicar, sin consultar documentación, por qué un LLM puede dar una respuesta diferente a la misma pregunta en dos llamadas consecutivas, qué parámetro controla eso, y cuándo querés que sea determinista vs creativo. Podés calcular aproximadamente cuántos artículos de FreePress entran en el context window de Claude 3.5 Sonnet dado su tamaño promedio.

- [ ] **Embeddings y Vector Databases**

Qué es un embedding: una transformación de texto (o imagen, o audio) en un vector de números flotantes de alta dimensión donde la distancia entre vectores refleja similitud semántica. "Rey" y "Reina" están cerca; "Rey" y "hamburguesa" están lejos. Cosine similarity como métrica de similitud. Modelos de embedding: `text-embedding-3-small` de OpenAI, `voyage-3` de Voyage AI (el recomendado para uso con Claude). Vector databases: Pinecone (managed), Weaviate (open source), Chroma (local, para desarrollo), pgvector (extensión de PostgreSQL — la opción más simple si ya usás Postgres). Cuándo embeddings superan a keyword search: búsquedas semánticas ("artículos sobre sostenibilidad" encuentra también artículos sobre "medio ambiente" y "ecología"), y cuándo keyword search sigue siendo mejor (búsqueda exacta de términos técnicos, nombres propios, IDs).

**Por qué importa:** FreePress tiene un motor de búsqueda. Si está implementado con `ILIKE '%término%'` o incluso con PostgreSQL FTS, no puede encontrar un artículo sobre "cómo reducir las emisiones de carbono" cuando el usuario busca "sustentabilidad ambiental". Los embeddings hacen que esa búsqueda semántica sea posible. Pero embeddings tienen costo (llamada a la API por cada chunk indexado) y complejidad operacional (una base de datos más). Saber cuándo el tradeoff vale es el conocimiento que importa.

**Ejercicio:** Indexá los artículos de FreePress con embeddings usando pgvector (para no agregar otra base de datos). Usá `voyage-3` o `text-embedding-3-small` para generar los embeddings de cada artículo. Implementá un endpoint `/api/search/semantic?q=` que convierta la query del usuario en un embedding y busque los artículos más similares usando `cosine_distance`. Comparé los resultados con la búsqueda por keywords para 5 queries donde la semántica importa.

**Done cuando:** Podés implementar búsqueda semántica end-to-end: chunking del texto, generación de embeddings, almacenamiento en pgvector, y búsqueda por similitud. Podés explicar por qué la búsqueda semántica puede dar resultados peores que keyword search para queries específicas (nombres propios, términos técnicos exactos) y cómo el hybrid search (keyword + semántico con reranking) mitiga ese problema.

**Recursos:**
- [pgvector GitHub](https://github.com/pgvector/pgvector) — la extensión de PostgreSQL para vectores. La opción más pragmática si ya usás Postgres. README muy claro con ejemplos de SQL.
- [Voyage AI Docs](https://docs.voyageai.com/) — los modelos de embedding recomendados por Anthropic para uso con Claude. Mejor performance que OpenAI embeddings para tareas de retrieval.
- [*"What Are Embeddings?"*](https://vickiboykis.com/what_are_embeddings/) de Vicki Boykis — la explicación más accesible y rigurosa de embeddings sin asumir background de ML. Gratuito online.
