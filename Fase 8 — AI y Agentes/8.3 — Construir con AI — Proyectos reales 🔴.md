- [ ] **MCP (Model Context Protocol) — En profundidad**

MCP no es solo "qué es" — es un protocolo estándar abierto de Anthropic para conectar modelos de AI con herramientas y fuentes de datos externas. Arquitectura: MCP hosts (el cliente AI, ej: Claude Desktop, tu app), MCP servers (los servicios que exponen capabilities), transporte (stdio para local, HTTP/SSE para remoto). Los tres tipos de capabilities que un MCP server puede exponer: Resources (datos que el modelo puede leer — artículos, archivos, resultados de queries), Tools (funciones que el modelo puede llamar — buscar, crear, modificar), Prompts (templates predefinidos reutilizables). Cómo construir un MCP server real en TypeScript usando el SDK oficial.

**Por qué importa:** MCP resuelve el problema de integración entre AI y tus sistemas existentes de forma estandarizada. Sin MCP, cada integración es ad-hoc: un tool use custom, un formato de respuesta inventado, documentación propia. Con MCP, cualquier cliente compatible (Claude Desktop, tu app, herramientas de terceros) puede usar tu MCP server sin modificaciones. Para FreePress, un MCP server significa que Claude Desktop puede buscar artículos, crear borradores, y gestionar publicaciones — sin que vos tengas que construir una UI especial para cada cliente.

**Ejercicio:** Construí un MCP server completo para FreePress en TypeScript usando el SDK oficial de MCP. Exponé al menos: (1) Resource: `articles://list` que devuelve los artículos recientes con su metadata; (2) Tool: `search_articles(query: string)` que busca artículos por contenido; (3) Tool: `create_draft(title: string, content: string, tags: string[])` que crea un borrador en FreePress; (4) Tool: `publish_article(id: string)` que publica un borrador existente. Conectalo a Claude Desktop y verificá que podés usar lenguaje natural para operar FreePress desde Claude.

**Done cuando:** Tenés un MCP server de FreePress corriendo que Claude Desktop puede usar para buscar y crear artículos con lenguaje natural. Podés explicar la diferencia entre un Resource y un Tool en MCP, cuándo exponer algo como uno u otro, y cómo el protocolo de transporte afecta el deployment (stdio solo funciona localmente, HTTP/SSE es necesario para acceso remoto).

**Recursos:**
- [MCP Docs Oficiales](https://modelcontextprotocol.io/introduction) — la documentación completa del protocolo. La guía de "Building MCP Servers" en TypeScript es el punto de partida para el ejercicio.
- [MCP TypeScript SDK](https://github.com/modelcontextprotocol/typescript-sdk) — el SDK oficial en TypeScript. El README tiene un ejemplo mínimo funcional que podés adaptar para FreePress.
- [Anthropic — MCP Announcement](https://www.anthropic.com/news/model-context-protocol) — el artículo original de Anthropic explicando el problema que MCP resuelve y por qué lo hicieron estándar abierto. Contexto valioso para entender la motivación del protocolo.

- [ ] **RAG (Retrieval-Augmented Generation) — Implementación real**

El problema que resuelve RAG: los LLMs tienen conocimiento fijo hasta su fecha de training y no saben nada de tu base de datos privada. RAG conecta el modelo con tu información en tiempo de inferencia. El pipeline completo: ingest (obtener los documentos), chunk (dividir en fragmentos manejables), embed (convertir cada chunk en un vector), store (guardar en vector database), retrieve (buscar los chunks más relevantes para la query del usuario), augment (agregar los chunks al prompt como contexto), generate (el modelo responde usando ese contexto). Chunking strategies: por tamaño fijo (simple pero puede cortar oraciones), por párrafo (respeta la semántica), por sección (si el documento tiene estructura), con overlap (para no perder contexto entre chunks). Reranking: usar un modelo especializado para reordenar los resultados de retrieval por relevancia real, no solo por similitud vectorial. Hybrid search: combinar keyword search (exacto, bueno para términos técnicos) con semantic search (flexible, bueno para consultas en lenguaje natural).

**Por qué importa:** FreePress tiene cientos de artículos publicados. Un asistente que puede responder preguntas sobre ese contenido ("¿hay artículos sobre TypeScript avanzado publicados este mes?", "resumime los tres artículos más populares sobre Docker") tiene valor real para los lectores y editores. Sin RAG, el modelo inventa respuestas que suenan plausibles pero son incorrectas. Con RAG, el modelo responde basándose en el contenido real de FreePress — y podés verificar las fuentes.

**Ejercicio:** Implementá un asistente RAG completo para FreePress con estos pasos: (1) Pipeline de ingest: cuando se publica un artículo en FreePress, hacer chunk por párrafo con overlap de 20%, generar embeddings con Voyage AI, y guardar en pgvector. (2) Endpoint de consulta: `/api/assistant?q=` que recupera los 5 chunks más relevantes, los reordena con un reranker (Cohere Rerank o Voyage Rerank), y genera una respuesta con Claude citando las fuentes. (3) Verificá que el asistente responde correctamente a 5 preguntas sobre artículos reales de FreePress y rechaza responder preguntas sobre temas no cubiertos en el contenido.

**Done cuando:** El asistente de FreePress puede responder preguntas sobre el contenido publicado citando los artículos fuente, y explícitamente dice "no tengo información sobre eso en FreePress" cuando la pregunta no tiene respuesta en el corpus. Podés modificar la estrategia de chunking y medir si el retrieval mejora o empeora para un conjunto de queries de prueba.

**Recursos:**
- [*"Building RAG-based LLM Applications for Production"*](https://www.anyscale.com/blog/a-comprehensive-guide-for-building-rag-based-llm-applications-part-1) de Anyscale — la guía más completa de RAG en producción, con benchmarks de distintas estrategias de chunking y retrieval.
- [LangChain Docs — RAG](https://python.langchain.com/docs/tutorials/rag/) / [LlamaIndex Docs](https://docs.llamaindex.ai/en/stable/) — los dos frameworks más usados para construir pipelines RAG. Útiles para ver cómo se estructura el problema, aunque podés implementarlo sin ellos.
- [Cohere Rerank API](https://docs.cohere.com/docs/rerank-2) — el reranker más accesible para proyectos pequeños. Tiene free tier y mejora significativamente la calidad del retrieval sobre similitud cosine pura.
- [*"RAG vs Fine-tuning"*](https://www.anyscale.com/blog/fine-tuning-vs-retrieval-augmented-generation) — cuándo RAG es la respuesta correcta y cuándo fine-tuning tiene más sentido. Fundamental para tomar la decisión correcta en FreePress.
