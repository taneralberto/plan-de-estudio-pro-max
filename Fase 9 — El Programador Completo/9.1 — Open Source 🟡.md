# 9.1 — Open Source 🟡 [ADVANCED]

- [ ] **Contribuir a proyectos open source**

No es solo "arreglar un typo en el README". Es el proceso completo de integrarse a una comunidad técnica: leer el contributing guide antes de tocar código, entender la arquitectura del proyecto leyendo issues y PRs anteriores, abrir un issue bien escrito antes de hacer una PR (describir el problema, el contexto, los pasos para reproducirlo, el comportamiento esperado), hacer PRs que los maintainers quieran mergear (scope acotado, tests incluidos, descripción que explica el por qué). Etiqueta de comunicación asíncrona en comunidades distribuidas: cómo dar feedback constructivo en code review, cómo responder al feedback sin ponerse defensivo, cómo seguir contribuyendo si una PR es rechazada.

**Por qué importa:** El portfolio de contribuciones open source es el CV técnico más honesto que existe — es código real, revisado por otros developers reales, en condiciones reales. Un solo PR bien hecho en un proyecto relevante (Angular, NestJS, TypeORM) dice más sobre tu nivel técnico que diez proyectos personales. Además, contribuir te expone a bases de código a escala que nunca vas a ver en proyectos personales: cómo organiza su testing suite Angular, cómo gestiona breaking changes NestJS, cómo documenta su API TypeORM.

**Ejercicio:** Elegí una de las dependencias principales de FreePress (TypeORM, NestJS, o Angular). Explorá sus issues abiertos etiquetados `good-first-issue` o `help wanted`. Elegí uno que puedas resolver — no el más fácil, sino el que más entendés. Antes de abrir la PR, abrí un issue o comentá en el existente diciendo que vas a trabajar en él. Hacé la PR con tests incluidos y descripción clara del enfoque. Documentá el proceso: cuánto tiempo te tomó entender el codebase, qué aprendiste del proceso de review.

**Done cuando:** Tenés al menos una PR mergeada (o en proceso de revisión activo) en un proyecto open source que no es tuyo. Podés explicar cómo funciona el flujo de contribución del proyecto (cómo correr los tests, cómo hacer el setup del entorno de desarrollo, cuáles son las convenciones de código) sin consultar la documentación.

**Recursos:**
- [firstcontributions.github.io](https://firstcontributions.github.io/) — guía paso a paso para la primera contribución, con ejemplos prácticos de fork + PR.
- [firsttimersonly.com](https://www.firsttimersonly.com/) — agrega issues de múltiples repos etiquetados específicamente para primera contribución.
- [*"How to Contribute to Open Source"*](https://opensource.guide/how-to-contribute/) de GitHub — la guía más completa sobre contribución open source, incluyendo cómo evaluar si un proyecto es saludable antes de invertir tiempo.

- [ ] **Crear y mantener tus propias librerías**

Semantic versioning (MAJOR.MINOR.PATCH — qué tipo de cambio justifica cada número, y por qué importa para tus usuarios). CHANGELOG (documentar qué cambió en cada versión — Added, Changed, Deprecated, Removed, Fixed, Security). API design para librerías: cómo diseñar una API pública que sea fácil de usar correctamente y difícil de usar incorrectamente. Backward compatibility: cómo evolucionar la API sin romper a los usuarios existentes (deprecation warnings, migration guides). npm publishing: configuración del `package.json` para publicación, `.npmignore`, peer dependencies, provenance. GitHub Actions para CI/CD del paquete: correr tests, buildear, y publicar automáticamente en cada tag.

**Por qué importa:** FreePress va a acumular utilities, helpers y componentes que podrían ser reutilizables en otros proyectos. Saber cómo extraer eso en una librería publicable — con versionado correcto, documentación, y CI — es la diferencia entre código que vive solo en FreePress y código que beneficia a otros y te da visibilidad en la comunidad. Además, diseñar una API pública te obliga a pensar desde la perspectiva del usuario de tu código, no del autor — esa habilidad mejora el código que escribís en todos los contextos.

**Ejercicio:** Extraé alguna utility de FreePress que sea reutilizable (ej: el sistema de structured logging con pino, el middleware de correlation IDs, el rate limiter de Redis) y publicala como un paquete npm bajo tu scope (`@tu-usuario/nombre-paquete`). Incluí: TypeScript types, README con ejemplos de uso, CHANGELOG con la versión inicial, y un GitHub Action que publique automáticamente en npm cuando se pushea un tag semver. Documentá las decisiones de API design que tomaste.

**Done cuando:** Tenés un paquete npm publicado bajo tu scope que otro developer podría instalar con `npm install @tu-usuario/paquete` y usar inmediatamente con los tipos correctos en TypeScript. El CHANGELOG documenta qué hay en la v1.0.0. El CI publica automáticamente cuando creás un tag.

**Recursos:**
- [*"Keep a Changelog"*](https://keepachangelog.com/) — el estándar de facto para CHANGELOGs. Una página, directa al punto.
- [*"Semantic Versioning 2.0.0"*](https://semver.org/) — la especificación completa de semver. Leerla una vez, usarla siempre.
- [npm Docs — Contributing packages](https://docs.npmjs.com/packages-and-modules/contributing-packages-to-the-registry) — guía oficial de npm para publicar paquetes con scope y provenance.
- [*"Designing Good APIs"*](https://www.infoq.com/articles/API-Design-Joshua-Bloch/) de Joshua Bloch — charla/artículo clásico sobre principios de diseño de APIs que minimicen el error del usuario.
