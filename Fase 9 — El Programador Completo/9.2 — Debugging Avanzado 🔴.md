# 9.2 — Debugging Avanzado 🔴 [CORE]

> ⚠️ **Práctica continua — no esperés a llegar a la Fase 9.** Empezá a aplicar el método científico y estas herramientas desde el día que abrís FreePress por primera vez.

- [ ] **Debugging sistemático — El método**

La metodología de debugging (reproducir, reducir, hipótesis, testear, repetir) está explicada en profundidad en `2.0 — Debugging Básico`. Acá nos enfocamos en herramientas avanzadas.

> **Si todavía usás "probar cosas al azar", volvé a 2.0 primero.**

**Recursos adicionales:**
- [Git Docs — bisect](https://git-scm.com/docs/git-bisect) — binary search en el historial para encontrar el commit exacto que introdujo un bug en O(log n) pasos
- [*"How to Report Bugs Effectively"*](https://www.chiark.greenend.org.uk/~sgtatham/bugs.html) de Simon Tatham — cómo escribir reportes de bugs que permiten resolverlos

- [ ] **Herramientas de debugging**

Node.js debugger con VS Code y Chrome DevTools (breakpoints condicionales, watch expressions, call stack, REPL en el contexto del breakpoint — infinitamente más útil que `console.log`). `strace` en Linux: intercepta y registra las syscalls de un proceso en tiempo real — cuando no sabés por qué un proceso se cuelga o falla sin output visible, `strace -p <pid>` te muestra qué está haciendo en el kernel. Wireshark y `tcpdump` para analizar tráfico de red: cuando el bug está en la comunicación entre dos servicios y los logs de ambos lados no muestran el problema. Heap snapshots y memory profiling en Node.js: identificar memory leaks tomando dos snapshots en momentos distintos y comparando qué objetos crecieron. Chrome Performance Recorder para CPU profiling del frontend.

**Por qué importa:** El 90% de los bugs de FreePress en producción se pueden resolver con logs y métricas. El otro 10% — los bugs intermitentes, los memory leaks graduales, los problemas de red entre servicios — requieren herramientas más potentes. No conocerlas no significa que esos bugs no existan: significa que tardás 10 horas en resolver algo que con la herramienta correcta tomaría 30 minutos. `strace` y Wireshark no son herramientas de sysadmin — son herramientas de cualquier developer que trabaja en sistemas reales.

**Ejercicio:** Hacé tres ejercicios de debugging con herramientas reales en FreePress: (1) Configurá el Node.js debugger en VS Code para el backend de NestJS, poné un breakpoint condicional que solo se activa cuando el `userId` de un request es específico, y explorá el call stack completo; (2) Simulá un memory leak en FreePress (ej: guardando objects en un array global que nunca se limpia) y detectalo tomando dos heap snapshots con 30 segundos de diferencia en Chrome DevTools; (3) Usá `tcpdump` o Wireshark para capturar el tráfico HTTP entre el frontend y el backend de FreePress y verificá que los headers de autenticación se envían correctamente.

**Done cuando:** Cuando FreePress tiene un memory leak en producción (la memoria del proceso NestJS crece gradualmente sin bajar), podés identificar qué tipo de objeto está acumulándose usando heap snapshots sin necesidad de reiniciar el proceso. Podés configurar el debugger de VS Code para adjuntarse a un proceso Node.js ya corriendo — no solo iniciar uno nuevo en modo debug.

**Recursos:**
- [Node.js Docs — Debugging Guide](https://nodejs.org/en/learn/getting-started/debugging) — la guía oficial para debuggear Node.js con VS Code y Chrome DevTools. El punto de partida para el ejercicio.
- [Chrome DevTools — Memory](https://developer.chrome.com/docs/devtools/memory-problems/) — la guía oficial para detectar memory leaks con heap snapshots. Incluye el flujo completo de toma y comparación de snapshots.
- [*"strace Wow Much Syscall"*](https://www.brendangregg.com/blog/2014-05-11/strace-wow-much-syscall.html) de Brendan Gregg — introducción práctica a `strace` con ejemplos reales. El autor también tiene el libro definitivo de performance en Linux (*Systems Performance*).
