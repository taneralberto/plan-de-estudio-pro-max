# 9.3 — Mantenibilidad y Deuda Técnica 🟡

---

- [ ] **Refactoring — El arte de mejorar código sin cambiar comportamiento**

Cuándo refactorizar: antes de agregar una feature (el código actual no te deja extenderlo limpiamente), durante code review (cuando ves que el código nuevo es correcto pero difícil de leer), y nunca como proyecto separado de semanas ("el gran refactor" que nunca termina). El prerequisito ineludible: tests que confirmen que el comportamiento no cambió. Técnicas concretas: Extract Function (cuando una función hace demasiado, extraé parte en otra con nombre descriptivo), Replace Conditional with Polymorphism (cuando tenés `if type === 'A' / else if type === 'B'` que crece — polimorfismo lo resuelve abierto/cerrado), Introduce Parameter Object (cuando una función recibe 6 parámetros — agruparlos en un objeto con nombre), Rename Variable (el refactor más frecuente y más subestimado — un nombre bueno elimina la necesidad de un comentario).

**Por qué importa:** FreePress va a crecer. El código que escribís hoy lo vas a leer en 6 meses sin contexto, cuando tenés que agregar una feature que no estaba prevista. La deuda técnica no es código malo — es código que fue correcto para el contexto en que se escribió pero que ahora frena el avance. Refactorizar continuamente en pequeños pasos (el modelo del Boy Scout: "dejá el código mejor de como lo encontraste") es la única forma de mantener la velocidad de desarrollo a largo plazo. El "gran refactor" no funciona porque nunca hay tiempo para terminarlo y el codebase sigue creciendo mientras tanto.

**Ejercicio:** Elegí el módulo más complejo de FreePress que más te cuesta modificar — ese que cada vez que tenés que tocarlo abrís el archivo con miedo. Primero escribí characterization tests (tests que documentan el comportamiento actual, correcto o no). Luego aplicá al menos tres técnicas de refactoring de Fowler sobre él, en pequeños commits con el test suite verde en cada uno. Al final, pedí a alguien (o a un LLM) que lea el código antes y después y evalúe la legibilidad.

**Done cuando:** Podés abrir cualquier módulo de FreePress y aplicar al menos una técnica de refactoring de Fowler sin romper los tests existentes — y sin tardar más de 30 minutos en identificar qué técnica aplica. Podés explicar por qué "el gran refactor" es una antipattern organizacional, no solo técnico.

**Recursos:**
- [*"Refactoring"*](https://www.amazon.com/Refactoring-Improving-Existing-Addison-Wesley-Signature/dp/0134757599) de Martin Fowler — LA referencia absoluta. Segunda edición usa JavaScript. El catálogo de técnicas (capítulos 6-12) es el que más vas a consultar.
- [refactoring.guru](https://refactoring.guru/refactoring) — versión online y gratuita del catálogo de refactorings con ejemplos visuales. Útil como referencia rápida mientras trabajás.
- [*"Clean Code"*](https://www.amazon.com/Clean-Code-Handbook-Software-Craftsmanship/dp/0132350882) de Robert Martin — complemento de Refactoring, más enfocado en principios de escritura que en técnicas de transformación. Leer con espíritu crítico — algunos ejemplos del libro son debatibles.

- [ ] **Legacy Code — Cómo trabajar con código que da miedo**

Qué es legacy code según Michael Feathers: código sin tests — no código viejo. El problema no es la antigüedad sino la imposibilidad de modificarlo con confianza. Characterization tests: cuando no hay tests, antes de tocar nada escribís tests que documentan el comportamiento actual del código (correcto o no) para tener una red de seguridad. Seams: un punto donde podés cambiar el comportamiento del código sin modificarlo — clave para agregar tests a código no diseñado para serlo. Strangler Fig aplicado a código: envolver gradualmente la funcionalidad legacy con código nuevo hasta poder eliminar el viejo, en lugar de reescribir todo de una vez.

**Por qué importa:** FreePress es hoy un proyecto nuevo. Pero en 2 años, las partes que escribiste al principio — sin experiencia en DDD, sin tests, con las decisiones de arquitectura que después aprendiste que eran incorrectas — van a ser tu legacy code. Y en cualquier trabajo, el primer día vas a heredar un codebase que nadie entiende completamente. Saber trabajar con legacy code es saber trabajar con el 90% del código que existe en el mundo real.

**Ejercicio:** Identificá en FreePress el módulo más viejo o más complejo que no tiene tests. Sin entender completamente qué hace, escribí characterization tests: ejecutá el código con distintos inputs, observá los outputs, y escribí tests que documenten exactamente esos outputs. Una vez que tenés la red de seguridad, identificá un seam — un punto donde podés inyectar una dependencia mockeada — y usalo para hacer el módulo más testeable sin cambiar su comportamiento observable.

**Done cuando:** Podés abrir un módulo de FreePress sin tests, agregarle characterization tests en menos de una hora sin entender completamente su implementación, y luego hacer un refactor pequeño con la confianza de que los tests te van a avisar si rompiste algo. Podés explicar la diferencia entre characterization tests y tests de comportamiento esperado, y cuándo cada uno tiene sentido.

**Recursos:**
- [*"Working Effectively with Legacy Code"*](https://www.amazon.com/Working-Effectively-Legacy-Michael-Feathers/dp/0131177052) de Michael Feathers — el clásico absoluto. Cada capítulo es una técnica específica para un tipo de problema de legacy code. Denso y práctico al mismo tiempo.
- [*"Refactoring"*](https://www.amazon.com/Refactoring-Improving-Existing-Addison-Wesley-Signature/dp/0134757599) de Martin Fowler — el capítulo sobre "Code Smells" complementa perfectamente el trabajo de Feathers.
- [*"The Mikado Method"*](https://www.amazon.com/Mikado-Method-Ola-Ellnestam/dp/1617291218) de Ellnestam y Bäckman — técnica para hacer refactors grandes de forma incremental sin bloquearse. Complemento práctico de Feathers para refactors complejos.
