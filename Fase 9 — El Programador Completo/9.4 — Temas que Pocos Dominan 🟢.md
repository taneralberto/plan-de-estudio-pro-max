# 9.4 — Temas que Pocos Dominan 🟢 [ADVANCED]

- [ ] **Internacionalización (i18n) y Localización (l10n)**

No es solo traducir strings. Es el modelo completo de adaptación cultural: formatos de fecha (en EEUU el mes va primero — 04/19/2026; en Argentina el día va primero — 19/04/2026; en Japón el año va primero — 2026/04/19), moneda (símbolo, separador decimal, separador de miles — varían por locale), números (1.000,50 en Argentina es 1,000.50 en EEUU), dirección de texto (RTL — Right-to-Left — para árabe, hebreo, persa — afecta layout completo), pluralización (en inglés solo "1 article" vs "N articles"; en ruso hay cuatro formas de plural). Time zones: el infierno real — guardar siempre en UTC, convertir solo en el display, y nunca asumir que el servidor y el usuario están en la misma zona horaria.

**Por qué importa:** FreePress tiene audiencia potencialmente global. Si un lector en Japón ve "Published on 04/19/2026" no sabe si es el día 4 del mes 19 (imposible) o el mes 4 del día 19. Si el CMS muestra "Article published yesterday" calculado en el timezone del servidor (UTC), un lector en Buenos Aires podría ver el artículo como publicado "mañana". Estos bugs son invisibles en desarrollo (todos en el mismo timezone), aparecen en producción, y son muy difíciles de reproducir localmente.

**Ejercicio:** Auditá FreePress para problemas de i18n: (1) ¿Todas las fechas se guardan en UTC en la base de datos? Verificalo con `SELECT NOW() AT TIME ZONE 'UTC'` en Postgres. (2) ¿El frontend convierte las fechas al timezone del usuario antes de mostrarlas? Implementalo usando `Intl.DateTimeFormat` nativo. (3) Configurá `@ngx-translate` en el frontend de FreePress para al menos dos idiomas (español e inglés). Las fechas y números deben usar `Intl` — no librerías externas para formateo básico.

**Done cuando:** Podés cambiar el locale del browser a `ar-SA` (árabe — Saudi Arabia) y FreePress muestra las fechas en formato árabe con dirección RTL sin que el layout se rompa. Podés explicar por qué guardar un timestamp en la base de datos sin timezone es siempre un error — incluso si hoy solo tenés usuarios en Argentina.

**Recursos:**
- [*"Internationalization in Angular"*](https://angular.dev/guide/i18n/overview) — la guía oficial de Angular para i18n, incluyendo el pipe `date` con locales y `@angular/localize`.
- [MDN — Intl API](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Intl) — la API nativa del browser para formateo de fechas, números y monedas por locale. Usarla siempre antes de una librería externa.
- [*"Falsehoods Programmers Believe About Time"*](https://gist.github.com/timvisee/fcda9bbdff88d45cc9061606b4b923ca) — lista de todas las asunciones incorrectas sobre tiempo y timezones. Leerla una vez y guardarla.

- [ ] **Regex — De verdad**

No copiar y pegar de Stack Overflow. Entender desde los fundamentos: character classes (`[a-z]`, `\d`, `\w`, `\s`), quantifiers (`*`, `+`, `?`, `{n,m}`), anchors (`^`, `$`, `\b`), groups (`(abc)` capturante, `(?:abc)` no capturante), backreferences (`\1` para referenciar el grupo 1 capturado), lookahead (`(?=...)` positivo, `(?!...)` negativo), lookbehind (`(?<=...)` positivo), non-greedy matching (`*?` en lugar de `*`). Cuándo NO usar regex: parsing de HTML o XML (usa un parser DOM), parsing de código fuente (usa un parser AST), cualquier cosa con anidamiento arbitrario (regex no puede manejar gramáticas recursivas).

**Por qué importa:** En FreePress vas a necesitar validar emails, parsear slugs, extraer menciones de usuarios de comentarios, sanitizar input. Copiar regex de Stack Overflow sin entenderlas es una bomba de tiempo: el 80% de las regex "de validación de email" de internet son incorrectas según RFC 5321, y algunas tienen catastrophic backtracking que puede paralizar el servidor con un input adversarial de 50 caracteres. Entender regex te da la capacidad de escribir una que resuelva exactamente tu problema — y de leer una desconocida y saber si es correcta.

**Ejercicio:** Escribí las siguientes regex para FreePress desde cero, sin copiar: (1) Validación de slug de artículo — solo letras minúsculas, números y guiones, empezando y terminando con letra o número, sin guiones consecutivos; (2) Extracción de menciones `@usuario` de comentarios — capturando el nombre de usuario sin el `@`; (3) Detección de URLs en texto para convertirlas en links — sin falsos positivos en puntuación al final de oración. Para cada una, probala en regex101.com con al menos 5 casos positivos y 5 negativos.

**Done cuando:** Podés leer una regex de producción desconocida de mediana complejidad y explicar qué valida/extrae en menos de 2 minutos. Podés identificar si una regex tiene riesgo de catastrophic backtracking (patrón `(a+)+` o similar) antes de usarla en producción.

**Recursos:**
- [regex101.com](https://regex101.com/) — el mejor tool para escribir, testear y debuggear regex. Tiene explicación visual de cada parte de la expresión. Indispensable.
- [*"Mastering Regular Expressions"*](https://www.amazon.com/Mastering-Regular-Expressions-Jeffrey-Friedl/dp/0596528124) de Jeffrey Friedl — el libro de referencia si querés entender realmente cómo funcionan los motores de regex internamente. Denso pero sin igual.
- [regexcrossword.com](https://regexcrossword.com/) — crucigrama de regex para practicar de forma gamificada. Sorprendentemente efectivo para consolidar el conocimiento.

- [ ] **WebAssembly (WASM)**

Qué es WASM: un formato de bytecode binario que corre en el browser (y en Node.js) con performance cercana al nativo — no es un reemplazo de JavaScript, es un complemento para código CPU-intensivo. Por qué existe: JavaScript no puede compilar a código máquina eficiente para ciertas operaciones numéricas intensivas — WASM sí. Cómo funciona: código en Rust, C, C++ o Go se compila a `.wasm`, que el browser ejecuta en una sandbox junto con JavaScript. La interfaz JS ↔ WASM: pasar datos entre los dos mundos (solo números — strings y objetos requieren serialización). Casos donde WASM tiene sentido real: procesamiento de imágenes en el browser, codecs de video/audio, emuladores, motores de física, criptografía, parsers complejos.

**Por qué importa:** FreePress tiene un editor de contenido rico. El rendering de Markdown a HTML, la generación de thumbnails antes del upload, o la compresión de imágenes en el cliente antes de enviarlas al servidor son tareas CPU-intensivas que bloquean el main thread en JavaScript puro. WASM permite hacer esas operaciones sin bloquear la UI — lo que distingue un editor fluido de uno que se congela medio segundo cada vez que insertás una imagen. No vas a escribir WASM desde cero — vas a usar librerías existentes (Sharp compilado a WASM, marked.js acelerado con WASM).

**Ejercicio:** Implementá compresión de imágenes en el cliente en FreePress antes del upload: cuando el usuario selecciona una imagen de portada para un artículo, comprimila en el browser usando [Squoosh](https://github.com/GoogleChromeLabs/squoosh) o `@squoosh/lib` (que usa WASM internamente) antes de enviarla al servidor. Medí: tiempo de compresión, reducción de tamaño (apuntá a 70% de reducción con calidad aceptable), y si el main thread se bloquea (debería hacerlo en un Web Worker, no en el hilo principal).

**Done cuando:** Podés integrar una librería compilada a WASM en el frontend de FreePress, ejecutarla en un Web Worker para no bloquear el main thread, y comunicar los resultados de vuelta al componente Angular. Podés explicar qué tipos de problemas son buenos candidatos para WASM y cuáles no lo son (DOM manipulation, operaciones I/O — WASM no tiene acceso a esas APIs).

**Recursos:**
- [MDN — WebAssembly](https://developer.mozilla.org/en-US/docs/WebAssembly) — la referencia completa del browser API de WASM con ejemplos de integración con JavaScript.
- [*"Rust and WebAssembly"*](https://rustwasm.github.io/docs/book/) — el tutorial oficial de compilar Rust a WASM. Ideal si querés entender cómo se construye una librería WASM desde cero.
- [Squoosh](https://squoosh.app/) / [squoosh GitHub](https://github.com/GoogleChromeLabs/squoosh) — el mejor ejemplo real de WASM en producción: compresión de imágenes en el browser hecha por Google. El código fuente es una excelente referencia de integración JS ↔ WASM.
