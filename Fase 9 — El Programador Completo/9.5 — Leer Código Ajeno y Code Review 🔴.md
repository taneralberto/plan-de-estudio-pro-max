# 9.5 — Leer Código Ajeno y Code Review 🔴 [CORE]

> ⚠️ **Habilidad crítica en la era de AI.** Con AI generando más código, la habilidad se desplaza de "escribir código" a "evaluar código" — propio, ajeno, y generado por máquinas.

- [ ] **Archaeology de codebases — Cómo abordar un proyecto desconocido**

El momento en que abrís un proyecto que no escribiste y tenés que entenderlo para agregar una feature o arreglar un bug. Por dónde empezar: no abrás `src/index.js` — abrís `package.json` para ver dependencias y scripts, `README.md` para entender el propósito, y la estructura de carpetas para entender la arquitectura. Buscá el "punto de entrada" real (donde la app arranca) y traceá el flujo de una request completa desde el usuario hasta la base de datos. Identificá los "puntos de gravedad" — los archivos que más referencias reciben — suelen ser el core del sistema. Cómo leer tests para entender comportamiento esperado cuando la documentación no existe o está desactualizada.

**Por qué importa:** La mayor parte de tu carrera profesional vas a pasar leyendo código que otro escribió, no escribiendo código nuevo. Un developer junior pierde horas intentando entender un proyecto desde el código; un senior lee `package.json`, identifica que usa NestJS con TypeORM, y ya sabe qué patrones esperar. Esa diferencia no es magia — es método. Y con AI generando código, la capacidad de entender rápidamente qué hace un código (y si lo hace bien) se vuelve crítica.

**Ejercicio:** Abrí un proyecto open source que nunca viste (ej: un repositorio de NestJS con >1000 stars). Sin leer documentación, respondé en 30 minutos: (1) ¿cuál es el propósito del proyecto?, (2) ¿cuál es el punto de entrada principal?, (3) ¿qué framework/librerías core usa?, (4) ¿dónde está la lógica de autenticación si existe?, (5) ¿cuál es el archivo más "importante" (más referenciado)? Verificá después contra el README y la documentación.

**Done cuando:** Podés abrir un proyecto nuevo de tamaño mediano (50-100 archivos) y en 1 hora explicar su arquitectura general, identificar los 3 módulos más importantes, y saber dónde modificarías para agregar una feature específica — sin haber escrito una sola línea del código.

**Recursos:**
- [*"Code Reading: The Open Source Perspective"*](https://www.amazon.com/Code-Reading-Perspective-Developer-Library/dp/0201799405) de Diomidis Spinellis — el libro clásico sobre cómo leer código ajeno sistemáticamente.
- [GitHub — Exploring repositories](https://docs.github.com/en/repositories/working-with-files/using-files/navigating-code-on-github) — cómo usar GitHub para navegar código eficientemente (búsqueda de símbolos, saltar a definición, etc.).
- [*"How to Read Code"*](https://github.com/hamedseq/how-to-read-code) — guía práctica con ejemplos concretos.

- [ ] **Code review efectivo — Cómo revisar y dar feedback constructivo**

Code review no es "aprobar o rechazar" — es una conversación técnica que mejora el código y el conocimiento del equipo. La estructura de un buen review: (1) entender el contexto (qué problema resuelve el PR), (2) revisar la arquitectura general antes de los detalles, (3) identificar problemas reales (bugs, seguridad, performance) vs preferencias personales, (4) dar feedback accionable con "por qué", no solo "cambiá esto". Conventional Comments: prefijos que clarifican la intención del comentario (`💡 suggestion:`, `🔧 fix:`, `❓ question:`, `⚠️ issue:`, `💭 nitpick:`). Cómo dar feedback sin ser agresivo: "esto podría causar X" vs "esto está mal". Cuándo pedir cambios vs cuándo aprobar con comentarios.

**Por qué importa:** Un mal code review es tiempo perdido para todos: el autor se frustra con comentarios vagos, el reviewer no transmite conocimiento, y el código no mejora. Un buen review enseña, previene bugs, y construye cultura de calidad. En equipos con AI, el review se vuelve aún más crítico: alguien tiene que verificar que el código generado tiene sentido, respeta la arquitectura, y no introduce patrones de error típicos de AI.

**Ejercicio:** Hacé un code review real usando Conventional Comments. Puede ser: (1) un PR abierto en un proyecto open source, (2) un PR de un teammate, o (3) un PR tuyo viejo revisado con perspectiva fresca. El review debe tener al menos un comentario de cada tipo: `⚠️ issue:` (algo que rompe o es incorrecto), `💡 suggestion:` (mejora opcional), `❓ question:` (algo que no entendés), `💭 nitpick:` (detalle menor). Cada comentario debe explicar el "por qué".

**Done cuando:** Podés revisar un PR de 200-300 líneas en 20 minutos, identificar los 3 problemas más importantes (bugs, arquitectura, seguridad — no estilo), y escribir comentarios que el autor pueda accionar sin necesidad de una reunión sincrónica.

**Recursos:**
- [Conventional Comments](https://conventionalcomments.org/) — el estándar de facto para estructurar comentarios de code review.
- [Google Engineering Practices — Code Review](https://google.github.io/eng-practices/review/) — las prácticas internas de Google para code review, públicas y detalladas.
- [*"The Art of Constructive Code Review"*](https://medium.com/@sandya.sankarram/the-art-of-constructive-code-review-9d9f6d9b3820) — cómo dar feedback que ayuda sin frustrar.

- [ ] **Detectar code smells — Patrones de código problemático**

Code smells no son bugs — son indicadores de problemas más profundos. Los más comunes: Long Method (un método de 100 líneas probablemente hace más de una cosa), Large Class (una clase que creció sin control), Feature Envy (un método que usa más datos de otra clase que de la propia), Duplicate Code (el mismo código en 3 lugares distintos), Shotgun Surgery (un cambio requiere modificar 10 archivos), Middle Man (una clase que solo delega sin agregar valor), Dead Code (código que nadie usa pero nadie borró). Cómo distinguir un smell de una decisión intencional. Cuándo refactorizar vs cuándo dejarlo.

**Por qué importa:** Un code smell es como un dolor de cabeza: no es la enfermedad, pero indica que algo no está bien. Ignorarlos lleva a código que se vuelve progresivamente más difícil de modificar — la "deuda técnica" que se paga con intereses. Detectarlos temprano permite refactorizar cuando el costo es bajo. Y cuando revisás código ajeno (o de AI), saber identificar smells te permite decir "esto funciona pero va a causar problemas" antes de que los cause.

**Ejercicio:** Tomá un módulo de FreePress que escribiste hace tiempo (o un proyecto viejo). Identificá al menos 3 code smells usando la lista de Fowler. Para cada uno: (1) nombralo correctamente, (2) explicá por qué es un problema, (3) proponé un refactor concreto. No hace falta implementar el refactor — solo identificar y proponer.

**Done cuando:** Abrís cualquier archivo de código y en 10 minutos podés identificar al menos 2 code smells y explicar por qué son problemas (no solo nombrarlos). Sabés la diferencia entre un smell y una decisión de diseño intencional.

**Recursos:**
- [*"Refactoring"*](https://refactoring.com/) de Martin Fowler — el libro fundacional. El catálogo de code smells en el capítulo 3 es referencia obligada.
- [SourceMaking — Code Smells](https://sourcemaking.com/refactoring/smells) — catálogo interactivo de code smells con ejemplos.
- [*"Clean Code"*](https://www.amazon.com/Clean-Code-Handbook-Software-Craftsmanship/dp/0132350882) de Robert Martin — capítulos sobre funciones y clases como guía de lo que NO hacer.

- [ ] **Revisar código generado por AI — Patrones de error típicos**

La AI genera código que compila y funciona — pero tiene patrones de error específicos que hay que aprender a detectar. Over-confidence: código que parece correcto pero tiene edge cases no considerados (ej: no maneja null/undefined, asume arrays nunca vacíos). Hallucinated APIs: métodos que no existen, parámetros incorrectos, nombres de librerías inventadas. Context blindness: código técnicamente correcto que ignora convenciones del proyecto (naming diferente, patrones distintos, imports inconsistentes). Over-engineering: soluciones complejas para problemas simples (un factory pattern para crear un objeto). Security naivety: código que funciona pero expone vulnerabilidades (SQL concatenado, inputs sin sanitizar). Cómo verificar código AI sin reescribir todo: tests, revisión de edge cases, comparación con código existente del proyecto.

**Por qué importa:** La AI no va a reemplazar a los developers — pero los developers que saben revisar código AI van a reemplazar a los que no. Un PR con código AI bien revisado es productividad; un PR con código AI sin revisión es deuda técnica esperando a cobrarse. La diferencia entre "la AI me ayudó" y "la AI me metió en problemas" está en tu capacidad de detectar estos patrones.

**Ejercicio:** Usá un modelo de AI para generar una feature completa para FreePress (ej: sistema de tags para artículos). Sin ejecutar el código, revisalo e identificá: (1) al menos 1 hallucinated API o método incorrecto, (2) al menos 1 edge case no manejado, (3) al menos 1 inconsistencia con el estilo del proyecto. Luego ejecutá y verificá si tus detecciones eran correctas.

**Done cuando:** Podés revisar código generado por AI en 15 minutos, identificar los 3 problemas más probables (APIs incorrectas, edge cases, inconsistencias), y escribir los tests necesarios para verificar que funciona correctamente antes de hacer merge.

**Recursos:**
- [*"How to Review AI-Generated Code"*](https://github.com/microsoft/ai-review_checklist) — checklist de Microsoft para revisar código AI.
- [Anthropic — Responsible Deployment](https://www.anthropic.com/research/responsible-deployment-policy) — guía sobre qué puede y no puede hacer la AI, relevante para entender sus limitaciones.
- [*"The Hallucination Problem"*](https://www.ibm.com/think/topics/ai-hallucination) — entendiendo por qué la AI genera código que parece correcto pero no lo es.

- [ ] **Debugging en código ajeno — Investigar sin el contexto mental del autor**

Debuggear código propio es difícil; debuggear código ajeno es un nivel distinto porque no tenés el modelo mental de por qué se escribió así. El proceso: (1) aceptá que no vas a entender todo de entrada, (2) identificá el comportamiento esperado vs observado, (3) traceá el flujo de ejecución (logs, debugger, console.log si es necesario), (4) formulá hipótesis sobre la causa, (5) verificá con experimentos. Herramientas específicas: `git blame` para ver cuándo y por qué se introdujo una línea, `git log -p -- filename` para ver el historial de cambios de un archivo, búsqueda de issues/PRs relacionados en el repo. Cuándo refactorizar mientras debuggeás vs cuándo solo arreglar el bug.

**Por qué importa:** En producción, los bugs no te van a esperar a que termines de entender el sistema completo. Necesitás poder entrar, identificar el problema, arreglarlo, y salir — idealmente dejando el código mejor de lo que lo encontraste. La habilidad de debuggear código ajeno rápidamente es lo que separa a alguien que puede responder a incidentes de alguien que necesita que otro le explique el sistema primero.

**Ejercicio:** Encontrá un bug abierto en un proyecto open source (GitHub Issues con label "bug"). Sin preguntar en los comments, intentá: (1) reproducir el bug localmente, (2) identificar el código responsable, (3) entender por qué se comporta así, (4) proponer un fix. No hace falta enviar el PR — el ejercicio es el proceso de investigación.

**Done cuando:** Podés abrir un proyecto desconocido, reproducir un bug reportado por otro, identificar el código responsable en menos de 2 horas, y explicar el fix propuesto sin haber escrito el código original.

**Recursos:**
- [*"Debugging"*](https://www.amazon.com/Debugging-Indispensable-Software-Hardware-Problems/dp/0814474578) de David Agans — aplica igual para código ajeno que propio.
- [*"Your Code as a Crime Scene"*](https://www.amazon.com/Your-Code-Crime-Scene-Learn/dp/1680500384) de Adam Tornhill — técnicas forenses para entender codebases usando el historial de git.
- [Git Blame & Log](https://git-scm.com/book/en/v2/Git-Basics-Viewing-the-Commit-History) — documentación oficial para usar git como herramienta de archaeology.
